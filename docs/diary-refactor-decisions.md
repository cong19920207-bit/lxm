## 1. 背景与目标

- **用户端**：按契约消费 `GET /api/diary/list`，能展示、标已读。
- **管理端**：补齐日记历史页（TD-006）；生成链路读取 `diary_rules`（TD-007）。
- **验证**：对话与日记共用 `llm_client` + `VOLC_*`；对话已稳定则视为环境基线已通过，日记异常时再单独查非流式/超时等。

## 3. 问题 5：若列表只用 `items`，「旧的」`diaries` 是否还有其它影响？

- **后端**：`GET /api/diary/list` 的 Pydantic 模型一直是 **`items`**，**从未**在正式响应里提供 `diaries` 字段。
- **仓库内**：除 **`frontend/pages/diary.html`** 外，**无**其它前端/脚本读取 `res.data.diaries`；`backend/routers/diary.py` 里的变量名 `diaries` 仅为 ORM 列表局部变量，**与 JSON 字段无关**。
- **结论**：改为只读 **`items`** 不会破坏其它已对接客户端；**不存在**「旧 API 仍返回 diaries」的兼容层需求。若存在**仓库外的私有脚本**曾错误依赖 `diaries`，需自行改为 `items`（概率极低）。

## 5. 方案审核：仍不清晰处、逻辑冲突、业务遗漏

### 5.1 仍待实现阶段澄清（非阻塞文档）

- **分页**：列表接口返回 `total`、`page_size`；前端当前以「本批条数 < 20」推断 `noMore`，若将来 `page_size` 可配置，需与后端一致或改用 `total` 计算，避免误判「没有更多」。
- **空状态与等级**：若仅修 `items`、仍用 `localStorage.relationship_level`，可能出现 **等级已变但空状态文案未更新**；本次未强制「进页拉关系」，可作为后续小优化。
- **手动批跑（M2a）**：需在运维文档中写清 **命令、依赖环境变量、与当日重复生成逻辑**（已有「当日一条」则跳过）；避免运维误以为是「强制每人再写一条」。

### 5.2 逻辑冲突检查

- **TD-007 回退** 与 **读配置**：回退到硬编码时，**调度时刻**若仍依赖代码内默认 Cron，与「库里有自定义 hour」的认知可能不一致——产品/运维需知：**解析失败 = 内容与时钟都回退到代码默认**，直至配置修复并**重启**（H1）。
- **TD-012**（第三方 doubao 配置落库但运行时读 env）：未来若 LLM 改读 `third_party:doubao`，与日记/对话同源；**当前不冲突**，但长期存在「后台第三方页与真实调用」两套真相，与日记改造独立。

### 5.3 业务 / 场景遗漏（优先对方案）

| 场景 | 说明 |
|------|------|
| 1 级无当日互动 | 按 PRD **不生成**；用户长期只有 1 级且只在「北京时间当日、UTC 已跨日」聊天 | 可能 **UTC 日界** 与用户感知「今天」不一致；你已决定时区后置，**实现阶段**在文案或后续时区债里消化。 |
| Misfire + 当日未跑 | M2a 手动补跑前，需确认 **是否仍算同一 UTC 日**；补跑后用户端才能看到「今天」的日记。 |
| LLM 失败 | 不写库、无重试 UI；与对话失败体验一致，**不单独列为本次遗漏**，但运营手册可写「查日志与火山配额」。 |
