## 1. 背景与目标

### 1.1 当前功能

方案确认时，管理后台仅有 4 个角色：`super_admin`、`ops_admin`、`ai_trainer`、`tech_ops`。鉴权使用 Admin JWT（Bearer，`type=admin`），`require_role` 依据数据库实时角色判断。部分生活流页面已有局部只读逻辑，但当时不存在覆盖全后台的只读角色。

安全审查同时发现 JWT 默认密钥、锁定后旧 Token、登录计数竞态、发布确认仅在前端、操作日志凭据暴露等问题，详见漏洞专档。

### 1.2 本期目标

1. 完成已经确认纳入本期的后台安全加固并验证。
2. 新增 `observer`（展示名“观察者”），允许查看除账号管理外的后台业务数据，但禁止业务写入和系统内建导出。
3. 后端作为权限安全边界；前端只读控制只改善体验，不能替代后端鉴权。
4. 明确延期项和剩余风险，不把未实施内容描述为已修复。

### 1.3 分阶段交付

| 阶段 | 内容 | 进入下一阶段条件 |
|------|------|------------------|
| 阶段 A：安全加固 | VULN-002、003、004、005、006、008 的响应统一部分、009 | 本期安全项测试通过，原有四角色回归通过，人工确认并独立部署；运行稳定后才可进入阶段 B |
| 阶段 B：观察者 | 后端权限总闸、读权限、账号角色、前端全量只读改造 | 观察者权限测试、全部变动页面可用性验收和全角色回归通过后独立部署 |

VULN-001 发布测试流程和 VULN-007 后端强制改密不属于阶段 A 门禁，必须保持“已知未修”状态。

## 3. 目标权限模型

### 3.1 观察者后端权限

统一规则：

- `GET`、`HEAD` 默认允许进入统一鉴权流程，但仍必须通过 Admin JWT、数据库实时账号状态、Token 版本和具体端点 `require_role` 校验；“默认允许”不等于自动授予所有读取权限。
- CORS 中间件处理的 `OPTIONS` 预检作为匿名基础设施请求放行，不返回后台业务数据，也不属于观察者业务读取权限。
- `POST`、`PUT`、`PATCH`、`DELETE` 默认返回 403。
- 精确放行：
  - `POST /api/admin/auth/logout`
  - `POST /api/admin/auth/change-password`
- 账号管理接口继续只允许 `super_admin`，观察者不可读取账号列表或详情。
- 当前 3 个系统内建导出接口——`POST /api/admin/operation-logs/export`、`POST /api/admin/stats/report/export`、`POST /api/admin/system/logs/export`——分别增加专用观察者拒绝依赖；未来新增导出/下载接口也必须与 HTTP 方法无关地显式拒绝。
- 当前和以后新增的 GET/HEAD 接口不得修改 MySQL 业务数据、权限、账号状态、配置版本，不得触发生成、测试、发布、修复、回填、删除或异步任务。允许读取权威数据后进行幂等、有限 TTL 的 Redis 缓存回填或同类内部只读派生缓存。

观察者修改自己的密码只属于安全自助操作，不代表获得任何业务写权限。改密成功后递增 `token_version`，前端清除本地 Token 并跳转登录页。

### 3.2 观察者前端权限

- 35 个现有后台页面均纳入菜单和直接访问检查；其中当前盘点的 26 个含非 GET 调用页面需逐页核对写入口，账号管理页由前端立即拦截/跳转，其余约 25 个业务页面执行只读改造。
- 在 `admin/static/js/admin-api.js` 提供统一角色判断和只读应用方法，例如 `isObserver()`、`applyObserverReadOnly()`；覆盖首屏静态控件和后续动态插入控件。
- 新增统一写操作标记（如 `data-write-action`）：写操作按钮隐藏；用于展示配置的文本输入框/文本域设为只读；无法只读的选择框、开关和文件控件禁用。
- `adminRequest()` 对观察者非 GET 请求提供前端体验兜底，仅精确放行登出和修改自己密码；后端总闸仍是唯一安全边界。
- 保留搜索、筛选、分页、Tab、详情、复制和纯 GET 刷新等读取操作，不得按元素类型粗暴禁用全部按钮或表单。
- AI 测试工具页可查看配置或历史结果，但不能发起测试、保存用例或触发生成。
- Header 展示“观察者”，保留修改自己密码和退出入口。
- `sessionStorage` 中角色只影响页面展示；后端每次请求仍使用数据库实时身份，前端角色陈旧不能形成越权。

后台页面是静态 HTML，页面文件本身不是权限边界；直接访问时即使浏览器先取得 HTML，也不得取得账号数据或执行受限操作。遗漏前端标记时，用户可能看到一个最终返回 403 的按钮，但不能绕过后端权限。因此验收既要检查体验，也要直接测试 API。

### 3.3 导出边界

“观察者禁止导出”仅指系统提供的导出、下载和生成报表文件能力。观察者仍能查看允许读取的数据，因此无法从技术上完全阻止截图、复制、浏览器开发工具读取或分页聚合。文档和验收不得把该能力描述成防数据外带或防抓取方案。

### 3.4 敏感读取边界

- 观察者可读取第三方服务状态和非敏感配置，但 API Key、Secret、Token、Password 等凭据只返回“已配置/未配置”，不显示首尾字符或其他可推断片段。
- `memory-rules.html` 可读取 Step6 Prompt、全局记忆和非敏感 DashVector 参数，但不能保存 Prompt、测试连接、修改密钥或删除记忆。
- Embedding 服务配置实际位于 `third-party.html`，观察者只读取状态和非敏感参数；`vector-token-config.html` 的召回和 Token 数值可读但不可保存。
- 操作日志和系统日志必须先经过 A5 的共享脱敏工具后才能向观察者返回；该读取脱敏同样适用于其他管理员角色。

## 4. 阶段 A：安全加固设计

### A1. JWT 密钥启动守卫（VULN-002）

- `ADMIN_JWT_SECRET` 不再使用代码默认回退。
- 缺失、空值、纯空白、`admin_secret_change_me`、`your_admin_jwt_secret_here` 均导致启动失败。
- 配置读取函数负责统一校验，lifespan 启动阶段再显式调用同一校验函数，形成“使用时守卫＋启动时快速失败”的双保护，禁止复制两套判断规则。
- `.env.example` 保留示例作用，但示例值必须被程序识别为不可运行值。
- CI 和测试环境必须显式设置独立测试密钥。
- 当前部署若已有自定义密钥，可以继续使用；本期不强制轮换。
- 本期不增加最短长度、复杂度或强制轮换规则，避免改变已经生效的自定义密钥策略。

### A2. Token 版本和会话撤销（VULN-003、VULN-006）

`admin_users` 新增：

```text
token_version INTEGER NOT NULL DEFAULT 0
```

新签发的 Admin JWT 必须携带版本；`get_current_admin` 同时校验：

- Token 类型、签名和过期时间；
- 账号存在且 `is_active=True`；
- 账号未锁定；
- Token 版本等于数据库当前版本。

版本递增矩阵：

| 事件 | 是否递增 | 结果 |
|------|----------|------|
| 连续失败达到阈值并锁定 | 是 | 所有旧会话失效 |
| 超级管理员手动解锁 | 否 | 锁定时已递增，旧会话不能复活 |
| 用户修改自己的密码 | 是 | 当前及其他设备重新登录 |
| 超级管理员重置密码 | 是 | 所有旧会话失效 |
| 主动登出 | 是 | 同账号所有设备旧会话失效 |
| 实际角色变更 | 是 | 防止旧角色 Token 在角色来回调整后恢复；仅修改备注或提交相同角色不递增 |

不含 `token_version` 的历史 Token 拒绝；部署后所有管理员重新登录一次。本期不实现单设备撤销或 JWT 黑名单。

本期不新增管理员停用/重新启用接口和 UI。现有 `is_active` 仍参与鉴权，但日常运维不得直接修改数据库模拟停用；紧急人工停用如确需执行，必须同时人工递增 `token_version` 并留审计记录，否则重新启用可能使旧 Token 恢复有效。

### A3. 登录事务和统一失败响应（VULN-004、VULN-008）

- 查询管理员账号后使用 `SELECT ... FOR UPDATE` 锁定目标行。
- 在同一事务内完成锁定状态检查、密码校验、失败计数、第五次锁定、版本递增和成功登录后的计数清零。
- 连续 5 次错误密码锁定、仅超级管理员解锁、不自动解锁的线上规则不变。
- 不存在账号、密码错误、锁定、停用统一返回 HTTP 200，响应为 `{"code": 20001, "data": null, "message": "账号或密码错误"}`，不返回剩余次数、锁定状态或账号状态。
- 不存在账号时执行伪密码哈希校验，降低明显的响应时间差异。
- 真实失败原因只进入服务端安全日志，并遵守脱敏规则。

统一响应不解决恶意锁号。本期不引入 Redis 或网关级 IP＋账号限流。

### A4. life-config 服务端 CONFIRM（VULN-005）

- 发布请求模型增加 `confirm_text`。
- 请求字段允许缺省或空值进入端点业务校验，服务端严格要求 `confirm_text == "CONFIRM"`；缺失、空值或错误值统一通过 `ApiResponse` 返回业务码 `20021`。
- 当前 7 处 life-config 前端发布调用全部显式传递 `confirm_text: "CONFIRM"`。
- 不改变现有草稿优先展示、生效版本、历史版本和回滚逻辑。
- 不增加发布流程业务测试集、测试凭证、状态机或数据库迁移；本项仍必须增加自动化回归测试。

CONFIRM 只用于降低误操作，不替代权限校验或二次认证。

### A5. 操作日志凭据脱敏（VULN-009）

封装一个共享、无状态、幂等的递归脱敏工具，统一替换为 `[REDACTED]`。工具不得维护请求级状态；单字段处理异常时该字段按失败关闭方式遮蔽，但不得让正常业务操作因审计脱敏异常而失败。

双层处理：

1. `log_operation` 写入 `before_value`、`after_value` 和可能包含凭据的目标描述前递归脱敏。
2. 操作日志列表、详情和导出返回前再次调用同一工具，遮蔽历史遗留明文。
3. 系统日志列表和导出返回前也调用同一工具，所有管理员角色获得一致的安全读取结果。
4. 对字典和数组直接递归；对字符串先尝试按 JSON 解析，无法解析时只匹配已知凭据赋值格式和 Authorization/Bearer 形式，不能用整段字符串遮蔽代替精确脱敏。

大小写不敏感的凭据字段范围：

- `api_key`、`secret_key`、`api_secret`、`access_key_secret`；
- `access_token`、`refresh_token`、`authorization`；
- `password`、`pepper`、`private_key`；
- 其他语义明确的 `_secret`、`_token` 字段。

统一替换为 `[REDACTED]`。Prompt、普通配置、对话、记忆、`max_tokens`、描述和版本等非凭据内容不得误脱敏。

本期不批量改写历史数据库记录；后台读取会遮蔽，但具备数据库直接访问权限的人仍可能看到历史明文。

### A6. 阶段 A 门禁

全部满足后才可人工确认进入阶段 B：

1. 漏洞专档中本期项目均填写真实验证结果。
2. JWT 配置、Token 版本、行锁并发、登录统一响应、CONFIRM 和日志脱敏测试通过。
3. 使用真实 MySQL 8 完成并发错误登录测试；SQLite 单元测试不能代替行锁验证。
4. 现有 4 个角色的登录、读写、发布、账号管理和日志导出回归通过。
5. 阶段 A 前后端同步发布，部署后通知所有管理员重新登录，并完成运行稳定性确认。
6. VULN-001、VULN-007 及恶意锁号风险仍明确标为未修，不得以“阶段 A 完成”掩盖。

## 5. 阶段 B：观察者实现步骤

### B1. 角色与账号

- 账号角色校验增加 `observer`；数据库字段为字符串，长度足够，无需为角色增加迁移。
- 在现有 Pydantic 角色校验、模型注释、账号角色选项/标签、Header 标签和菜单配置中最小增加 `observer`，本期不重构角色注册中心。
- 仅 `super_admin` 可以创建、修改、重置、解锁或删除观察者账号。
- 账号管理菜单不向观察者开放，静态页面直访由前端拦截/跳转，账号数据接口由后端拒绝。

### B2. 服务端集中总闸

- 在 `get_current_admin` 或等价的统一管理员依赖中实现观察者方法级总闸。
- 总闸执行顺序为：JWT 与 `token_version` 校验、数据库实时账号/角色校验、锁定与启用状态校验，然后再执行观察者方法规则。
- 精确匹配登出和自助改密例外，不能使用过宽的路径前缀白名单。
- 对导出/下载接口增加与 HTTP 方法无关的显式拒绝。
- CORS `OPTIONS` 由中间件匿名处理，不进入后台业务端点，不应被实现成需要 Admin JWT 的观察者读取请求。
- 自动化盘点所有 `/api/admin/*` 路由：除登录外必须有管理员鉴权；所有写路由必须受观察者总闸保护。
- 已有路由上的 `require_role` 继续保留，处理各业务角色之间更细的授权，不由总闸替代。

### B3. 业务读取权限

- 将观察者加入除账号管理外的必要读取角色集合。
- 现有同时用于 GET 和写请求的角色集合必须拆分为读取集合与写入集合，禁止直接把 `observer` 加进共用的 `_ALLOWED_ROLES` 后意外获得写权限。
- 对话、记忆、操作日志、系统日志、配置、第三方配置等按确认范围可读。
- 操作日志必须先完成 A5 双层脱敏，避免只读权限扩大凭据暴露。
- 用户 Open API Key 和第三方凭据对观察者仅返回“已配置/未配置”；其他管理员继续遵守既有接口权限和统一日志脱敏规则。

### B4. 前端完整只读改造

- 公共菜单、角色名称和 Header 统一支持观察者。
- 对全部含写入口页面添加统一写操作标记并应用只读助手，公共助手同时处理首屏静态控件和动态生成的表格行按钮、弹窗提交。
- 写按钮隐藏；展示型文本输入设为只读；选择框、开关和文件控件禁用；搜索、筛选、分页、Tab、详情和纯 GET 刷新保持可用。
- `adminRequest()` 增加观察者非 GET 前端兜底，仅放行登出和自助改密；不得以该前端兜底替代后端 403。
- 移除观察者的账号管理菜单；静态页面被直接打开时立即拦截/跳转，且不能加载账号数据。
- 保留自助改密和退出入口。
- 对未能使用公共标记的动态按钮、表格行操作和弹窗提交逐项处理。

### B5. 契约与交付

- 阶段 A 实现、验证并经人工确认后，只同步已经生效的安全契约，观察者继续标记未实现；阶段 A 独立部署并稳定后才开始阶段 B。
- 阶段 B 实现并验证后再更新 `docs/contract.md` 和相关长期规则中的角色与 RBAC 契约，并独立部署。
- 漏洞专档只在取得真实验证结果后把对应项目改为已验证；VULN-001、VULN-007 和已接受风险不得提前关闭。
- 本 PRD 的“待开发”状态只能在实际测试结果写入后更新，不得因完成方案确认而标为已完成。

## 6. 数据与接口影响

### 6.1 数据库迁移

本期仅一项必需迁移：

```text
admin_users.token_version INTEGER NOT NULL DEFAULT 0
```

- 现有行回填为 0。
- `observer` 使用现有字符串角色字段，无角色迁移。
- 行锁、JWT 启动守卫、CONFIRM、日志读取脱敏均不增加字段。
- VULN-001 延期，因此本期不增加测试凭证、草稿版本或发布状态表。

### 6.2 接口语义变化

| 接口/范围 | 变化 |
|-----------|------|
| Admin JWT | 新 Token 携带 `token_version`；历史无版本 Token 失效 |
| 登录 | 失败场景统一响应；内部计数改为行锁事务 |
| 登出 | 变为同账号全会话撤销 |
| 修改自己密码 | 成功后全会话撤销并要求重新登录；观察者允许调用 |
| 账号安全操作 | 锁定、重置密码、实际角色变更触发版本递增；本期不新增停用/重新启用功能 |
| `POST /life-config/publish` | 请求必须包含正确 `confirm_text`；缺失或错误返回业务码 `20021` |
| 观察者业务接口 | 读默认允许；写默认 403；账号管理和导出显式 403 |

### 6.3 部署顺序

阶段 A：

1. 在所有环境准备有效的 `ADMIN_JWT_SECRET`，测试环境显式使用独立密钥。
2. 在代表性 MySQL 8 环境验证 `token_version` 迁移、回填和兼容性。
3. 完成阶段 A 自动化测试、真实 MySQL 并发测试、四角色回归和人工确认。
4. 同步部署阶段 A 后端安全改动及 life-config 前端 CONFIRM 参数调整。
5. 通知所有管理员重新登录，确认运行稳定；失败则停止后续阶段。

阶段 B（仅阶段 A 稳定后开始）：

1. 完成观察者角色、后端总闸、读取权限、导出拒绝和前端只读改造。
2. 完成路由权限矩阵、全部变动页面可用性测试、观察者验收和原有四角色同页面回归。
3. 更新已经验证的角色/RBAC 契约并独立部署阶段 B。

代码回滚不会自动撤销数据库字段，但回滚到旧鉴权代码会重新引入已修安全问题；因此安全验证失败时应优先停止发布或回滚完整应用版本，并保留兼容字段。

## 8. 测试与验收

### 8.1 阶段 A

- 缺失、空值及公开示例 JWT 密钥均拒绝启动；自定义密钥正常。
- 历史无版本 Token 被拒绝；新 Token 正常。
- 锁定、改密、重置、登出、实际角色变更后旧 Token 失效；解锁后旧 Token 不恢复；备注或相同角色更新不误撤销会话。
- 串行和真实 MySQL 8 并发错误登录均准确计数，第五次锁定并递增版本；SQLite 仅用于其他单元/接口测试。
- 不存在、错密、锁定、停用统一返回 HTTP 200、业务码 `20001`、`data: null` 和“账号或密码错误”，且无剩余次数。
- life-config 无/错 CONFIRM 返回业务码 `20021`；正确 CONFIRM 的合法调用及当前 7 个前端入口通过。
- 嵌套对象和数组中的凭据在操作日志写入、列表、详情、导出以及系统日志列表、导出中均脱敏；非凭据字段不误伤。
- 4 个现有角色完整回归。

### 8.2 阶段 B 后端

- 自动枚举所有 `/api/admin/*` 路由，除登录外均需管理员鉴权。
- 观察者所有 `POST/PUT/PATCH/DELETE` 默认 403；仅登出和修改自己密码例外。
- CORS `OPTIONS` 匿名预检正常，但不返回业务数据；业务 GET/HEAD 仍需完整管理员鉴权。
- 观察者无法访问账号管理和 3 个现有内建导出接口；自动化清单确保未来导出不被 GET 默认规则放行。
- 观察者可以读取已确认的业务数据。
- 观察者读取第三方凭据和用户 Open API Key 时只获得“已配置/未配置”，不能获得首尾字符。
- 当前 GET/HEAD 接口不存在业务状态写入副作用；有限 TTL 的幂等缓存回填不判为业务写入。
- 绕过前端直接请求仍不能写入或导出。
- 原有 4 个角色的合法权限不被观察者总闸误伤。

### 8.3 阶段 B 前端

- 菜单无账号管理；直接打开账号静态页会立即拦截/跳转，账号数据 API 返回 403。
- 搜索、筛选、分页、详情和只读数据展示可用。
- 新增、编辑、保存、删除、发布、回滚、导入、导出、测试等入口不可用。
- 自助改密可用，成功后清除 Token 并返回登录页。
- 动态创建的按钮、表格行操作和弹窗提交均已覆盖。
- 35 个页面全部纳入盘点；账号页禁入，剩余约 25 个含写入口业务页面逐页验证可进入、数据可加载、只读控件正确、保留的读取操作可用。
- 所有权限变动页面按 5 角色权限矩阵逐页验证：预期可访问的角色必须能进入页面、加载数据、使用合法读取操作并完成原有合法写操作；预期无权的角色必须正确跳转或收到 403。观察者重点验证只读控件，原有 4 个角色重点确认未被统一助手、角色集合拆分或请求兜底误伤。

### 8.4 测试执行方式

- 以现有 pytest 为主要自动化门禁，补充安全鉴权、脱敏、路由清单和静态页面契约测试。
- MySQL 行锁必须在独立、非生产的 MySQL 8 集成环境执行并发测试。
- 本期不新建 CI/CD 工作流，也不引入 Playwright 等浏览器自动化基础设施；前端动态交互使用逐页人工浏览器验收清单。
- 页面验收清单至少记录页面、角色、直接访问结果、首屏数据加载、搜索/筛选/分页/详情、静态与动态写控件、合法写操作回归及结论，不能只验证按钮是否隐藏。
- 测试或未来 CI 环境必须显式设置独立 `ADMIN_JWT_SECRET`，不得复用生产密钥。

测试报告必须记录实际命令、通过/失败数量和失败原因；测试失败时不得把文档状态改成“可发布”。
