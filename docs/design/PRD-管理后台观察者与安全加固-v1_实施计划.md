# PRD-管理后台观察者与安全加固-v1 实施计划

> 唯一施工依据：`docs/design/PRD-管理后台观察者与安全加固-v1_开发步骤拆解.md`  
> 完成状态依据：`docs/progress/PRD-管理后台观察者与安全加固-v1_progress.md`  
> 契约草案目录：`docs/contract/drafts/管理后台观察者与安全加固/`  
> 创建日期：2026-07-15

## 1. 执行原则

- 全部 STEP 严格按编号顺序执行，一次只实施并验收一个 STEP。
- 每个 STEP 原文规定的前置条件、开发任务、排除范围、验证要求和五项完成标志均为硬性门禁。
- 任一完成标志未满足时，不把该 STEP 更新为 ✅，不进入下一 STEP；阻塞及真实证据写入 progress。
- 自动化测试从项目根目录以 `PYTHONPATH=. pytest ...` 形式执行，并记录真实命令、通过数、失败数和失败原因。
- MySQL 行锁与迁移验证必须使用独立非生产 MySQL 8，SQLite 不得替代。
- 不实施步骤文档明确排除的功能；不引入 CI/CD 或 Playwright。

## 2. 里程碑与进入闸门

| 里程碑 | STEP | 交付能力 | 进入闸门 |
|---|---|---|---|
| M1 安全加固 | STEP-001～017 | 安全能力完成、验证并独立部署 | 无 |
| M2 观察者基础 | STEP-018～029 | 后端只读权限和公共前端底座完成 | M1 已独立部署、稳定并由人工确认 |
| M3 页面闭环 | STEP-030～040 | 35 页面五角色验收、契约同步并独立部署 | M2 全部 STEP 验收通过 |

## 3. M1：安全加固与独立部署

执行顺序：

`001 密钥双守卫` → `002 token_version 迁移` → `003 JWT 版本校验` → `004 登录行锁` → `005 统一失败响应` → `006 改密撤销会话` → `007 登出撤销会话` → `008 安全操作版本矩阵` → `009 发布 CONFIRM 后端校验` → `010 七个前端发布入口` → `011 共享脱敏工具` → `012 操作日志双层脱敏` → `013 系统日志脱敏` → `014 自动化安全回归` → `015 MySQL 8 并发验证` → `016 四角色回归` → `017 契约、独立部署与人工门禁`

M1 完成条件：STEP-001～017 的五项完成标志全部满足；生成 `M1_契约草案.md`；仅在 STEP-017 把已经验证的阶段 A 安全事实同步到正式契约。独立部署和人工稳定性确认缺一不可。STEP-017 未完成前，`.cursorrules` 保持四角色，M2 不开放。

## 4. M2：观察者后端与公共前端底座

执行顺序：

`018 observer 角色扩展` → `019 方法级集中总闸` → `020 CORS OPTIONS 边界` → `021 三个导出显式拒绝` → `022 用户/统计/日志读取拆分` → `023 AI 配置读取拆分` → `024 记忆/向量/知识等读取拆分` → `025 系统监控与第三方读取拆分` → `026 生活流读取拆分` → `027 凭据状态化展示` → `028 全量路由与副作用审计` → `029 公共菜单、Header、只读助手与请求兜底`

M2 完成条件：STEP-018～029 的五项完成标志全部满足；生成 `M2_契约草案.md`。本阶段不提前修改 35 个页面的专项逻辑，也不把 observer/RBAC 草案同步到正式契约。

## 5. M3：全页面只读闭环与最终发布

执行顺序：

`030 账号页与 API 专项验证` → `031 用户/报表/日志页面` → `032 系统/第三方/AI/看板页面` → `033 人格/Prompt/安全规则页面` → `034 记忆/向量/知识页面` → `035 Agent/关系/日记页面` → `036 生活计划/世界观页面` → `037 朋友圈/评论/感知页面` → `038 生活流配置页面` → `039 后端权限自动化门禁` → `040 35 页面五角色验收、契约同步与独立部署`

M3 完成条件：STEP-030～040 的五项完成标志全部满足；35 个后台页面完成五角色逐页人工浏览器验收；`login.html`、`error.html` 按匿名/公共页面单独验收；生成 `M3_契约草案.md` 后，才同步正式 observer/RBAC 契约、漏洞状态、PRD、progress 和 `.cursorrules` 五角色体系。阶段 B 必须独立部署并完成人工稳定性确认，任一失败不得关闭 PRD。

## 6. 单 STEP 交付记录

每个 STEP 在 progress 中至少记录：

1. 改动文件；
2. 真实验证命令；
3. 通过数量、失败数量及失败原因；
4. 契约条目草稿；
5. 阻塞项；
6. 完成日期和进度更新。

## 7. 已知文档冲突处理

- 功能清单 F-030 的依赖 `STEP-030` 为自依赖；执行时以 STEP-030 正文和 progress 表一致的 `STEP-029` 为准。
- “100% 按文档”指覆盖全部 40 个 STEP 及全部完成标志，不扩展步骤文档明确排除的范围。
