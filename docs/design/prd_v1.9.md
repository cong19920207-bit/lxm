### 0.1 人设基础文本（lxm_base_persona）

**字段类型**：后台大文本框（Textarea），支持管理员随时编辑

**结构**：按五字段组织——角色背景、性格特征、情感偏好、语言风格、行为模式。该结构与现有 IM 消息系统已在使用的人设字段结构一致。

### 0.3 分类词汇表（categories_vocab）

**字段类型**：后台标签编辑器（可增删），独立管理页面

**用途**：约束 LLM-01 生成 `life_plan_outline.categories` 时只能从固定列表中选取，不允许自由生成新词。

**约束原因**：`categories` 参与 `feed_post.dedup_hash`（`venue_type + category + city` 三字段组合）的去重计算，词汇必须保持稳定统一，否则去重机制会因词汇漂移而失效。这与情绪词汇（见 0.4）性质不同——分类词汇的锁定是**系统正确性约束**，情绪词汇的处理方式是**内容表达自由度**考量，两者判断标准不一致，不可混用同一套开放策略。

**初始词汇表**：

```
工作 / 学习 / 旅游 / 购物逛街 / 探店美食 / 户外散步 / 休闲在家 / 看展文化 / 运动健身 / 社交
```

后台支持增删该列表，增删后 LLM-01 下次生成即时生效。

**关联影响排查（v1.9.4 已闭合，原附录 A-4）**：经代码仓排查，IM 记忆体系使用 `兴趣-偏好` / `经历-出行` 等 **Key 前缀**与 `memory_type` 向量分类，与生活流 `categories_vocab`（工作/旅游/探店美食等）**命名空间独立、无冲突**。后台展示建议：生活流配置称「生活流内容分类」，记忆管理维持「记忆 Key」，避免运营口语混淆。

### 0.4 情绪词汇表（emotion_vocab）与兜底机制

**字段类型**：后台标签编辑器（可增删）+ 两张配套映射表

**设计原则：核心词优先 + 自由兜底**。与 `categories_vocab` 的完全锁定不同，情绪表达服务于"反 AI 真实感"这一核心产品原则，不宜完全锁死，因此采用以下机制：

- LLM-03（心理快照）与 LLM-04（emotion 附带生成路径）在 Prompt 中被引导为"优先从以下核心情绪词中选取，如果均不能准确贴合当前场景的细腻状态，可以使用更贴切的词汇"
- `emotion_value` / `feed_post.emotion` 字段因此可能是核心词，也可能是 LLM 自由生成的词
- 该字段后续会被 P-12（人物图）、P-13c（情绪图）用于查询关键词映射表，若查询命中核心词表则精确匹配；若命中的是自由词（表中没有），则使用统一的兜底关键词组，不阻断图片生成流程

**当前核心词汇表（14 个）**：

```
慵懒 / 雀跃 / 低落 / 平静 / 焦虑 / 满足 / 怀念 / 烦躁 / 期待 / 感慨 / 孤独 / 无聊 / 迷茫 / 释然
```

（较 PRD v1.7 举例的 4 词扩展为 14 词，新增：孤独、无聊、迷茫、释然）

**配套映射表**（详见 9.4 图片生成配置）：
- `emotion_img_keyword`：情绪→人物图表情/状态关键词，覆盖上述 14 个核心词
- `emotion_atmosphere_desc`：情绪→情绪图氛围描述词，覆盖上述 14 个核心词
- `emotion_fallback_img_keyword` / `emotion_fallback_atmosphere_desc`：兜底关键词组，emotion 值未命中核心词表时使用，各为固定的一组通用关键词，无需随情绪词增多而持续维护

## 一、方案概述

### 1.1 功能定位

本系统并非传统社交朋友圈，而是 AI 虚拟人"林小梦"的生活流内容展示平台。核心目标是通过模拟真实人类的生活轨迹与社交行为，强化林小梦的"真人感"，同时作为对话系统的自然入口，降低用户开口成本，提升用户粘性与陪伴体验。

### 1.2 核心设计原则

- **全局统一内容**：所有用户看到完全相同的生活流内容，大幅降低生成成本与工程复杂度
- **人设一致性优先**：所有内容严格遵循林小梦的人格设定与生活节奏
- **弱关联个性化**：全局内容保持中立，个性化体验通过评论回复与对话层实现
- **反 AI 真实感**：刻意引入"不完美"元素，避免过度精致的 AI 感

> ⚠️ **v1.9 补充说明**：本系统与现有 AI 日记功能（`ai_diary`，按用户个性化生成、基于对话历史与关系等级）是两条完全独立、并行共存的业务线，互不替代、互不影响。前者是"全局统一内容"，后者是"按用户个性化内容"，设计哲学不同，请勿混淆。

### 1.3 整体架构

系统采用单向流水线架构，分为四个核心模块：

1. **上游**：Life Planner 生活状态规划引擎（LIFE000）：生成生活计划（含周大纲与每日具体场景）
2. **中游**：她的宇宙（PER003）：基于每个生活场景独立生成心理状态快照；同次调用自动生成并维护静态观点事件库
3. **内容层**：朋友圈内容生成引擎（LIFE001）：一对一抽取场景与对应快照，生成文案与图片
4. **下游**：Feed 展示系统（LIFE002）：展示内容、承接互动

> ⚠️ 各模块保持独立，下游模块只读取上游数据，不触发、不通知上游模块。

> ⚠️ **v1.9 命名变更说明**：原"世界观系统"更名为**"她的宇宙"**。原因：经代码核查，现有 IM 系统已存在一张 `world_state` 表，是**按用户**、由对话事件触发、用于对话记忆检索的机制，与本模块（**全局**、由生活场景触发、记录角色固定观点）是完全不同的两个概念，仅中文命名撞车。为避免后台同时出现两个"世界观"入口造成管理员混淆，中文业务概念统一改名为"她的宇宙"。**注意：此改名仅限中文业务概念/章节标题/后台入口名称，数据库层面表名 `worldview_snapshot`/`worldview_event` 及字段名 `worldview_trigger` 保持不变**，因为这些是纯技术标识符，不会出现在管理后台界面上，不产生实际混淆风险，且不影响本文档技术方案的连续性。

### 1.4 LLM 调用总览

全系统共 7 个 LLM 调用点，按上下游顺序统一编号：

| 编号 | 所属模块 | 所属任务 | 触发时机 | 调用频率 | 模型 |
|------|---------|---------|---------|---------|------|
| LLM-01 | LIFE000 | 周大纲生成 | 每周日 23:00 | 每周 1 次 | DeepSeek（独立可配置，见 9.1） |
| LLM-02 | LIFE000 | 日场景细节生成 | 每日 00:20 | 每日 1 次 | DeepSeek（独立可配置，见 9.1） |
| LLM-03 | PER003（她的宇宙） | 心理快照 + 她的宇宙事件联合生成 | 每日 00:45 | 每日每 scene 各 1 次（最多 5 次） | DeepSeek（独立可配置，见 9.2） |
| LLM-04 | LIFE001 | 朋友圈文案生成（hashtags + emotion 附带输出） | 每日 01:00 | 每日 2–3 次 | DeepSeek（独立可配置，见 9.3） |
| LLM-05 | LIFE004 | 评论回复生成 | 用户评论后按关系阶段延迟启动 | 每条用户评论触发 1 次 | DeepSeek（独立可配置，见 9.5） |
| LLM-06 | LIFE004 | 点赞感知 IM 生成 | 特殊档：窗口内前 N 次 100%+短延迟；常规：30%+关系档长延迟（见 6.1.3） | 按概率/特殊档触发（同帖仅一次） | DeepSeek（独立可配置，见 9.5） |

## 二、Life Planner 生活状态规划引擎（LIFE000）

### 2.1 核心职责

作为整个系统的事实层，负责生成林小梦的完整生活故事线，为她的宇宙与朋友圈内容生成提供统一的内容基准。

**v1.3 起，LIFE000 拆分为两级：**

- **周大纲规划层**：按自然周提前规划接下来每天的城市与内容分类，承担生活比例调度与多日行程连贯性职责
- **日场景细节生成层**：基于当日已锁定的大纲条目，生成当日具体的生活场景内容

---

### 2.2 周大纲规划层

#### 2.2.1 职责定位

负责按自然周提前规划接下来一周每天的"城市"与"内容分类"，解决两个问题：

1. 多日行程（尤其长途旅游）的连贯性与"出发-途中-返回"完整叙事弧线，避免日生成阶段因缺乏全局视角而出现穿帮或行程突然中断
2. 承接原本在日生成阶段控制的生活比例约束（主场城市本地 / 周边短途 / 长途旅游），统一在周维度做调度决策

#### 2.2.2 生成时机

- 每周日 **23:00** 自动触发，生成下一个自然周（周一至周日）的完整大纲（**LLM-01**）
- 若 23:00 生成失败，**23:30** 自动重试一次
- 若重试仍失败，该自然周大纲**整体缺失**，不再有任何后续自动补救机制，由后台管理员人工逐日补录
- 生成成功/失败/重试均写入系统日志（触发时间、结果、失败原因等），可在后台日志页面查看

#### 2.2.3 大纲内容规则（v1.8 更新）

- 颗粒度：每个自然日一条记录，包含：
  - `city`：当天所在城市
  - `categories`：当天内容分类，可多选，**必须从后台维护的 `categories_vocab` 固定枚举表中选取**（详见 0.3 节），不允许自由生成列表外的分类词，多个分类用换行符 `\n` 分隔存储
- 长途旅游（每次不超过 7 天）必须在**同一自然周内**完整收尾（出发-途中-返回都在同一周内说完），不允许跨自然周；因此大纲生成时**不需要**参照上一自然周末尾的状态做续接

> ⚠️ **v1.8 变更说明**：v1.7 及更早版本中 `categories` 为 LLM 自由生成（仅给出"如'工作''旅游''逛街'"举例），因其参与 `dedup_hash` 去重计算，词汇不稳定会导致去重机制失效，v1.8 起改为固定枚举约束，详见 0.3 节。

#### 2.2.4 生活比例约束（软约束）

- 比例目标：主场城市本地生活约 70% / 周边短途出行约 20% / 长途旅游约 10%（每次不超过 7 天）
- **主场城市**为后台可配置项，默认"杭州"，可配置为其他城市（如角色人设改为北京，则比例围绕北京展开）
- 短途出行的具体目的地不做城市白名单配置，由 LLM 根据"周边短途"语境自行判断
- 统计周期：**自然月**。每月独立统计，不做跨月失败补偿——若某周大纲整体生成失败，该周自然消耗当月时间窗口，不追加补偿天数；自然月切换即重置统计，无需手动重置入口
- 约束力度：**软约束**。大纲生成时，将"当月累计的本地 / 短途 / 长途天数统计"作为参考信息注入 Prompt，由 LLM 自行把握节奏，不做系统级硬性天数分配
- 跨自然月统计口径：若某次周大纲覆盖的 7 天跨越两个自然月，按**每一天实际所属的自然月**分别计入对应月份统计，不整周打包计入大纲生成时所在月份

#### 2.2.5 已生成判定与后台手动生成（v1.8 补充参数化说明）

- "已生成"判定：按自然日落库计数，哪怕一个自然周只成功落库 1 天，也视为"该周已有大纲"
- 后台手动生成入口：仅当"当前自然周内，今天及以后的剩余自然日"范围内**完全没有任何一天落库**时，手动一键生成按钮才可用
- 手动生成范围：只生成"今天及以后"的剩余自然日（已经过去的自然日不补），例如今天是周三，手动生成则只生成周四至周日
- 只要范围内已有 ≥1 天落库，手动整周生成按钮**锁定**，只能通过后台对单日条目进行查看 / 新增 / 编辑 / 删除
- 缺失的自然日（无论是整周生成失败还是部分缺失），**不设任何自动重试机制**，完全依赖人工后台逐日补录

> ⚠️ **v1.8 新增说明**：LLM-01 的 Prompt 模板须支持**动态天数参数化**（`{{days_count}}`），而非固定按 7 天输出。常规每周日 23:00 自动触发时 `days_count = 7`；后台手动生成剩余自然日时，`days_count` 按实际剩余天数动态传入（如周三触发补录则 `days_count = 5`），输出的 JSON 记录条数与 `days_count` 一致。原 Prompt 模板中固定输出 7 条记录的写法需相应调整，具体结构详见《提示词规格文档》P-01-U。

#### 2.2.6 数据结构示例

```json
{
  "week_start_date": "2026-06-01",
  "days": [
    { "date": "2026-06-01", "city": "杭州", "categories": "工作" },
    { "date": "2026-06-02", "city": "苏州", "categories": "工作\n旅游" },
    { "date": "2026-06-03", "city": "苏州", "categories": "旅游" },
    { "date": "2026-06-04", "city": "杭州", "categories": "旅游" }
  ]
}
```

#### 2.2.7 周大纲生成 Prompt 模板（示例，后台可配置，v1.8 更新）

> 对应 **LLM-01**，Prompt 详细结构见《提示词规格文档》P-01-S / P-01-U。

```

你是"林小梦"生活轨迹规划助手，需要为她规划 {{days_count}} 个自然日（{{week_start_date}} 至 {{week_end_date}}）的生活大纲。

人设基础信息：
- 主场城市：{{home_city}}（后台配置项，默认"杭州"）
- 生活节奏参考比例（软约束，非强制对齐）：主场城市本地生活约70% / 周边短途出行约20% / 长途旅游约10%（每次出行不超过7天，且必须在同一自然周内完整收尾，不能跨自然周）

- 按日期顺序，输出 {{days_count}} 条记录
```

### 2.3 日场景细节生成层

#### 2.3.1 生成时机

- 每日 **00:20** 自动触发，生成"次日"的具体生活场景内容（**LLM-02**）
- **前置条件**：次日必须已存在周大纲锁定的条目（`city` + `categories`），若不存在，**当日直接跳过**，不触发生成、不计入失败重试
- 若存在大纲条目：生成场景，**成功标准为场景数 ≥ 2**
- 若生成场景数 < 2，视为失败，**00:30 自动重试一次**
- 若重试后场景数仍 < 2，彻底失败，**当日不发布任何朋友圈内容，无需告警通知，后台管理员可手动补发**
- 生成触发、跳过、成功（含场景数）、失败、重试均写入系统日志，可在后台日志页面查看

#### 2.3.2 强约束规则

当日场景生成必须**严格遵循**周大纲锁定的 `city` 与 `categories`，不能跑偏；自由发挥仅体现在场景的具体内容描述上。

#### 2.3.3 场景内容规则

- 数量：每日最少 2 个、最多 5 个生活场景，允许场景之间存在空白时间（体现真实休息状态）
- 时间范围：所有场景安排在 **06:00–20:00** 之间，时间分配合理且随机
- 时间字段：使用**时间段**而非时间点，如 `09:00-10:30`，更贴近真实生活节奏

#### 2.3.4 人设偏好与 venue_type 自由度说明（v1.8 补充）

- 喜好：咖啡馆、散步、逛市集、在家发呆、看展（对应 `lxm_likes` 配置项，详见 0.2 节）
- 厌恶：太吵的地方、人太多的景区（对应 `lxm_dislikes` 配置项，详见 0.2 节）
- `venue_type`（场所类型）**维持完全自由发挥**，不做枚举约束，由 LLM-02 根据场景内容自行生成（如"制片厂""教室""咖啡厅"，也可以是词表外的新词，如"宠物店"）。原因：`venue_type` 承载的是场景细节的真实感与丰富度，收紧为枚举会牺牲日场景生成的细节自由度，与 `categories` 因承担去重职责而必须锁定的情况不同
- **关联影响**：`venue_type` 的自由发挥会导致 4.4 图片生成环节的 `venue_type_img_keyword` 映射表无法保证 100% 覆盖，v1.8 已在 4.4.5 节补充对应的兜底机制（通过 `category` 兜底），不影响本节的场景生成自由度设计

#### 2.3.5 场景数据结构

| 字段 | 说明 |
|------|------|
| `scene_id` | 场景唯一标识 |
| `time_range` | 时间段，如 "09:00-10:30" |
| `city` | 当天所在城市，继承自周大纲锁定值，与大纲保持一致 |
| `category` | 本场景对应的内容分类，从当日大纲 `categories` 中选取（受 `categories_vocab` 约束） |
| `venue_type` | 场所类型，如制片厂 / 教室 / 咖啡厅，由日生成自由发挥，不做枚举约束 |
| `description` | 约 400 字详细描述，具体描述这个时间段、城市、分类、场所类型下林小梦做了什么、看到了什么、想了什么 |

> ⚠️ **v1.3 起移除字段**：`location`（具体地点融入 `description` 自由描述中）、`scene_tag`（去重逻辑完全交由下游 LIFE001 处理）、场景/计划级情绪字段（详见 2.4 说明）

#### 2.3.7 日场景细节生成 Prompt 模板（示例，后台可配置）

### 2.4 关于情绪与续接连贯性机制的说明（v1.3 调整）

- **情绪**：v1.3 起，Life Planner 不再生成任何结构化情绪字段，也不在日生成阶段做任何"延续昨天情绪"的处理。情绪完全融入每日 400 字场景描述的自由发挥中，按"独立连载小说"的方式处理，不追求跨天情绪一致性。
- **原"续接连贯性"机制**整体移除，城市连续性现由周大纲统一提前规划并强约束锁定，情绪连续性已取消。
- **说明**：本节所述"不生成结构化情绪字段"针对 LIFE000（生活计划层）。结构化的 `emotion_value` 由下游她的宇宙（PER003，LLM-03）在读取生活场景后独立生成，详见第三章 3.2 节。二者不是同一环节，请勿混淆。

### 2.5 模型选型

周大纲生成（LLM-01）与日场景细节生成（LLM-02）均使用 **DeepSeek**，具体版本由后台**各自独立**的模型配置项决定（LLM-01、LLM-02 分别可单独切换，互不影响），便于后续按节点灰度升级。

## 三、她的宇宙（PER003）

> ⚠️ **v1.9 命名变更**：本模块原名"世界观系统"，为避免与现有 IM 系统中按用户维度的 `world_state`（对话记忆机制）在后台管理入口层面产生混淆，中文业务概念统一改名为**"她的宇宙"**。模块编号 PER003 及数据库层面表名/字段名不变，详见 1.3 节说明。

### 3.1 定位

她的宇宙分为**动态层**和**静态层**两个相互独立的部分，均独立于其他模块运行，仅作为数据提供方被下游读取。

---

### 3.2 动态层 — 场景心理状态快照

#### 3.2.1 职责

为每个生活场景生成林小梦的主观感受与情绪解读，作为朋友圈文案生成的增强输入，使内容不只是事实描述，而是带有"林小梦式"的情绪色彩。

#### 3.2.2 生成时机与重试机制

**触发时机**

- 每日 **00:45** 自动触发，与 Life Planner 相互独立，不接收任何上游触发通知（**LLM-03**）
- 触发后首先检查当日 `life_plan.gen_status = 'ready'` 是否存在数据：
  - 若不存在（当日场景生成失败或跳过）→ **整个任务直接跳过**，不处理任何场景
  - 若存在 → 遍历当日所有场景，每个场景**单独**调用一次 LLM 生成心理状态快照
- 快照与场景通过 `scene_id` 关联，独立存储

**超时与重试规则**

- 每条快照单次 LLM 调用超时时间：**45 秒**
- 失败分为两类，处理逻辑完全一致：
  - **技术失败**：超时、HTTP 5xx、网络断开等
  - **内容失败**：LLM 返回成功但缺少必要字段（如 `emotion_value` / `focus_tag` 为空）或返回格式不符合预期
- 重试策略：失败后**立即重试**，最多 **3 次**（含首次，即最多额外重试 2 次）
- 所有重试均失败 → 标记该快照 `gen_status = 'failed'`，**不阻断**其余场景的继续处理

#### 3.2.3 快照内容结构（v1.8 更新）

每份快照包含：

- **自然语言感受**（A）：林小梦对这个场景的主观描述，约 1–3 句话，口语化
- **结构化字段**（B）：
  - `emotion_value`：情绪值标签，**优先从后台维护的 `emotion_vocab` 情绪词汇表中选取**（当前 14 个核心词：慵懒、雀跃、低落、平静、焦虑、满足、怀念、烦躁、期待、感慨、孤独、无聊、迷茫、释然），若均不能准确贴合当前场景的细腻状态，可使用更贴切的词汇（核心词优先+自由兜底机制，详见 0.4 节）
  - `focus_tag`：当前关注点（如：想念某人、享受孤独、对生活的感慨）
  - `worldview_trigger`：触发的价值观标签（如：自由、慢生活、人情味），用于第二部分她的宇宙话题生成

> ⚠️ **v1.8 变更说明**：v1.7 中 `emotion_value` 举例词汇为 4 个，且未明确是否约束选取范围。v1.8 起明确采用"核心词优先+自由兜底"机制，核心词表扩展为 14 个，详见 0.4 节。该字段的自由生成能力不受影响，仅新增了图片生成环节的兜底保护机制。

#### 3.2.4 与朋友圈生成的对应关系

- 朋友圈生成时，**一对一抽取**场景及其对应的心理状态快照作为输入
- 若当日场景数量（N）少于计划朋友圈发布数量，**朋友圈数量自动降为 N 条**，与场景数对齐
- 若某个场景的快照生成失败（`gen_status = 'failed'`），对应朋友圈**降级生成**（仅用场景 `description` 数据，不带快照）；即使无快照，LLM 仍可从约 400 字场景描述中感知情绪色彩，文案不会缺乏情绪，只是少了结构化锚点的精准引导
- **她的宇宙快照是朋友圈的"情绪增强输入"，而非情绪的唯一来源**（详见 LIFE001 第 4.3.2 节）

#### 3.2.5 日志记录

以下关键节点均写入系统日志，可在后台日志页面查看：

| 事件 | 级别 | 说明 |
|------|------|------|
| 任务触发（00:45） | INFO | 记录触发时间 |
| life_plan 检查结果 | INFO | 当日 ready 场景数量；若为 0 则记录任务跳过 |
| 单条快照生成开始 | INFO | 记录 scene_id |
| 单条快照生成成功 | INFO | 记录 scene_id、耗时、第几次尝试 |
| 单条快照生成失败（中途重试） | WARN | 记录 scene_id、失败类型（超时/内容校验失败）、当前重试次数 |
| 单条快照最终失败 | ERROR | 记录 scene_id、已达最大重试次数，标记 failed |
| 静态层 event 写入 | INFO | 记录 event_name、操作结果（新增 / 已存在跳过） |
| 整体任务完成 | INFO | 记录成功快照数、失败快照数 |

#### 3.2.6 后台管理

- 可查看每日所有场景对应的心理状态快照内容
- 支持手动编辑、删除指定快照
- 快照被编辑后，若对应朋友圈尚未生成，下次生成时使用修改后的版本

---

### 3.3 静态层 — 她的宇宙事件库

#### 3.3.1 职责与数据来源

记录林小梦对各类事件/话题的固定看法，作为底层人设积累，供未来业务接入使用。**当前阶段不接入任何下游生成流程。**

**数据来源（v1.4 新增）**：由她的宇宙处理每个生活场景时，在**同一次 LLM 调用（LLM-03）**中自动生成并写入。每处理一个场景，系统提取该场景触发的 `worldview_trigger` 话题，并生成林小梦对该话题的固定看法，upsert 写入事件库。相同 `event_name` 已存在时**保留首次写入版本，不覆盖**，保证人设一致性。

管理员也可在后台手动增删改查，不依赖自动生成。

#### 3.3.2 生成规则（v1.8 更新）

- **触发时机**：处理每个 scene 时，在生成动态快照的**同一次 LLM 调用（LLM-03）**中额外输出静态事件看法
- **event_name 生成要求**：描述性短语，不能过于简短，需让阅读者清晰理解话题内容（如"在人多景区的感受与应对方式"，而非仅"景区"）；由 LLM 根据当前场景内容自行生成
- **event_view 生成要求**：100–200 字，需包含以下三个维度：
  - 林小梦的**核心态度**（喜欢 / 排斥 / 矛盾 / **无感**）
  - 触发该态度的**典型场景或具体细节**
  - 她在这类情境下**通常会怎么做 / 怎么选择**
- **重复处理**：以 `event_name` 为唯一键，已存在则不覆盖（INSERT IGNORE 语义），保留首次生成版本
- **Prompt 内容**：具体提示词结构见《提示词规格文档》P-03-U，后台可配置（见第九章）

> ⚠️ **v1.8 变更说明**：核心态度选项由"喜欢/排斥/矛盾等"明确扩展为四个选项，新增"无感"，用于覆盖林小梦对某些话题没有明显倾向、仅做中立描述性观察的情况。原三选一设计会导致这类场景被强行套用最近似的态度，造成她的宇宙内容失真。

#### 3.3.3 数据结构

| 字段 | 说明 |
|------|------|
| `event_name` | 话题名称（唯一键，描述性短语） |
| `event_view` | 林小梦对该话题的看法（100-200字，含核心态度/典型场景/行为倾向） |
| `source_scene_id` | 首次触发生成的 scene_id，可溯源至来源场景 |
| `created_at` | 首次写入时间，后台可按时间查看与溯源 |
| `updated_at` | 最后更新时间 |

- 重复 event_name：不覆盖，保留首次版本（INSERT IGNORE）
- 管理员手动编辑后，`updated_at` 更新，`source_scene_id` 保持原始记录不变

#### 3.3.4 后台管理

- 支持增删改查
- 可查看每条记录的 `source_scene_id` 和 `created_at`，追溯该观点的生成来源与时间
- 当前阶段仅作数据积累，不影响任何生成逻辑

### 3.4 Prompt 配置

她的宇宙所有 Prompt 后台可独立配置（详见第九章），与 Life Planner 和朋友圈模块的 Prompt 完全分离，互不影响。

### 3.5 模型选型

动态快照生成与静态事件看法生成（均为 **LLM-03**）使用 **DeepSeek**，具体版本由后台独立的模型配置项决定，便于后续切换。

## 四、朋友圈内容生成引擎（LIFE001）

### 4.1 生成时机

- 每日 **01:00** 独立触发朋友圈内容生成任务（**LLM-04**）
- 触发时检查当日生活计划是否存在：存在则正常生成，不存在则跳过，当日不发布
- 朋友圈模块与 Life Planner、她的宇宙模块相互独立，不接收任何上游触发通知
- **时序说明**：她的宇宙于 00:45 触发，预计在 00:45–01:00 之间完成所有场景快照生成（每条 45s 超时、最多 3 次）。01:00 触发时快照应已处于 `ready` 或 `failed` 终态，不存在 `generating` 中间态。若读取时仍为 `generating`，视同 `failed` 降级处理。

### 4.2 发布计划生成规则

#### 4.2.1 发布次数

每日随机生成 2 或 3 次发布，**最终发布数量取随机值与当日去重后可用场景数量的较小值**：

| 随机次数 | 时间窗口 |
|----------|----------|
| 2 次 | 10:00–12:00、15:00–20:00 |
| 3 次 | 10:00–12:00、15:00–20:00、20:00–23:00 |

- 每个窗口内随机选取一个具体时间点作为实际发布时间（均匀随机，无权重）
- 内容在 01:00 提前生成完毕，到达对应时间点后由查询层过滤自动对用户端可见
- 三个窗口的时间区间后台可配置（详见第九章）

#### 4.2.2 场景与朋友圈的对应关系

- 每条朋友圈**一对一**对应一个生活场景及其她的宇宙快照
- 场景按当日时间顺序依次分配给各发布窗口
- 已分配的场景不重复使用

### 4.3 文案生成规则

#### 4.3.1 文风要求（反 AI 风格）

- 允许有停顿（……）、少量错别字、废话、情绪跳跃和克制的 emoji，具体规则见 `lxm_writing_style` 配置项（0.2 节）
- 绝对不能出现任何针对特定用户的引用，所有内容对任何用户都合理
- 不出现 `lxm_content_limits`（0.2 节）中列出的内容禁区话题

#### 4.3.2 情绪增强输入机制 & emotion 字段输出路径

**输入增强（LLM-04 的可选输入段）：**

LIFE001 生成文案时，以她的宇宙的心理状态快照作为**可选增强输入**注入 Prompt，非必要依赖：

| 快照状态 | 处理方式 |
|----------|----------|
| `gen_status = 'ready'` | 将 `emotion_value` / `focus_tag` / `feeling_text` 注入 Prompt 可选段，提供精准情绪锚点 |
| `gen_status = 'failed'` 或不存在 | 省略快照段，LLM 直接从 LIFE000 场景 `description`（约 400 字）中感知情绪色彩 |

**`feed_post.emotion` 字段写入路径（双路径，均在 LLM-04 调用环节处理）：**

| 快照状态 | emotion 字段来源 | LLM-04 输出 schema |
|----------|----------------|-------------------|
| `gen_status = 'ready'` | 直接复制 `worldview_snapshot.emotion_value`，LLM-04 无需输出 emotion | `{post_text, hashtags}` |
| `gen_status = 'failed'` 或不存在 | 由 LLM-04 从场景 `description`（约 400 字）中判断生成，同样遵循"核心词优先+自由兜底"机制（详见 0.4 节），**与文案同次调用附带输出，不发起独立 LLM 调用** | `{post_text, hashtags, emotion}` |

**单套 Prompt 模板，有无快照均适用**：快照段为可选注入段，存在时提升情绪精准度与"林小梦味"，不存在时文案仍有情绪，只是少了结构化锚点；emotion 字段在快照缺失时由 LLM-04 自动判断并附带输出。

#### 4.3.3 内容分类占比

以下占比以 **Prompt 软约束**方式注入，由 LLM 在生成时自行把握长期节奏，不做系统级强制分配：

| 类型 | 参考占比 |
|------|---------|
| 日常碎碎念 | 40% |
| 情绪 / 感受流 | 25% |
| 她的宇宙延伸 | 20% |
| 生活记录 | 15% |

#### 4.3.4 话题标签（#话题）

- AI 在 LLM-04 调用时**附带输出**，作为结构化字段 `hashtags` 一并返回，不发起独立 LLM 调用
- 客户端以特殊颜色标记展示
- 话题数量随机，概率分布后台可配置（详见第九章）：
  - 0 个：50%
  - 1 个：30%
  - 2 个：10%
  - 3 个：10%
- @功能预留，**第一版本不做**

### 4.4 图片生成规则

#### 4.4.1 图片数量分布

| 图片数量 | 占比 |
|----------|------|
| 0 张（纯文字） | 30% |
| 1 张 | 35% |
| 2–3 张 | 25% |
| 4 张 | 10% |

#### 4.4.2 图片类型权重

| 类型 | 权重 |
|------|------|
| 人物自拍 | 40% |
| 日常生活 | 30% |
| 风景 / 旅行 | 20% |
| 情绪表达 | 10% |

#### 4.4.3 生成策略（v1.8 更新：非人物图按子类型拆分）

**图片类型决策（多图时）：**
- 每条朋友圈整体**抽签确定一个图片类型**（按 4.4.2 权重随机），该条帖子内所有图片均走同一类型的生成策略
- 图片类型确定后存入 `feed_post.image_type` 字段

**人物 / 自拍类（图生图）：**
- 使用后台配置的固定参考基准图，通过 LiblibAI IMG 1 图生图接口生成
- 保证人物形象跨帖一致性
- 基准图后台可更换；更换后新生成内容按新基准，历史内容不变
- 图生图相似度参数：若 LiblibAI 接口支持，后台提供配置项
- 存储路径见 11.2 节

**非人物类（文生图，v1.8 拆分为三个独立子类型）：**
- 使用星流 Star-3 Alpha，照片级真实感
- **v1.8 起，非人物类按 `image_type` 的三个子值（`daily`/`scenery`/`emotion`）拆分为三个独立的 Prompt 配置项**（对应《提示词规格文档》P-13a/P-13b/P-13c），各自独立配置与调试，互不影响：
  - `daily`（日常生活）：强调生活细节与静物质感，无人物
  - `scenery`（风景/旅行）：强调地方感与旅行随拍质感，无人物
  - `emotion`（情绪表达）：以氛围感、光影、静物细节传达情绪，强调情绪共鸣而非叙事
- 拆分原因：三种子类型在构图逻辑和内容侧重上差异明显，独立配置便于精细调优、故障隔离度更高（单个子类型模板出问题只影响该类型，不影响其余两类）

**并行生成与超时降级策略：**
- 同一条朋友圈中，人物组与非人物组图片**并行生成**
- 各组**独立降级**：某组失败仅该组缺失，其余组正常发布
- **超时规则**：单张图片轮询上限 **3 分钟**；整条朋友圈所有图片整批上限 **15 分钟**，超时后对已完成图片正常打包发布，未完成图片视为失败降级
- 所有图片生成完成后统一打包上传 OSS，通过 OSS 链接聚合成完整朋友圈
- 若所有图片均失败，自动降级为纯文字发布，不展示占位图

**临时文件处理：**
- 图片从 LiblibAI 下载后，先写入本地临时文件用于 WebP 压缩处理，压缩完成后上传至 OSS
- 临时文件在该张图片上传 OSS **成功或失败后立即清理**，不依赖定时批量清理任务；多图并行生成时，各临时文件独立管理、独立清理，互不影响
- 完整存储与分发架构详见 11.2

#### 4.4.4 真实感增强

- 自拍图加入轻微模糊、构图不完美、光线不均等"假瑕疵"
- 模拟 iPhone 拍摄风格与 EXIF 信息
- 同一条朋友圈多张图片保持视觉一致性（说明：非人物类多图的一致性目前依赖同批次共享的 `city`/`season`/`time_period_light`/关键词等输入实现风格趋同，非像素级强绑定；人物类多图靠固定基准图强绑定；**2026-07-10**：另按序号追加构图变体后缀 + 独立 seed，避免画面雷同，见 TB-LF-008）

#### 4.4.5 图片关键词映射与兜底机制（v1.8 新增）

图片生成 Prompt 组装时，需要将若干中文场景字段转换为英文关键词，映射来源与兜底策略如下：

| 输入字段 | 映射表 | 覆盖情况 | 兜底策略 |
|---------|--------|---------|---------|
| `venue_type`（自由发挥，见 2.3.4） | `venue_type_img_keyword` | **不完整**，因 `venue_type` 无枚举约束，LLM-02 可能随时生成表外新词 | 查不到时，退一级查该场景所属的 `category`（固定枚举，见 0.3 节）对应的 `category_img_keyword` 映射表，该表词条数量有限，可做到 100% 覆盖 |
| `emotion_value`（核心词优先+自由兜底，见 0.4 节） | `emotion_img_keyword`（P-12用）/ `emotion_atmosphere_desc`（P-13c用） | 覆盖 14 个核心词，非核心词无法覆盖 | 查不到时，使用固定的 `emotion_fallback_img_keyword` / `emotion_fallback_atmosphere_desc` 通用兜底关键词组 |
| `category` | `category_img_keyword`（v1.8 新增，见 9.4） | 完全覆盖（词表固定，约 10 个） | 无需兜底 |

**设计原则**：兜底不是"退化为无关的通用词"，而是"退到更粗粒度但仍与当天实际内容相关的层级"，确保图片主题始终与朋友圈文案内容方向一致，不产生文不对图的违和感。

### 4.5 内容去重机制

#### 4.5.1 结构化去重

以场景的 **`venue_type + category + city` 三字段组合**作为去重 key：

- 生成前检查：当前场景的三字段组合，在过去 **7 天内**已发布的朋友圈中是否存在相同 `dedup_hash`
- 若命中去重 → 跳过该场景，不为其生成朋友圈，当日可发布数量对应减少
- `dedup_hash` 存储在 `feed_post` 表中，基于 `venue_type + category + city` 三字段组合生成（如 MD5 哈希），用于后续查重

#### 4.5.2 文本相似度过滤（v1.9.1 定案）

- 在结构化去重基础上，对生成后的文案与近期已发布朋友圈历史进行**文本相似度比对**
- 基于已发布朋友圈历史内容本身，不依赖生活计划表
- **比对窗口**：近 **7 天**内已发布（`generation_status = ready` 且对用户可见）的 `feed_post.content_text`，与 4.5.1 结构化去重窗口对齐
- **算法**：轻量**字符级相似度**（如 Jaccard 系数或最长公共子串比率，实现择一即可）
- **阈值**：相似度 **≥ 0.75** 视为重复，**放弃该条发布**（不自动改写重试）
- **配置**：阈值写入 `admin_config`（建议 `config_key = feed_text_similarity_threshold`），后台可调
- 命中后记录 INFO 日志（含 scene_id、命中 post_id、相似度分值），该条当日可发布数量对应减少

### 4.6 旅游叙事上下文注入

当当日场景所在城市为**非主场城市**时，LIFE001 自动计算当日在旅途中的阶段，并以软提示方式注入 Prompt，使文案自然体现行程进展感：

**阶段判断逻辑**（基于周大纲城市序列）：

| 昨日城市 | 今日城市 | 明日城市 | 判断结果 |
|----------|----------|----------|----------|
| 主场城市 | 非主场城市 | 非主场城市或不存在 | 出发日 |
| 非主场城市 | 非主场城市 | 非主场城市 | 途中 |
| 非主场城市 | 非主场城市 | 主场城市 | 返回日 |
| 主场城市 | 非主场城市 | 主场城市 | 一日游 |

**注入内容（3 项，均为软提示）**：
1. 本周完整城市序列（让 LLM 知道完整旅程上下文）
2. 今天是旅程的第几天 + 当前阶段（出发 / 途中 / 返回 / 一日游）
3. 一句方向引导提示（如出发日强调期待感，返回日强调留恋感，途中强调沉浸感）

**约束方式**：软提示，LLM 自行将阶段感融入文案，不做硬性规定，不要求明确点明"今天出发了"等字样

**主场城市**：与 LIFE000 使用同一后台配置项

对应 Prompt 可独立后台配置（见第九章），具体 Prompt 结构见《提示词规格文档》P-04-U 及 P-05。

- 超出范围的历史数据不返回给客户端（数据仍存于数据库，不删除）

网络异常时显示"网络不太好，稍后再看"友好提示。

### 5.6 首页朋友圈入口与徽标逻辑

#### 5.6.1 入口位置

首页"朋友圈入口"替换原"记忆入口"位置，入口支持双徽标提示系统，让用户直观感知新内容与未读回复。

#### 5.6.2 徽标状态

| 状态 | 徽标展示 |
|------|---------|
| 无新动态 | 无徽标 |
| 有新朋友圈（用户本次会话后首次进入前未访问） | **[New]** 标签 |
| 有未读 LXM 评论回复 | **数字角标**（如 `3`），逻辑同 IM 未读消息计数 |
| 两者同时存在 | **仅展示数字角标**，[New] 标签隐藏 |

#### 5.6.3 点击行为

| 进入时状态 | 行为 |
|-----------|------|
| 仅有新朋友圈（[New]） | 进入列表顶部（倒序，最新帖在顶），进入即视为已读，[New] 消失 |
| 仅有未读 LXM 回复（数字角标） | 自动滚动至最近一条未读 LXM 评论回复位置，停留即算已读 |
| 两者同时存在（数字角标优先） | 优先自动滚动至最近一条未读 LXM 评论回复位置 |

#### 5.6.4 已读判定规则

- **新朋友圈**：进入朋友圈页面即视为全部已读，[New] 标签消失；同时将 `users.last_feed_entered_at` 更新为当前时间
- **未读 LXM 评论回复**：对应回复内容进入用户视口（曝光）即视为已读，数字角标相应减少；未全部看完时，下次进入仍定位至剩余最近一条未读位置，直至角标归零

#### 5.6.5 [New] 徽标存储实现（v1.9.1 新增）

- **判定字段**：`users.last_feed_entered_at`（用户最近一次进入朋友圈页的时间；`NULL` 表示从未进入）
- **展示条件**：存在 `scheduled_publish_time > last_feed_entered_at`（或 `last_feed_entered_at IS NULL`）且对用户可见的新帖时，展示 [New] 标签
- **清除时机**：用户进入朋友圈页面时写入当前时间，[New] 消失（与 5.6.4 一致）
- **数字角标**：仍独立由 `feed_comment.lxm_reply_read_at IS NULL` 统计，与 [New] 互不替代

### 5.7 已读状态追踪

LXM 对用户评论的回复为每个用户私有内容（其他用户不可见），已读状态独立追踪，存储于 `feed_comment.lxm_reply_read_at` 字段：

- `NULL`：LXM 已回复但用户尚未看到，计入首页未读角标数量
- 有时间戳：用户已看到该回复，不计入未读角标

**写入时机**：LXM 评论回复内容进入用户视口时，客户端上报已读事件，后端将对应记录的 `lxm_reply_read_at` 更新为当前时间。

### 5.8 帖子可见性与互动关系

- **可见即可互动**：凡在历史可见范围内的帖子，均可点赞与评论，不设额外互动时间窗口
- **帖子隐藏（`is_visible = 0`）**：对用户完全不可见，不可互动；管理员恢复展示后，该帖子下的历史点赞数与评论数据**完整复原**，不丢失

### 5.9 新帖实时提示（SSE，v1.9.3 定案）

用户**已打开朋友圈页面**时，后台有新帖到达发布时间且对用户可见，页面顶部展示「X 条新动态」提示条（交互细节见《朋友圈页面展示逻辑规范》5.1）。与首页 **[New] 徽标（5.6）独立**：[New] 在用户**未进入**朋友圈时由 `last_feed_entered_at` 判定；SSE 仅服务**已在页内**的即时提示。

#### 5.9.1 架构原则

- **独立服务**：新建 `GET /api/feed/events`（JWT 鉴权，`text/event-stream`），与对话链路 `POST /api/chat/send` / `resend` 的请求级 SSE **零复用、零混流**
- **连接范围**：仅朋友圈页 `mounted` 时建立 `EventSource`，`unmounted` 断开；**不对**首页/IM 页维持长连接
- **推送触发**：帖子 `generation_status = ready`、对用户可见（`is_visible = 1` 且在 5.4 可见范围内）、且 `scheduled_publish_time ≤ now` 时，向当前在线且已连接 SSE 的用户推送

#### 5.9.2 事件格式

每行一条 JSON（与对话 SSE 事件命名空间隔离）：

```json
{"type":"feed_new","delta":1}
```

| 字段 | 说明 |
|------|------|
| `type` | 固定 `feed_new` |
| `delta` | 本次新增条数，通常为 `1`；短时间连发可多条事件或合并为 `delta:N`，前端仅累加 |

**不传**帖子正文、图片 URL、`post_id`（省流量、减泄露面）。

#### 5.9.3 前端行为

- 本地累加 `pendingNewCount`，渲染「X 条新动态」提示条
- **点击提示条**：滚回列表顶部 → 调用 Feed 列表 API 拉取最新一页 → `pendingNewCount` 清零
- 用户已在顶部未滚动：**仍显示提示条**，不自动插入列表（与 UI 规范「点击后加载」一致）
- **断线**：指数退避重连；重连成功后**不补发**历史事件，依赖下拉刷新兜底

#### 5.9.4 与 [New] 的分工

| 机制 | 场景 | 判定 |
|------|------|------|
| 首页 [New] | 用户不在朋友圈页 / 离开后再次进入首页 | `scheduled_publish_time > last_feed_entered_at`（5.6.5） |
| SSE 提示条 | 用户正在朋友圈页内浏览 | `feed_new` 事件累加 |

二者互不替代；进入朋友圈页写入 `last_feed_entered_at` 后 [New] 清除，与是否点击 SSE 提示条无关。

## 六、互动系统（LIFE004）

### 6.1 点赞功能

#### 6.1.1 点赞数计算规则

```
展示点赞数 = base_likes × like_multiplier + real_likes
```

- `base_likes`：内容生成时随机确定（1–8），固定不变
- `like_multiplier`：内容生成时随机确定（1–3），固定不变；**仅放大 base_likes，不放大真实点赞数**，避免真实点赞累积后展示数过度膨胀
- `real_likes`：用户真实点赞累计数，用户点赞 +1，取消点赞 -1

示例：base_likes=5，like_multiplier=3，real_likes=20 → 展示数 = 5×3+20 = **35**

#### 6.1.2 取消点赞行为

- 支持取消点赞，`real_likes` 随之 **-1**
- 取消点赞**不撤回**已触发的 LXM 主动 IM 消息：点赞行为触发 IM 是既成事实（行为已发生、消息可能已在队列中），撤回无意义且实现复杂

#### 6.1.3 林小梦对点赞的感知回复

用户点赞后，按以下规则判断是否触发 LXM 通过 IM 发送一句简短反应（**LLM-06**），回复发生在 **IM 对话系统**中，不在朋友圈评论区。话术统一使用 **P-07**（特殊档与常规档共用同一 Prompt 模板）。

> ⚠️ **v1.9.2 定案**：取消原「终身首次点赞」规则；改为**点赞特殊触发窗口**（业务配置，非「新用户」用户标签）。窗口起算锚点为 **`users.created_at`**。

**触发概率（每次点赞独立判定）**

| 档位 | 条件 | 触发概率 | 延迟 |
|------|------|----------|------|
| **特殊档** | `now - users.created_at ≤ like_aware_special_window_hours` **且** `relationship.like_aware_special_used_count < like_aware_special_max_count` | **100%** | `like_aware_special_delay_sec`（后台可配，建议默认 30 秒） |
| **常规档** | 不满足特殊档 | **30%** | 按当前关系阶段延迟窗口（见 6.3 表「点赞 IM 常规」行） |

**特殊档计数**：`LIKE_AWARE` **入队成功**（`agent_aware_queue.status = pending`）时，`like_aware_special_used_count` **+1**（与特殊档绑定；`failed` 不回退，便于测试可后台手动重置计数）。

**去重规则**：同一用户对**同一帖子**，点赞 IM 触发最多一次；用户取消点赞后再次点赞，不重复触发。实现上查询 `agent_aware_queue` 是否已存在该 `user_id` + `post_id` + `trigger_type = LIKE_AWARE` 的记录（含 `pending`/`sent`/`failed`，不因 `failed` 自动重试）。

**点赞关系存储**：per-user 点赞状态存于 `feed_like` 表（见 11.4），用于前端 👍 高亮、`real_likes` 准确 ±1、取消点赞；与 IM 触发去重相互独立。

**触发后时序**：

1. 系统判定触发（概率命中或特殊档必中）后，按档位计算 `due_at`，写入独立排队表
2. 到达计划时间后，独立轮询任务消费该条记录，启动 LLM-06 生成 IM 消息内容（**P-07**）
3. 生成完毕后写入 `agent_message` 表（`trigger_type = LIKE_AWARE`），发送至 IM 对话界面

**关系阶段升级时**：已在排队表中的记录**不重算延迟**，按入队时的阶段与档位发出。

> ✅ **v1.9 架构确认（原 v1.8 标注为"待补充说明"，本轮已确认定案）**：

> LLM-06 与 LLM-07（已读感知，见第七章）采用**与现有 AGE003（`agent_service.py` 全量扫描判定）及 Future 槽（`future_handler.py` + `step8_subchain.py`）完全独立、零代码耦合的第三条业务线**，具体架构如下：

> - **触发与排队**：新建独立排队表，支持同一用户多条并发排队（区别于 Future 槽的单槽位设计，因同一用户可能短时间内对多条不同帖子分别点赞/已读，需各自独立计时，不能相互覆盖）
> - **独立轮询**：新建独立定时任务扫描排队表中到期记录，架构参考 Future 槽的"到期轮询消费"模式，但代码完全独立
> - **独立生成**：新建轻量生成函数，直接对接 P-07（点赞感知）/ P-08~P-11（已读感知常规四档）/ P-14（已读感知特殊档，见 7.2）Prompt 单次调用 DeepSeek，**不复用 Step8 子链路**（Step8 依赖 `future_action` 变量且是面向"续接对话话题"设计的重量级管线，与本场景不匹配）
> - **明确不复用的判定逻辑**：不查 `calculate_action_score` 评分门槛、不查每日消息总量上限、不查发送间隔限制、成功后不回填任何共享计数器、不做黑名单检查（封禁用户在鉴权中间件层已被拦截，不可能产生点赞/已读事件，此检查冗余）
> - **落库与展示复用**：最终生成内容仍写入现有 `agent_message` 表（新增 `trigger_type = LIKE_AWARE` / `READ_AWARE`），`action_score` 固定写 **0**（仅满足表字段非空，**不参与**任何发送判定，与 P0-P4 评分逻辑无关），落库前调用现有 `timeline_seq_service.allocate_sort_seq` 分配排序号，保证在 IM 时间线中与聊天消息正确穿插排序；前端完整复用现有未读消息列表、未读计数（`/api/agent/unread-count`）、标记已读接口，不新建展示层
> - **设计理由**：点赞/已读感知是用户对内容的即时反馈，与"AI 主动判断是否发起打扰"（P0-P4/Future 槽场景）本质不同，不应共用同一套节流哲学；PRD 已通过特殊窗口+次数、30% 概率、同帖同用户仅一次、按关系阶段延迟窗口等产品层面的规则实现节流，无需再叠加通用防打扰机制
> - **后台管理入口**：见新增 10.8 节

对应 Prompt 与特殊档参数后台可配置（详见第九章 9.5）。

### 6.2 评论功能

#### 6.2.1 评论规则

- **评论区为用户私有（v1.7 新增）**：每位用户仅能看到自己发布的评论及对应的 LXM 回复，不同用户之间互不可见——即同一帖子下，不同用户打开评论区看到的内容是各自独立的
- 用户只能回复林小梦，不能互相回复；**交互（v1.9.1 明确）**：仅通过底部行 **💬 icon** 弹出输入框发评；**不支持**点击某条评论行切换为「回复 TA」或嵌套回复他人；用户可对同帖**多次发评**（平铺追评），林小梦以「回复 @我」接上，每条评论独立一行写入 `feed_comment`
- 评论按时间**正序**展示（平铺列表）
- 回复样式完整复刻微信朋友圈评论形态，且当前用户查看自己的评论区时，**用户自己的评论落款与 LXM 回复中的"回复 @XX"均统一显示为"我"**（不取真实昵称，天然规避新用户尚未被系统学习到称呼时的边界情况），如：
  ```
  我：好喜欢这里~
  林小梦 回复 @我：真的超美的对吧！
  我：你下次还会去吗？
  林小梦 回复 @我：应该会吧，好想再去一次……
  ```
  **展示层强制规则（v1.9.1）**：用户侧评论区落款与 LXM「回复 @XX」引用**一律显示「我」**，不使用 `users.username`、真名、偏好称呼或任何账号 ID；LLM 生成侧称呼仍走 `user_hobby_name`/`user_real_name`（皆空则 Prompt 省略称呼段，自然用「你」）。详见《朋友圈页面展示逻辑规范》4.5 节。

#### 6.2.2 林小梦回复规则

- 林小梦**必回**所有用户评论，回复出现在**朋友圈评论区**，不走 IM（**LLM-05**）
- 按当前关系阶段的延迟窗口（见 6.3）启动 LLM-05 生成回复
- **全局首次评论 override**：用户该生命周期内第一次发出任意评论时，延迟固定为 **30 秒后启动 LLM-05 流程**；是否首次由 `relationship.has_ever_commented_feed` 判定（见 11.4），首次评论成功后置为 `1`，便于测试重置
- **v1 实现范围（v1.9.1 定案）**：MVP 为**简单回复**——单次 LLM-05（P-06）输入仅含朋友圈正文、用户评论、关系阶段、称呼（有则注入）；**不**读取 `user_interest_memories`、**不**对用户评论做 Embedding 向量检索、**不**触发 IM 对话 Step6 记忆提取/落库
- **二期规划（延后）**：统一对评论内容做记忆提取，并优化回复前记忆引用规则（兴趣偏好类，如"你也喜欢咖啡嘛"）；P-06 中 `{{user_interest_memories}}` 可选段结构保留，v1 恒为空不注入
- **禁止**引用地点、事件、情绪等私密信息（二期启用记忆引用时仍适用）
- **称呼来源**：回复中若需要提及用户称呼，**复用现有 `relationship_info.user_hobby_name`/`user_real_name` 机制**（详见《提示词规格文档》），而非独立的"用户昵称"字段——该机制本身已处理"两者皆空时不提及称呼"的兜底情况，无需额外设计
- **回写方式**：LLM-05 输出为纯文本（不做 JSON 包装），系统直接将该文本写入 `feed_comment.lxm_reply`（TEXT 字段），同时写入 `lxm_reply_at`（回复时间）并将 `gen_status` 更新为 `ready`；前端直接读取 `lxm_reply` 字段渲染评论区回复内容，无需额外的结构化解析
- 对应 Prompt 后台可配置（详见第九章）

**失败处理机制**：

- 单次 LLM-05 调用超时时间：**45 秒**
- 技术失败（超时 / HTTP 5xx / 网络断开）与内容失败（缺少必要字段 / 格式不符）处理逻辑完全一致
- 失败后**立即重试**，最多 **3 次**（含首次，约 135 秒内完成全部尝试）
- 3 次均失败 → **静默不回**，`gen_status = 'failed'`，后台记录 ERROR 日志，管理员可在后台评论管理页手动补发

## 七、已读感知系统（LIFE005）

### 7.1 触发规则与点赞互斥

- 用户停留某条 Feed 即算已读，不设秒数门槛
- **点赞 IM 互斥（v1.9.2 新增）**：触发已读 IM 判定前，先查近 **`read_suppress_after_like_im_hours`**（默认 **6** 小时）内，该用户是否存在 **概率命中且已入队** 的 `LIKE_AWARE` 记录（`agent_aware_queue.status IN ('pending','sent')`）；若存在 → **本次不发**已读 IM（避免短时内点赞 IM 与已读 IM 叠发）。**未入队的 30% 未中点赞不计入**。
- 通过互斥检查后：同一条 Feed 只触发一次已读感知；实现上查询 `agent_aware_queue` 是否已存在该 `user_id` + `post_id` + `trigger_type = READ_AWARE` 的记录（含各 status，同帖不重复入队）
- 若用户同时浏览多条未读 Feed，取**最近发布的一条**作为触发素材

### 7.2 已读特殊触发窗口（v1.9.2 定案，原 7.2「注册当日」废止）

> ⚠️ 本节为**已读感知业务特殊档**配置，锚点 **`users.created_at`**，与点赞特殊档（6.1.3）**独立配置、独立计数**，非产品层「新用户」标签。

- 条件：`now - users.created_at ≤ read_aware_special_window_hours`（默认 **48** 小时）**且** `relationship.read_aware_special_used_count < read_aware_special_max_count`（默认 **1** 次）
- 待遇：入队 `READ_AWARE`，`due_at = 触发时刻 + read_aware_special_delay_sec`（默认 **60** 秒）；生成时**固定走 P-14**（帖文相关、好接话，见《提示词规格文档》），**不看**关系四档
- **计数**：`READ_AWARE` **入队成功**时 `read_aware_special_used_count` **+1**
- 窗口外或次数用尽后，走 **7.3 常规档**

### 7.3 常规触发逻辑

- 已读后 **30 分钟–2 小时**内随机触发主动对话消息（**LLM-07**），延迟区间按当前关系阶段后台可配
- 生成时按关系阶段选用 **P-08～P-11** 四档话术
- 走**独立的排队表 + 独立轮询任务**发起，与点赞感知（LLM-06）共用同一套架构，与 P0-P4/Future 槽零代码耦合
- 消息推送后，用户可通过朋友圈右上角消息通知入口查看

> ✅ **v1.9 架构确认（原 v1.8 标注为"待补充说明"，本轮已确认定案）**：LLM-07 的完整发送链路方案与 6.1.3 节 LLM-06 完全一致（独立排队表 + 独立轮询 + 独立生成函数 + 复用 `agent_message` 落库与 `sort_seq` 排序 + 不查评分/总量/间隔/黑名单），详见 6.1.3 节展开说明，此处不再重复。

### 7.4 话术映射（按关系阶段，v1.9 更新：三档扩展为四档）

| 档位 | 适用 | 话术 |
|------|------|------|
| **特殊档** | 7.2 窗口内前 N 次已读 | **P-14**（紧扣 `post_text`，引导用户好接话，详见《提示词规格文档》） |
| 陌生 | 7.3 常规 | 示例："你今天有上线" → **P-08** |
| 朋友 | 7.3 常规 | 示例："我感觉你刚刚看了……" → **P-09** |
| 亲密 | 7.3 常规 | 示例："你是不是刚看到我那条了，想说什么" → **P-10** |
| 知己（v1.9 新增） | 7.3 常规 | 语气更直接亲昵，可带轻微撒娇/依赖感 → **P-11** |

## 十一、技术架构

### 11.1 核心技术流程（v1.9 更新）

```
周日 23:00   [LLM-01] 周大纲生成（下一自然周，含本月比例软约束参考，days_count=7）→ 记录日志
             └─ 失败则 23:30 重试一次 → 记录日志
             └─ 两次均失败 → 该自然周大纲整体缺失，无自动补救，后台人工补录 → 记录日志
             └─ 大纲按自然日落库（date / city / categories，categories 受 categories_vocab 约束）

00:20        读取"次日"对应的周大纲条目（city + categories）→ 记录日志
             ├─ 不存在 → 当日直接跳过，不生成、不重试 → 记录日志
             └─ 存在 → [LLM-02] 生成当日生活场景细节
                  ├─ 严格遵循大纲 city + categories（强约束）
                  ├─ 2-5 个场景，时间 06:00-20:00，时间段格式如 "09:00-10:30"
                  ├─ venue_type 自由发挥，不受枚举约束
                  └─ 失败（场景数<2）→ 00:30 重试一次 → 记录日志
                       └─ 仍失败则放弃，当日不发布，无需告警，可手动补发 → 记录日志

00:45        她的宇宙触发，检查 life_plan.gen_status = 'ready' → 记录日志
             └─ 无 ready 数据 → 整个任务跳过 → 记录日志
             └─ 有数据 → 遍历每个 scene（各自独立，45s 超时，最多 3 次）：
                  [LLM-03] 同次调用：
                  ① 生成动态快照（feeling_text / emotion_value[核心词优先+自由兜底] / focus_tag / worldview_trigger）
                     → worldview_snapshot → 每步记录日志
                  ② 同次调用生成她的宇宙静态事件看法（event_name / event_view，核心态度含无感选项）
                     → worldview_event upsert（已有则跳过，记录日志）
                  └─ 单条失败 → 立即重试，最多 3 次 → 记录日志
                       └─ 仍失败 → 标记 failed → 记录错误日志

01:00        LIFE001 触发，读取当日 life_plan → 记录日志
             └─ 无计划 → 当日跳过，不发布 → 记录日志
             └─ 有计划 →
                  ① 去重检查：venue_type+category+city 三字段组合，7天窗口 → 记录每条结果日志
                  ② 确定发布数量：min(随机2或3, 去重后可用场景数) → 记录日志
                  ③ 按场景时间顺序分配发布窗口
                  ④ [LLM-04] 生成文案
                     ├─ 快照 ready → 注入情绪快照增强段；emotion 直接复制自 worldview_snapshot.emotion_value
                     └─ 快照 failed → 省略快照段降级生成；LLM-04 附带输出 emotion 字段（核心词优先+自由兜底，非独立调用）
                     （旅游叙事上下文可选段：非主场城市时注入）→ 记录日志
                  ⑤ 并行图片生成（单张 3 分钟 / 整批 15 分钟超时，超时降级；关键词映射未命中时启用兜底；临时文件随上传成败立即清理）→ 记录日志
                  ⑥ 落库：
                     - season：按 city+plan_date 计算（南半球白名单城市用反向月份规则）
                     - emotion：快照 ready 时复制自 worldview_snapshot.emotion_value；
                                快照缺失时使用 LLM-04 附带输出的 emotion 字段
                     - is_visible = 1（管理员手动控制）
                     - dedup_hash：基于 venue_type+category+city 生成
                  ⑦ 整体完成 → 记录汇总日志

计划时间点    查询层过滤：WHERE scheduled_publish_time <= NOW() AND is_visible = 1
             → Feed 对用户端自动可见，无需状态变更任务

用户点赞     判定点赞特殊档（created_at 起算窗口 + special_used < max）或常规档 30% → 记录日志
             ├─ 特殊档：100% 入队，due_at = 触发时刻 + like_aware_special_delay_sec，special_used +1
             └─ 常规档：30% 概率 → 未中则结束；命中后检查同帖是否已触发过
                  ├─ 已触发过 → 跳过
                  └─ 未触发 → due_at = 触发时刻 + 关系档延迟，写入排队表
             独立轮询到期 → [LLM-06] P-07 生成 → agent_message（LIKE_AWARE）
             real_likes + 1

用户取消点赞  real_likes - 1，已触发 IM 消息不撤回 → 记录日志

用户评论     检查是否全局首次评论（该用户生命周期首次）→ 记录日志
             ├─ 首次：30s 后启动 [LLM-05] 生成评论回复（override）
             └─ 非首次：按当前关系阶段延迟窗口启动 [LLM-05] 生成评论回复
             [LLM-05] 生成评论回复（45s 超时 × 最多 3 次，称呼来源复用 hobby_name/real_name 机制）→ 记录每步日志
             ├─ 成功 → 纯文本直接写入 feed_comment.lxm_reply，gen_status = ready → 记录日志
             └─ 最终失败（3次均失败）→ 静默不回，gen_status = failed，记录 ERROR 日志，后台可手动补发

已读感知      用户浏览帖子 → 近 read_suppress_after_like_im_hours 内有点赞 IM 入队？→ 是则跳过
             → 否：判定已读特殊档（7.2）或常规档（7.3）→ 写入排队表 → 轮询 → [LLM-07]（P-14 或 P-08~P-11）
                  ├─ 成功 → agent_message（READ_AWARE）+ sort_seq
                  └─ 失败 → failed，可后台补发
```

### 11.2 图片生成、存储与分发规范（v1.8 明确 OSS 范围）

**本版本 OSS 存储范围（v1.8 明确）**

- **仅 `feed_post` 相关的自动生成图片走阿里云 OSS + CDN**（朋友圈图片，随生活流每日自动生成）
- **角色头像、用户头像、人物参考基准图（`image_reference_url`）等配置类/存量图片，本版本继续走服务器本地存储**，不迁移至 OSS
- 原因：这类图片是相对静态的存量数据，当前无明显运维压力，不必为本次新增功能顺带迁移已稳定运行的存量系统，符合模块独立、不预先改动旁路模块的设计原则

**人物参考基准图存储路径（v1.8 明确）**

```
frontend/static/images/avatar/character-ref/base.png
```

- 与 IM 聊天头像物理隔离，不共用同一张图片文件（聊天头像追求美观辨识度，IMG1 参考图追求中性光线、清晰五官便于 AI 识别，两者选图标准不同）
- **公网可达（A-3）**：nginx 已将 `/static/` 对公网开放；LiblibAI 通过 URL 拉取参考图时须使用 **`https://<生产域名>/static/images/avatar/character-ref/base.png`**。本地 `localhost` 无法被云端拉取，**素材就位 ≠ 拉取验收完成**，部署后须实测一次 IMG1 试生成
- 当前为 **单图方案**：`image_reference_url` 单值（VARCHAR），格式 **PNG**（v1.9.4 由 jpg 改为 png，与仓库素材一致）；多参考图见附录 A-2（v1 不启用）

**LiblibAI API 使用规范**

- **认证方式**：HMAC-SHA1 签名，URL 拼接 AccessKey、Signature、Timestamp、SignatureNonce 四个参数
- **并发限制**：进行中任务并发 **1**（提交+轮询串行；2026-07-10 联调：账号侧上限约 1，原「默认 5」会导致多图后几张失败，见台账 TB-LF-008）；QPS 限制 1 次/秒
- **请求间隔**：多张图的生成请求之间必须间隔 1 秒以上，后台任务队列需加请求限速逻辑
- **同帖多图构图**：整帖同一 `image_type`；`count≥2` 时按序号追加构图变体后缀并使用独立 seed；单次 API `imgCount=1`（详见契约草案「同帖多图与限速」）

**临时文件处理**

- 图片从 LiblibAI 下载后，先写入本地临时文件用于 WebP 压缩处理，压缩完成后上传至 OSS
- 临时文件在上传 OSS **成功或失败后立即清理**，不依赖定时批量清理任务
- 服务启动时可选扫描临时目录，清理上次异常退出残留的过期临时文件（非强制，作为兜底保险措施）

**OSS 存储路径规范（仅适用于朋友圈自动生成图片）**

- 按业务前缀分目录存储，避免未来其他业务模块共用同一 Bucket 时产生路径冲突：
  - 朋友圈图片：`/lxm/posts/{post_id}/{filename}.webp`
- 新增其他业务时，按相同规则在 Bucket 根目录下新增独立前缀，不与上述路径混用

**OSS 与 CDN 绑定**

- OSS Bucket 设为私有读写，绑定阿里云 CDN 加速域名，CDN 侧开启"私有 Bucket 回源授权"
- 写入 `feed_post.image_urls` 的地址统一使用 **CDN 加速域名**，不使用 OSS 原始域名
- CDN 域名作为后台配置项（环境变量），不硬编码在代码中，便于后续更换

### 11.3 错误处理与降级策略（v1.9 更新）

| 场景 | 处理策略 |
|------|----------|
| 周大纲生成失败（LLM-01，23:00、23:30 均失败） | 该自然周大纲整体缺失，无自动补救，后台人工逐日补录，记录日志 |
| 当日无对应周大纲条目 | LLM-02 日场景生成直接跳过，不触发、不计入失败，记录日志 |
| 日场景生成失败（LLM-02，重试后场景数仍<2） | 当日不发布朋友圈，无需告警，后台可手动补发，记录日志 |
| 她的宇宙任务触发时无 ready 数据 | 整个 LLM-03 任务跳过，记录日志 |
| 单条快照技术失败（LLM-03，超时/5xx） | 立即重试，最多 3 次，仍失败标记 failed，记录错误日志 |
| 单条快照内容失败（LLM-03，缺字段/格式错误） | 同技术失败，立即重试，最多 3 次，仍失败标记 failed，记录错误日志 |
| 快照 failed 时 LIFE001 降级 | LLM-04 省略快照段，附带输出 emotion 字段，文案仍有情绪色彩，记录日志 |
| LIFE001 触发时快照为 generating 状态 | 视同 failed，LLM-04 降级处理，记录日志 |
| venue_type 未命中图像关键词映射（v1.8 新增） | 按 category 兜底查 `category_img_keyword`，记录日志 |
| emotion 未命中核心词映射（v1.8 新增） | 使用 `emotion_fallback_img_keyword`/`emotion_fallback_atmosphere_desc` 通用兜底关键词，记录日志 |
| 图片单张超时（>3 分钟） | 该张视为失败，其余正常 |
| 图片整批超时（>15 分钟） | 已完成图片正常打包，未完成视为失败降级，记录告警日志 |
| 人物图片组生成失败 | 仅该组缺失，其余图片正常发布 |
| 非人物图片组生成失败 | 仅该组缺失，其余图片正常发布 |
| 所有图片均生成失败 | 自动降级为纯文字发布，不展示占位图 |
| 文案生成失败（LLM-04） | 放弃本条发布，不补发，记录错误日志 |
| 评论回复 LLM-05 失败（45s×3次均失败） | 静默不回，gen_status = failed，后台记录 ERROR，管理员可在评论管理页手动补发 |
| 点赞 IM LLM-06 生成失败（v1.9 更新） | 排队表记录标记 failed，记录 ERROR 日志，管理员可在 10.8 节新增的后台入口手动补发 |
| 已读感知 LLM-07 生成失败（v1.9 新增） | 排队表记录标记 failed，记录 ERROR 日志，管理员可在 10.8 节新增的后台入口手动补发 |
| 网络异常 | 客户端显示友好提示，不展示空态 |

### 11.5 成本控制措施（v1.7 更新）

- 图片统一压缩为 WebP 格式（后端下载 LiblibAI 结果后本地压缩，上传 OSS 后通过 CDN 分发，详见 11.2），CDN 流量成本降低 60–70%
- 01:00 提前生成，避免高峰期资源竞争
- 积分消耗监控：开发阶段统计单张图片平均积分消耗，推算月度成本
- API 故障时自动降级为纯文字发布，不影响核心功能
