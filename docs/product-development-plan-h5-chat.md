## 一、背景与两个核心目标（对应最初诉求）

### 目标 A：对话可恢复、再进页不「丢一轮」

**问题现象**：用户发话后，在 AI 回复展示完成前离开页面，再进入时往往看不到本轮完整结果；极端情况下整轮未落库。  
**产品目标**：用户已发送的内容**可预期地被保存与再次展示**；在**未主动作废**的前提下，只要服务端已成功生成回复，**再进页应能看到完整对话结果**，**不依赖**客户端是否把 SSE 流读完。

### 目标 B：对话连贯可控（不必等打字结束才能继续说）

**问题现象**：等待上一条 AI 回复时输入/回车被锁，多句连发与真实聊天习惯不一致；失败时用「走神」类占位回复混在对话里，体验与统计口径混乱。  
**产品目标**：支持**多句未闭环**时的排队、打断与合并回复；失败用**可理解的失败态 + 可重试**，与正常 AI 话**区分**；在规则允许下**突破「单通道」**，同时用**上限与安全**防止滥用。

## 二、用户与价值简述


| 角色       | 价值                                               |
| -------- | ------------------------------------------------ |
| 终端用户（H5） | 弱网、切页、连发时**少丢话、少焦虑**；失败**可点重试**；聊天节奏接近自然对话。      |
| 运营 / 管理  | 后台仍以**落库对话**为准排查；情绪等延伸能力按 **TD-016** 分阶段，不冒充已完备。 |


---

## 三、产品原则（摘要）

1. **真相在服务端**：以 **DB + 代（generation）** 为权威；客户端只做展示与节流。
2. **主动打断 vs 被动离开**：**新输入**导致旧轮**作废**；**关页/断网**不自动等于作废，**LLM 已成功则闭环**。
3. **失败不冒充成功**：超时/错误**不写「走神」进对话流水**作 AI 正式回复；用**叹号 + 重发**表达失败与补救。
4. **一轮一答一情绪（结构化不变）**：无论用户连发几句，**一次调度**对应 **一条助手回复**、**一套 JSON（emotion + reply）**；emotion 表意为**整轮用户状态**（与 **TD-016** 对齐）。
5. **记忆与检索首版保守**：用于向量检索的 embedding 文本，**默认取本轮最后一条 user 原文**（与现网单句行为一致）；**检索 query 改写**见 **TD-017**，不纳入本期必达。

---

## 四、术语（与 TD-015 一致）

- **历史消息**：到最后一条**已闭环的助手消息**为止（含该助手及更早）。  
- **未闭环 / 新消息窗口**：最后一条历史助手之后，至**本轮助手成功落库**前的全部 user 行为（可多条、可带失败态）。  
- **一代（generation）**：一次调度 LLM 的尝试；**新输入打断**则旧代作废。

## 六、关键用户旅程（简）

1. **连发两句**：第一句触发调度可被第二句打断；最终**打包**后一次 AI 回复。
2. **超时**：45s 无有效返回 → user 行叹号 → 用户可继续输入或点叹号重发（受 2 次/分）；**不**出现假 AI 气泡入库。
3. **关页再进**：已发 user 仍在；若服务端已闭环 assistant，timeline **直接可见**完整轮次。
4. **重发**：同一批 user 行 **不新增行**；成功后叹号清除。

---

## 七、业务规则汇总表（产品可读）


| 主题           | 规则                                                                              |
| ------------ | ------------------------------------------------------------------------------- |
| 超时           | 仅对话 **45s**；非对话 **15s**。                                                        |
| 作废           | **新 user 进入并打断** → 旧代结果**丢弃**。                                                  |
| 闭环           | **未作废**且 **LLM 解析成功** → 写 assistant + 后置任务（成长、记忆提取、`ai_emotion` 等按现策略**每轮一次**）。 |
| 打包           | **≤10** 条 user 进 Prompt；更旧行**不入本包**但**仍在 DB**。                                  |
| 防抖           | **500ms** 合并自动调度。                                                               |
| 重发           | **每分钟每未闭环批次最多 2 次**（仅手动叹号）。                                                     |
| 检索 embedding | **默认末条 user**；改写见 TD-017。                                                       |
| 记忆入库         | **conversation_log** 存**原文**；**memory** 为 LLM **抽取事实**，非原话照抄。                   |

## 八、非功能与安全

- **幂等**：防重复入队、防双击重发。  
- **限流**：重发 + 队列上限，防刷 LLM。  
- **网关**：`proxy_read_timeout` ≥ 45s（略大于 LLM 超时）。  
- **隐私与合规**：内容安全前置；未过安检**无叹号**（不进入失败重试流）。

## 十二、小结

本方案将**最初两个目标**收敛为：  
**（A）可恢复的对话真相** — 入队即存、代与被动闭环、再进可见；  
**（B）连贯可控的对话节奏** — 队列、打断、打包、失败叹号与重发、上限与安全、统一一轮结构化输出。  

工程落地以 `**docs/chat-refactor-implementation-plan.md`** 为执行拆解；产品验收以本文 **§五 Must + §六旅程 + §七规则表** 为主检查单。
