## 本阶段新增/变更摘要

- **点赞/已读感知 IM 闭环**：独立排队表 `agent_aware_queue` 消费架构（LLM-06/07），与对话主链 Agent（P0~P4/Future）完全解耦——不评分、不共享日上限 8 次/30min 间隔计数器、不受黑名单限制。
- **SSE 实时新帖**：单进程内存注册表 + `GET /api/feed/events` + 30s 广播调度任务；`feed_post.sse_broadcasted` 去重。
- **已读上报闭环**：评论回复曝光 + Feed 卡片停留 3s 两类上报 API，联动已读感知 IM。
- **DB 变更**：Alembic 迁移 `v6b_step019_026_001`（`agent_aware_queue.{prompt_key, extra_context}` + `feed_post.sse_broadcasted`）。
- **M1 stub 替换**：`like_aware_service.on_like_hook` 由 M1 空实现替换为真实入队逻辑（**签名不变**）。
- **2026-07-10 补记**：`feed_new_broadcast_task`「到点」比较与 M1 列表一致，使用 `feed_now()`（Asia/Shanghai），见下表。
- **2026-07-11 补记（H5 提示条语义澄清）**：页内「林小梦更新了 N 条动态」仅表示 **SSE `feed_new` 新帖条数**，**不是** LXM 评论回复未读数（回复未读仍走首页 badge `unread_reply_count`）。
- **2026-07-11 补记（感知 IM 联调修复 · TB-LF-009）**：
  1. **点赞入队时序**：`feed_service.like_post` 须先 `commit` 点赞事务（释放 `feed_post` 行锁），再调 `on_like_hook`；否则嵌套 session 写 `agent_aware_queue` 时 FK 校验 `feed_post` 易 `Lock wait timeout`，入队失败被吞、点赞仍成功。
  2. **已读观察器挂载**：`observeReadTargets(el)` 在 `el` 即为 `.feed-item` 时须先 `observe` 自身（`querySelectorAll` 只匹配后代）；否则停留 3s 上报永不触发。
  3. **特殊档计数**：对齐 PRD「**入队成功**才 +1」——CAS 占位后 `enqueue`；入队异常则回滚占位，避免空耗 `like/read_aware_special_used_count`。

## 数据库变更（迁移 `v6b_step019_026_001`）

### `agent_aware_queue` 扩展字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `prompt_key` | `VARCHAR(32) NULL` | 入队时确定的 Prompt config_key；LIKE_AWARE=`prompt_p07`，READ_AWARE 特殊档=`prompt_p14`/常规档=`prompt_p08~p11`。消费时直接使用，不重算档位 |
| `extra_context` | `JSON NULL` | 入队上下文快照（`is_special` 等） |

> 说明：M2 复用 M1 已建的 `agent_aware_queue` 主结构；本表列名沿用 `trigger_type`（值 `LIKE_AWARE`/`READ_AWARE`）与 `post_id`（关联 `feed_post.id`），与实施计划描述的 `aware_type`/`related_post_id` 为同义映射。`status` 状态机：`pending → generating → sent/failed`。

### `feed_post` 扩展字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `sse_broadcasted` | `TINYINT(1) DEFAULT 0` | SSE 新帖广播去重标记（0=未广播/1=已广播），广播任务扫描后置 1 |

## 用户端 API（新增 3 个）

| Method | Path | 鉴权 | 说明 |
|--------|------|------|------|
| GET | `/api/feed/events` | JWT（**query `token`**） | SSE 长连接（`text/event-stream`）；EventSource 无法自定义 header，token 走 query 独立校验；15s 无事件发送 `: heartbeat` 注释行 |
| POST | `/api/feed/comments/{comment_id}/read` | JWT | 单条评论回复已读上报；写 `lxm_reply_read_at`（幂等）；越权返回 `code=10506` |
| POST | `/api/feed/{post_id}/read` | JWT | 单帖已读上报；校验可见+到点（`feed_now()`）后触发 `read_aware_service.on_feed_read` |

- SSE 事件体：`{"type": "feed_new", "delta": <本轮新帖数>}`。
- 断线不补发历史事件（PRD 5.9.3），依赖前端下拉刷新兜底。
- 新增错误码：`ERR_FEED_COMMENT_FORBIDDEN = 10506`「无权操作该评论」。

## 服务层接口

### `agent_aware_service`（新增）

- `enqueue(user_id, aware_type, related_post_id, prompt_key, delay_seconds, relationship_stage, extra_context=None) -> int`：写入 pending 记录，`due_at=now+delay`，返回 `queue_id`。
- `consume_pending(batch_size=20) -> int`：轮询消费入口。
- `consume_record(queue_id) -> bool`：原子锁 `pending→generating` → 按 `trigger_type` 分派 → 落 `agent_message`（`action_score=0`）+ `timeline_seq_service.allocate_sort_seq` → `sent`；失败 `failed`。
- `generate_aware_text(row, node_key, temperature)`（模块级）：LIKE/READ 共用生成逻辑（渲染 prompt_key → LLM **45s×2 重试** → 内容安全）。超时常量 `_AWARE_TIMEOUT=45.0`（与 DeepSeek 全局默认对齐，见 M1 契约「DeepSeek 超时 / 重试」）。

### `like_aware_service`（替换 stub）

- `on_like_hook(user_id, post_id)`：同帖去重（`pending`/`generating`/`sent`）→ 特殊档（窗口内+未满，100%，delay=`like_aware_special_delay_sec`）→ 常规档（`random<0.30`，delay 按 `like_regular_delay_{stage}_{min,max}`）；异常吞掉不影响点赞主流程。Prompt 恒为 `prompt_p07`，LLM 节点 `llm_06`（temp 0.7）。
- **特殊档计数（2026-07-11）**：先 CAS `like_aware_special_used_count +1`（`WHERE < max`）占位 → `enqueue` 成功则保留；`enqueue` 抛错则 **回滚 −1**（对齐 PRD 6.1.3「入队成功才 +1」）。并发占满则落常规档。
- **调用时序（2026-07-11）**：由 `feed_service.like_post` 在点赞 `commit` **之后**调用（见 M1 草案 like API 补记）；钩子内另开 `async_session_maker`，不得持有未提交的 `feed_post` X 锁。

### `read_aware_service`（新增）

- `on_feed_read(user_id, post_id)` / `on_feed_read_batch(user_id, post_ids)`：多帖取 `scheduled_publish_time DESC` 最近一条 → 同帖去重 → 6h 点赞互斥（`read_suppress_after_like_im_hours`，仅 `LIKE_AWARE` 且 `status IN ('pending','sent')`）→ 特殊档（P-14）→ 常规档（关系档→`prompt_p08/09/10/11`）。LLM 节点 `llm_07`（temp 0.8）。
- **特殊档计数（2026-07-11）**：与点赞对称——CAS 占位 → `enqueue`；失败回滚 `read_aware_special_used_count`（对齐 PRD 7.2）。

### `feed_sse_service`（新增，单进程内存）

- `register(user_id) -> asyncio.Queue` / `unregister(user_id, queue)` / `broadcast_new_feed(post_ids) -> int` / `online_user_count()`。

## 前端 `/pages/feed.html` 变更

- **SSE**：仅本页 `new EventSource('/api/feed/events?token=...')`；收到 `feed_new` 累加顶部提示条「林小梦更新了 **N** 条动态 · 查看 >」；点击滚顶 + 重新拉列表清零；`onerror` 指数退避（1s→30s）重连。
- **提示条语义**：N = 本页打开期间收到的 **新发帖 `delta` 累加**；与评论回复未读、首页双徽标独立。
- **已读上报**：`IntersectionObserver` 评论回复 0.6 曝光即上报 `/comments/{id}/read`（本地 Set 去重）；Feed 卡片 0.5 进入视口计时 3s 未离开上报 `/{post_id}/read`（提前离开清 timer）。曝光目标为已渲染的 `.comment-reply[data-comment-id]`（有 `lxm_reply` 时）；**不再依赖**「正在回复」占位节点。
- **已读观察器挂载（2026-07-11）**：`loadFeed` 对每张新卡调用 `observeReadTargets(el)` 时，`el` 即为 `.feed-item`；实现须 **先 `postDwellObserver.observe(rootEl)`（自身）**，再 `querySelectorAll('.feed-item')` 扫后代（兼容批量根）。仅扫后代会导致帖子停留上报永不触发（评论回复子节点不受影响）。
- **时间/列表展示**：相对时间解析与同日左列合并、meta 行等见 **M1 草案「前端页面 · 2026-07-11 UI 对齐」**；与 SSE 到点推送同一时区语义。
