### TB-LF-001 · LiblibAI 业务 payload 不符合开放平台 API

| 字段 | 内容 |
|------|------|
| **状态** | 已修复 |
| **严重度** | 高（阻断朋友圈出图业务路径） |
| **发现方式** | `scripts/smoke_liblib_oss_path.py` · 2026-07-07 容器内冒烟 |

#### 现象

- 使用 **与业务相同** 的 payload（仅 `prompt` + `negativePrompt`）调用 LiblibAI → 返回：  
  `生图参数未通过参数完整度校验，请检查参数配置`
- 使用 **官方结构**（`templateUuid` + `generateParams`）+ 同一套 AK/SK → **约 7s 成功出图**。
- 后续补齐官方结构后：文生图可通；图生图（selfie）在 UUID 已填时仍报缺 **`resizedWidth`**。

#### 根因

`backend/services/feed_image_service.py` 的 `_build_payload()` 早期只组装 `prompt` + `negativePrompt`（selfie 另加 `sourceImage`），未按开放平台要求组装 `templateUuid` + 完整 `generateParams`。  
图生图模板额外要求顶层 `resizedWidth` / `resizedHeight`（参考图缩放目标尺寸，与出图 `width`/`height` 语义不同）。

#### 为何单元测试未暴露

`tests/test_step012_feed_image.py` 对 `liblib_client.submit_task` / `poll_task` 做了 **mock**，未做真机请求，因此 STEP-012 标 ✅ 但真机联调仍会失败。

#### 建议修改方式（给 AI）— 已落实

1. **配置层**（`admin_config` / `life_feed_config` 种子）：
   - `liblib_text2img_template_uuid`（Star-3 / P-13a/b/c）
   - `liblib_img2img_template_uuid`（IMG1 / P-12；默认空，须控制台填写）
   - `liblib_gen_steps` / `liblib_gen_width` / `liblib_gen_height`
   - `liblib_img2img_strength`
   - `liblib_img2img_resized_width` / `liblib_img2img_resized_height`（默认 768/1024）
2. **服务层**：`_build_payload()` 输出官方结构；selfie 分支含 `sourceImage`、`strength`、`resizedWidth`、`resizedHeight`。
3. **后台**：`life-feed-system.html`「Liblib 生图参数」可编辑上述键（草稿/发布）。
4. **验收**：文生图/图生图真机出图 → OSS CDN URL 可 GET 200。

#### 涉及文件

- `backend/services/feed_image_service.py`（`_build_payload`、`_gen_one`）

### TB-LF-002 · FEED_IMAGE_REFERENCE_PUBLIC_URL 配置错误

| 字段 | 内容 |
|------|------|
| **状态** | 已修复 |
| **严重度** | 高（仅影响 `image_type=selfie` 图生图） |
| **发现方式** | 冒烟脚本 `ref_url.scheme` · env 审查 |

#### 现象

当前 `.env`：

```env
FEED_IMAGE_REFERENCE_PUBLIC_URL=oss://pj4-test/base/frontend/static/images/avatar/character-ref/base.png
```

冒烟：`ref_url.scheme` ❌；`liblib.img2img` 跳过或失败。

#### 根因

- 变量语义（`config.py` 注释）：**供 LiblibAI 公网拉取** 的 HTTP(S) URL。
- `oss://` 是阿里云 SDK 协议，Liblib 云端 HTTP 客户端**无法使用**。
- 即使改为 `https://pj4-test.oss-cn-hangzhou.aliyuncs.com/base/...`，当前 Bucket 为**私有**，匿名 GET 仍 **403**（2026-07-07 实测）。

#### 建议修改方式（给 AI / 运维）

**二选一（按环境）**：

| 环境 | 推荐 `FEED_IMAGE_REFERENCE_PUBLIC_URL` |
|------|--------------------------------------|
| 生产 | `https://<生产域名>/static/images/avatar/character-ref/base.png`（nginx 已开放 `/static/`，见 PRD 附录 A-3） |
| 本地联调 img2img | 需公网可达 URL：内网穿透、或 OSS 上该对象 **公共读**、或临时 signed URL（若走 signed URL 需评估有效期与代码是否支持） |

**不要**写 `oss://` 或 `http://127.0.0.1/...`（后者 Liblib 云端不可达）。

本地文件已存在：`frontend/static/images/avatar/character-ref/base.png`（nginx 容器可 200，仅本机）。

#### 涉及文件

- `.env` / `.env.example`
- 无需改代码即可修 **URL 格式**；**公网可达** 属运维

#### 修复记录

| 修复日期 | 修复说明 | 验收结果 |
|----------|----------|----------|
| 2026-07-08 | `.env` 改为 HTTPS；OSS `base/...` 公共读；`force-recreate backend` | `ref_url.scheme` ✅；`ref_url.reachable` ✅ 200/631193B |

```
Code: RequestTimeTooSkewed
```

1. macOS：**系统设置 → 日期与时间 → 自动设置日期与时间**（开启并立即同步）。
2. Docker Desktop 继承宿主机时钟；修正宿主机后 **无需** 单独改容器时区。
3. 验收：`scripts/smoke_liblib_oss_path.py` 中 `clock.skew`、`oss.put`、`oss.get` 为 ✅。

### TB-LF-005 · 无 CDN 时 OSS Key 能否让 H5 看图？

| 字段 | 内容 |
|------|------|
| **状态** | 已规避（测试环境） |
| **严重度** | 中（影响 `feed_post.image_urls` 在浏览器能否展示） |
| **发现方式** | 用户疑问 + CDN `403` 冒烟 |

#### 问题

已配置 `OSS_ACCESS_KEY_ID` / `OSS_ACCESS_KEY_SECRET`，未配置 CDN，能否正常完成「上传 + 用户端展示」？

#### 结论（给 AI）

**上传可以，浏览器直链展示不行（在私有桶前提下）。**

| 能力 | 是否靠 OSS AK/SK | 说明 |
|------|------------------|------|
| 后端上传到 OSS | ✅ | `feed_image_service._oss_put()` 用 `oss2` + AK/SK，**与 CDN 无关** |
| 后端 SDK 读对象 | ✅ | 同上 |
| 浏览器打开 `feed_post.image_urls` 里的 HTTPS 链接 | ❌（私有桶） | URL **不含** AK/SK；匿名 GET 被拒 |
| 变量名 `OSS_CDN_DOMAIN` | 命名来自 PRD | 实际是「写入 DB 的 URL 主机名」；**可填 OSS 公网域名**，但不等于有 CDN 服务 |

代码拼接方式（`feed_image_service.py`）：

```python
cdn = get_oss_cdn_domain()
return f"https://{cdn}/lxm/posts/{post_id}/{seq:02d}.webp"
```

#### 无 CDN 时的可选方案

| 方案 | 是否需要改代码 | 说明 |
|------|----------------|------|
| A. 阿里云 CDN + 私有桶回源鉴权 | 否（PRD 11.2 推荐） | `OSS_CDN_DOMAIN` 填 CDN 域名；控制台开回源授权 |
| B. Bucket 或 `lxm/posts/` 前缀 **公共读** | 否 | `OSS_CDN_DOMAIN=pj4-test.oss-cn-hangzhou.aliyuncs.com`；H5 可直接访问；**安全性较差**，仅测试桶可考虑 |
| C. 后端代理图片 `GET /api/feed/image?...` | **要** | 后端用 SDK 读流返回；URL 不写 OSS 直链 |
| D. 生成带签名的临时 URL 写入 DB | **要** | 需处理过期刷新 |

**当前 M1 设计意图**（PRD 11.2）：私有桶 + CDN 加速域名。  
**测试环境**若无 CDN：优先 **方案 B（测试桶公共读）** 或接受 H5 图片 403 直到配 CDN。

#### 与 TB-LF-003 的关系

时钟未同步时，连 **方案 B 的上传** 也会失败；须先修时钟，再谈公共读或 CDN。

|----------|----------|----------|
| 2026-07-08 | 测试桶 `pj4-test` 读写权限改「公共读」；「阻止公共访问」未开启 | `cdn.fetch` ✅ 200；参考图与 smoke 对象匿名 GET 均 200 |

### TB-LF-009 · 感知 IM 点赞入队锁等待 / 已读观察器未挂载 / 特殊档计数空耗

| 字段 | 内容 |
|------|------|
| **状态** | 已修复 |
| **严重度** | 高（点赞/浏览后长期无感知 IM） |
| **发现方式** | 本地 Docker 联调 · 2026-07-11（user=1 多日点赞 8 次、队列 0 条、`AUTO_INCREMENT=6`） |
| **关联契约** | `M2_契约草案.md` 服务层/前端补记；`M1_契约草案.md` like API 事务时序 |

#### 现象

- 用户大量浏览/点赞朋友圈后，`agent_aware_queue` 为空，无 `LIKE_AWARE`/`READ_AWARE` 的 `agent_message`。
- 日志曾出现：`[点赞IM] on_like_hook 异常 … Lock wait timeout exceeded`（约 50s，与 `innodb_lock_wait_timeout` 一致）；点赞 API 仍 200。
- 几乎无 `POST /api/feed/{id}/read`；评论回复 `/comments/{id}/read` 正常。

#### 根因

1. **点赞**：`like_post` 在未 `commit` 时调用 `on_like_hook`；外层持有 `feed_post` X 锁，钩子嵌套 session INSERT `agent_aware_queue`（FK→`feed_post`）等待超时，异常被吞。
2. **已读**：`observeReadTargets(el)` 传入帖子节点本身，仅用 `querySelectorAll('.feed-item')` 扫后代，**自身未 observe**，3s 停留上报不触发。
3. **特殊档**：先 +1 再 enqueue；入队失败时次数已消耗（与 PRD「入队成功才 +1」不符）。

#### 修复说明（2026-07-11）

1. `feed_service.like_post`：`commit` 后再 `on_like_hook`。
2. `feed.html`：`observeReadTargets` 先 `observe(rootEl)`（自身为 feed-item 时）。
3. `like_aware_service` / `read_aware_service`：特殊档 CAS 占位 → enqueue；失败回滚计数。

#### 验收方式

- 单测：`test_step019/020/021` 等通过。
- Docker 冒烟：已读入队 → 点赞入队（无 Lock wait）→ `consume_pending` 落 `agent_message`+`sort_seq`；Future 同表写入；6h 点赞互斥抑制已读；`/api/agent/unread-count` 正常。
- 产品口径：过 48h 特殊窗后点赞仍 30% + 陌生 1–2h 延迟，属 PRD 设计非缺陷。
