## 1. 严重级定义

| 级别 | 含义 |
|------|------|
| 高 | 可伪造身份或绕过关键发布/配置门禁，直接影响生产安全 |
| 中 | 削弱锁定、会话、发布或敏感信息保护，或扩大攻击窗口 |
| 低 | 信息泄露或需额外条件，风险可接受或延后 |

## 3. 逐条记录

### VULN-001 发布测试可伪造（本期延期）

| 项 | 内容 |
|----|------|
| 严重级 | 高 |
| 位置 | `backend/routers/admin/persona.py` 发布；`backend/routers/admin/prompt_mgmt.py` Step5/Step5.5 发布 |
| 描述 | 发布接口依赖请求体 `test_passed == true`，服务端没有可信的测试凭证；现有测试实现还存在缺少测试项、异常或空结果时失败开放的可能 |
| 复现思路 | 持有 `ai_trainer`/`super_admin` Token，直接提交 `test_passed: true` 和 `confirm_text: "CONFIRM"`，可绕过真实测试过程 |
| 本期决定 | 不在本期改造，作为高危已知未修风险保留；观察者因统一后端写操作总闸，不能直接调用测试或发布接口，但现有可发布角色仍受影响 |
| 后续完整方案 | 保存草稿、测试、发布三个动作分离；服务端记录被测试内容的哈希或测试凭证；测试通过后内容一旦变化即失效；发布必须手动触发并输入 CONFIRM，测试不得自动发布；运行时继续读取当前已发布版本 |
| 草稿与回滚边界 | 编辑器优先展示草稿，同时明确展示当前生效版本；回滚通过复制历史内容生成新版本，不重新激活旧记录；存在草稿时回滚必须先要求明确处理草稿，不能静默删除 |
| 验证结果 | N/A（本期不修） |

### VULN-002 Admin JWT 默认密钥

| 项 | 内容 |
|----|------|
| 严重级 | 高 |
| 位置 | `backend/config.py`：`ADMIN_JWT_SECRET` 默认回退为 `admin_secret_change_me`；`.env.example` 示例值为 `your_admin_jwt_secret_here` |
| 描述 | HS256 对称密钥若使用公开值，攻击者可本地签发任意 `type=admin` Token |
| 修复方案 | 删除默认回退；缺失、空值、纯空白及 `admin_secret_change_me`、`your_admin_jwt_secret_here` 均拒绝；配置读取函数和 lifespan 显式启动检查复用同一校验；测试和未来 CI 必须显式配置测试密钥 |
| 部署边界 | 当前环境已有的自定义密钥可继续使用，不因本期方案强制轮换；若密钥不变，仅此项不会使现有 Token 失效 |
| 验证步骤 | ① 缺失、空值、两个公开值均拒绝启动 ② 自定义密钥正常签发和校验 ③ 新部署及 CI 未配置时明确失败 |
| 验证结果 | （待开发后填写） |

### VULN-003 锁定后 Token 仍可用

| 项 | 内容 |
|----|------|
| 严重级 | 中 |
| 位置 | `backend/utils/admin_auth.py` → `get_current_admin`；`backend/models/admin_user.py` |
| 描述 | 当前登录会拒绝锁定账号，但已签发 Token 在过期前仍可调用接口；仅增加 `is_locked` 判断会导致解锁后旧 Token 再次可用 |
| 修复方案 | `admin_users` 增加 `token_version NOT NULL DEFAULT 0`；JWT 携带该版本；鉴权时同时检查 `is_active`、`is_locked` 和数据库实时版本 |
| 版本递增事件 | 锁定、修改自己密码、管理员重置密码、登出、实际角色变更；仅修改备注或提交相同角色不递增 |
| 恢复规则 | 解锁不恢复旧 Token；锁定时已经递增版本 |
| 功能边界 | 本期不新增管理员停用/重新启用接口和 UI；紧急直接数据库停用必须同时人工递增 `token_version` 并留审计记录 |
| 兼容决定 | 不含 `token_version` 的历史 Admin JWT 一律拒绝；上线后所有管理员重新登录一次 |
| 验证步骤 | ① 各递增事件发生后旧 Token 立即 401 ② 解锁后旧 Token 仍为 401 ③ 备注/相同角色更新不误撤销 ④ 重新登录获得新版本 Token ⑤ 历史无版本 Token 被拒绝 |
| 验证结果 | （待开发后填写） |
| 验证人/日期 | （待填写） |

### VULN-004 登录失败计数竞态

| 项 | 内容 |
|----|------|
| 严重级 | 中 |
| 位置 | `backend/routers/admin/auth.py`：`login_fail_count += 1` |
| 描述 | 非原子读改写，并发错误密码请求可能丢失计数，无法可靠达到 5 次锁定 |
| 修复方案 | 对目标管理员执行 `SELECT ... FOR UPDATE`，在同一事务内完成锁定检查、密码校验、失败计数、第五次锁定及登录成功清零；第五次锁定同时递增 `token_version` |
| 业务边界 | 保持线上规则：连续 5 次错误密码后锁定，仅超级管理员手动解锁，不自动解除 |
| 验证步骤 | ① 串行 5 次错密必锁 ② 在独立、非生产 MySQL 8 环境验证并发错密计数不丢失且达到阈值后锁定 ③ 锁定后计数保持 5、不再递增 ④ 正确密码登录成功并清零；SQLite 测试不得替代行锁并发验证 |
| 验证结果 | （待开发后填写） |

### VULN-005 life-config 发布无服务端 CONFIRM

| 项 | 内容 |
|----|------|
| 严重级 | 中 |
| 位置 | `backend/routers/admin/life_config_mgmt.py` → `POST /life-config/publish` |
| 描述 | 当前前端已有 CONFIRM 输入，但请求体和服务端模型未传递、校验 `confirm_text`，合法账号可绕过前端直接发布 |
| 修复方案 | 发布请求增加允许缺省/空值进入业务校验的 `confirm_text`；服务端严格校验其等于 `CONFIRM`；缺失或错误统一返回业务码 `20021`；当前 7 处 life-config 前端发布调用同步传值 |

| 验证步骤 | ① 无/错 CONFIRM 返回业务码 `20021` ② 正确 CONFIRM 且有权限时成功 ③ 无权限仍为 403 ④ 当前 7 处前端发布调用回归通过 |

### VULN-008 登录失败状态可枚举

| 项 | 内容 |
|----|------|
| 严重级 | 低 |
| 描述 | 当前不存在账号、密码错误及账号锁定的响应不同，并可能返回剩余尝试次数，可用于探测账号及状态 |
| 本期修复 | 不存在账号、密码错误、锁定或停用统一返回 HTTP 200、业务码 `20001`、`data: null` 和“账号或密码错误”；不再返回剩余次数；不存在账号时执行伪密码哈希校验，降低响应时间差异；真实原因仅写服务端脱敏安全日志 |
| 保留规则 | 线上连续 5 次失败锁定规则不变 |
| 剩余风险 | 攻击者若已知管理员账号，仍可能通过 5 次错误尝试恶意锁定；IP＋账号维度限流本期不做 |
| 验证步骤 | ① 各失败场景外部状态码和响应体一致 ② 不返回剩余次数或锁定状态 ③ 服务端可区分真实原因且日志已脱敏 ④ 5 次锁定回归通过 |

### VULN-009 后台日志敏感信息明文

| 项 | 内容 |
|----|------|
| 严重级 | 中 |
| 位置 | `backend/utils/admin_auth.py` → `log_operation`；`backend/routers/admin/operation_logs.py` 列表、详情和导出；`backend/routers/admin/system_monitor.py` 系统日志列表和导出 |
| 描述 | `before_value`、`after_value` 可能保存并返回配置中的 API Key、Token、密码或其他凭据；观察者获得日志只读权限后会扩大暴露面 |
| 修复方案 | 封装一个共享、无状态、幂等的递归脱敏工具；操作日志写入前以及列表、详情、导出读取时调用，系统日志列表、导出读取时复用；字典/数组直接递归，字符串先尝试解析 JSON，非 JSON 字符串仅匹配已知凭据赋值和 Authorization/Bearer 形式；命中值统一替换为 `[REDACTED]` |
| 脱敏范围 | 大小写不敏感匹配 `api_key`、`secret_key`、`api_secret`、`access_key_secret`、`access_token`、`refresh_token`、`password`、`authorization`、`pepper`、`private_key` 及明确的 `_secret`、`_token` 凭据字段 |
| 非脱敏范围 | Prompt、普通配置、对话、记忆、`max_tokens`、描述和版本等非凭据内容 |
| 历史数据边界 | 本期不批量改写历史数据库行；历史明文通过后台读取时被遮蔽，但数据库直接查询仍可能看到旧明文 |
| 工具失败边界 | 单字段处理异常时该字段按失败关闭方式遮蔽，但不得让正常业务操作因审计脱敏异常而失败 |
| 验证步骤 | ① 嵌套对象/数组凭据写入后为 `[REDACTED]` ② 历史模拟明文在操作日志列表、详情、导出及系统日志列表、导出读取时被遮蔽 ③ 目标描述中的凭据被遮蔽 ④ `max_tokens` 等非凭据字段不误脱敏 ⑤ 重复调用结果保持幂等 |
| 验证结果 | （待开发后填写） |
