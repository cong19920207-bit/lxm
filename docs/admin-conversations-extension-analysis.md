## 四、需产品 / 技术联确认的条目（场景 · 选项 · 建议 · 原因）

### 确认点 A：`role=assistant` 行是否带 `delivery_status` 键

**背景**  
送达态针对「用户句是否已闭环到 LLM」；助手行语义不同。

**场景**  

- **场景 A**：运营导出 JSON，希望每行结构一致。  
- **场景 B**：前端遍历列表，不想对 assistant 分支判空。

**可选方案**  

- **选项 1**：assistant 行 **始终带** `delivery_status: null`，`skipped_in_prompt: null`。  
- **选项 2**：**仅 user** 带两键；assistant **不出现**这两键（键更少）。

**建议**：**选项 1**  
**原因**：结构统一、Admin 前端表格绑定简单；与「仅 user 有语义」不矛盾（null 表示不适用）。

**若选错的影响**  

- 选项 2 时部分客户端若未做 `in` 判断可能 undefined（可文档说明）。

---

### 确认点 B：`delivery_status` 枚举值与展示文案

**背景**  
需与 H5 叹号、timeline、运营理解一致；**实现前**应冻结字符串或整型枚举。

**场景**  

- **场景 A**：超时失败，用户看到叹号；后台同一行应显示可筛选项「LLM 超时」。  
- **场景 B**：历史数据迁移前，旧行无失败概念。

**可选方案**  

- **选项 1**：英文蛇形枚举 + Admin 中文映射表（如 `failed_timeout` →「超时未回复」）。  
- **选项 2**：API 直接返回中文标签（**不推荐**，不利于 i18n 与统计）。  
- **选项 3**：整型 `0/1/2…` + 契约附表（机器友好，可读性差）。

**建议**：**选项 1**（与现有 `role` 等英文风格一致）  
**原因**：与 TD-015、H5 同一套值，契约可列附录；后台 HTML 用映射表展示。

**若选错的影响**  

- 选项 2 后续若要英文报表需再改 API。

---

### 确认点 C：历史行（迁移前已存在）的默认语义

**背景**  
新列上线后，旧 `conversation_log` 行需有默认值。

**场景**  

- **场景 A**：旧数据无法区分是否曾失败，统一标「已成功闭环」。  
- **场景 B**：旧数据标「未知/legacy」，避免运营误判为当前失败。

**可选方案**  

- **选项 1**：默认 `**completed`**（或 `delivered`），与「旧产品无失败态」叙事一致。  
- **选项 2**：默认 `**legacy_unknown`**，UI 显示「—」或「历史」。

**建议**：**选项 1**（若业务上旧数据均为「已正常完成」）  
**原因**：列表不出现大面积假「失败」；若确有历史脏数据再专项清洗。

**若选错的影响**  

- 选项 2 若文案不清，运营误以为系统长期故障。

---

### 确认点 D：是否在 Admin `list[]` 中增加 `sort_seq`

**背景**  
多 user 同轮打包时，**时间线顺序**与 **sort_seq** 可能更利于对账；契约当前未写。

**场景**  

- **场景 A**：运营按 `created_at` 已足够。  
- **场景 B**：同一秒内多条 user，需 `sort_seq` 与 H5 完全一致。

**可选方案**  

- **选项 1**：**增加** `sort_seq`（整型，user/assistant 均有）。  
- **选项 2**：**不增加**，保持契约字段最少。

**建议**：**选项 1**  
**原因**：模型已有 `sort_seq` 列（见项目规则）；只读透出成本低，排障价值高。

**若选错的影响**  

- 不增加时，极端同秒排序可能与 H5 timeline  cursor 逻辑对照稍烦。

---

### 确认点 E：`skipped_in_prompt` 是否对 Admin 开放

**背景**  
Q14 标记「本行未进入本轮 Prompt」；运营可能关心「为何模型没看到这句」。

**场景**  

- **场景 A**：运营需解释用户投诉「AI 没理我前面那句」。  
- **场景 B**：字段对非技术运营噪音大，仅研发查库即可。

**可选方案**  

- **选项 1**：**返回** `skipped_in_prompt`（bool），可选在 UI 隐藏，API 有。  
- **选项 2**：**不返回**，仅 H5 / DB 内部使用。

**建议**：**选项 1**  
**原因**：与 H5 对齐、零额外写入；UI 可默认不展示列。

**若选错的影响**  

- 选项 2 排障需直连 DB。

---

### 确认点 F：契约与实现发布顺序

**背景**  
`contract.md` 与 Admin 前端是否同步改。

**场景**  

- **场景 A**：先上后端字段，契约与 `user-detail.html` 下一迭代再改。  
- **场景 B**：同一版本后端 + 契约 + 可选列一起上。

**可选方案**  

- **选项 1**：与 `**chat-refactor-agent-tasks` 任务 6 或 8** 同一 PR 更新 **contract** `list[]` 字段说明；Admin 页**可选**另 PR。  
- **选项 2**：仅后端先行，契约滞后（短期与 `contract-change-review` 不一致）。

**建议**：**选项 1**  
**原因**：与既定「任务 8 集中更新 contract」可合并为：Admin 接口字段在 **任务 6 出代码、任务 8 补契约**；若任务 6 合并契约也可，但须同一 MR 避免漂移。

**若选错的影响**  

- 选项 2 易引发「文档与接口不一致」类问题。
