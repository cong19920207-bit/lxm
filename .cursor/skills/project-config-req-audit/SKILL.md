# PROJECT_CONFIG 与需求审查

## 何时使用

- 需求审查/细化开始前，**尚未**提供可解析的 `PROJECT_CONFIG_[项目名].md`。
- 用户只贴了需求文档，缺少「代码从哪进、看哪些文件、契约在哪」等上下文。
- 用户要求按模板建立或更新项目配置、同步技术债与约定。

**不具备 PROJECT_CONFIG 时**：先补配置（或明确拒绝盲目猜目录），再进入逐条需求确认；可展示/复制 [PROJECT_CONFIG_template.md](PROJECT_CONFIG_template.md)。

---

## PROJECT_CONFIG 是什么

面向当前仓库的**简要事实表**：让智能体在「新增/修改需求」讨论中不再臆测技术栈、入口路径、关键模块、文档名与项目特有约定。模板头部已说明命名方式：复制后改名为 `PROJECT_CONFIG_[项目名].md`，**上传或置于 Project 可引用位置**（如 Cursor Project 知识或对话附件）以便 `@` 引用。

---

## 创建步骤（从模板到可用）

1. 复制 [PROJECT_CONFIG_template.md](PROJECT_CONFIG_template.md) → 重命名为 `PROJECT_CONFIG_[项目名].md`。
2. 依序填写下方各节；**关键文件映射**与**项目约定**优先保证准确，能显著减少需求审查中的误判。
3. 将文件纳入团队约定位置（如仓库内 `docs/` 或仅 Project 文件），在对话中 `@` 该文件与需求文档、契约文档。

---

## 各节填写要点

| 节 | 填写要点 |
|----|----------|
| 基本信息 | 项目名、类型、技术栈，一行可扫；与真实依赖一致。 |
| 代码结构 | 根目录、主入口，便于从入口下钻。 |
| 关键文件映射 | **只列与本次/近期需求最相关**的少量路径 + 一句职责，忌全量罗列。 |
| 项目文档 | 契约、需求文档的文件名，便于 `@` 双文档对照。 |
| 项目约定 | 响应封套、错误码、命名、业务专有枚举/标识等，**减少字段与语义猜测**。 |
| 已知技术债 | TD 编号、现象、影响范围，避免在审查中重复踩坑。 |
| 本次需求背景 | 可选；跨项目参照或业务背景一句话即可。 |

---

## 与需求审查的衔接

1. **先读** PROJECT_CONFIG：确定根目录、入口、契约与需求路径、关键文件与约定。
2. **再**对照需求 MD 与（如有）`req-confirm` 流程做待确认项与代码对齐；技术债表用于放大冲突点。
3. 配置与代码发生漂移时，**以审查结论或合并后的真值为准**，回写 PROJECT_CONFIG 中的关键文件与约定，避免下次审查继续使用过时信息。

---

## 行为约束

- 不得在无 PROJECT_CONFIG 且用户未给等价信息时，**擅自假定** 目录结构或接口规范。
- 补全配置时，**优先**通过列目录、读入口与契约片段核对，再写回配置项。
- 模板与仓库内 [req-confirm/PROJECT_CONFIG_template.md](../req-confirm/PROJECT_CONFIG_template.md) 内容一致时，**任选其一**作为复制源；避免两处长期不一致，建议以本技能目录下模板或单一文档为唯一维护点。
