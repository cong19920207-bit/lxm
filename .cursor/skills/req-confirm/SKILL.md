# 需求确认（req-confirm）

## 前置条件

使用本 Skill 前，对话中应提供：

1. **需求文档**（MD 格式，含待确认项）
2. **PROJECT_CONFIG**（项目配置文件，见同目录 [PROJECT_CONFIG_template.md](PROJECT_CONFIG_template.md)）

缺少任一项时，停止执行并提示用户补充；可展示模板供用户复制填写。

---

## 执行流程

### 阶段一：初始化

1. 读取 PROJECT_CONFIG，获取：
   - 项目名、技术栈
   - 代码根目录
   - 契约文档路径
   - 关键文件映射
2. 读取需求文档，识别：
   - 需求类型：**新增功能** 或 **修改已有功能**（两者流程有差异，见下文）
   - 所有标注【待确认】的条目
   - 所有涉及现有字段/表/接口的条目
3. 读项目目录结构（先读目录，不盲目全量读文件）

---

### 阶段二：整体扫描

输出一份**全貌分析**，包含：

```markdown
## 本次确认全貌

### 需求类型
新增 / 修改（说明判断依据）

### 待确认条目清单
列出所有需要确认的点，编号排列

### 需要读代码的条目
条目编号 → 对应文件路径 → 原因

### 已识别的潜在冲突
条目编号 → 冲突简述

### 建议确认顺序
按依赖关系排序，说明原因
```

整体扫描完成后，询问用户是否按建议顺序逐条开始确认。

---

### 阶段三：逐条确认

每条格式如下：

```markdown
────────────────────────────────
【需求点 N】需求点标题
────────────────────────────────

需求描述
一句话说明这个需求点要做什么。

涉及的字段 / 表 / 接口
列出本需求点涉及的数据字段、数据库表、API 接口（用语义描述，不强制定义字段名）。

代码对齐
[需要读代码] 文件路径 + 读取原因
→ 读取后补充：发现的实现细节、与需求的吻合或差异

边界条件 A：自身边界
- 异常情况（如 LLM 调用失败、解析报错）
- 空值 / 缺省情况
- 并发 / 重复触发情况
- 失败回退策略

边界条件 B：影响边界
- 对现有功能的潜在影响
- 可能被破坏的现有逻辑

冲突（若有）
场景描述：具体说明在什么情况下会产生冲突，不只是列出冲突点。

选项
每个选项格式：
  ▸ 方案X：方案名称
    场景：什么情况下适用
    做法：简要说明
    风险：可能的问题

AI 建议：推荐哪个方案，给出理由
────────────────────────────────
```

确认完一条后，用户回复决策，记录结论，再进入下一条。

---

### 阶段四：文档更新

**确认过程中**：保留所有【待确认】标注，记录用户的决策选择。

**全部确认完成后**：输出清洁版需求文档：

- 去除所有【待确认】标注
- 将确认结论填入对应位置
- 更新「待确认事项汇总」表格（删除已确认条目，保留真正未决的）

---

## 新增 vs 修改的流程差异

| | 新增功能 | 修改已有功能 |
|--|---------|------------|
| 代码对齐重点 | 确认插入点、依赖关系 | 确认改动范围、回退影响 |
| 边界 B 重点 | 对下游的新影响 | 对现有逻辑的破坏风险 |
| 冲突类型 | 与现有机制的重叠 | 与现有实现的直接冲突 |
| 选项倾向 | 扩展 or 替换 | 兼容改造 or 重写 |

---

## 行为约束（硬规则）

- **不确定不写**：无法从需求文档或代码中确认的内容，标【待确认】，不自行补全
- **字段命名不定义**：用语义描述代替具体字段名，除非需求文档或代码已明确
- **发现冲突必须给选项**：不能只描述问题，必须给出 2-3 个处理方案
- **读代码先读目录**：判断需要读代码时，先读目录结构，再选择性读具体文件
- **跨步骤字段必须对齐**：涉及多个步骤共用的字段，确保文档前后一致，不出现同一语义用不同名称的情况
- **每次只确认一条**：确认完一条收到用户回复后，再进入下一条，不批量抛出

---

## 输出格式规范

- 整体扫描：一次性输出，不分段
- 逐条确认：每条单独输出，等待用户回复
- 文档更新：最终输出完整 MD 文件
