## 一、改动总览

| 步骤 | 类型 | 一句话描述 |
|------|------|-----------|
| Step1 | 扩展 | 并行获取阶段新增时间描述、活动状态、关系文字描述、用户印象的装载 |

## 二、数据层变更

> 具体字段命名与迁移方式，以实现时读代码为准；下述为语义描述。

### 2.1 扩展现有表

**`relationship` 表新增字段（语义）：**
- 关系文字描述（长文本）：由 Step6 记忆 LLM 更新，供 Step3 Prompt 使用
- 用户真实称呼（短文本）：Step6 更新，供下一轮 Prompt 使用
- 用户昵称/绰号（短文本）：Step6 更新
- 用户印象描述（长文本）：Step6 更新
- 未来预约时间戳（整数，可空）：Step6 写入，Step8 轮询消费
- 未来预约意图摘要（短文本，可空）：Step6 写入，Step8 消费
- 主动消息计数（整数，默认 0）：每次主动消息触发后累加，用户发新消息时清零

> **[待确认]** 上述字段是扩展 `relationship` 表还是新建关联表，取决于读代码后对 relationship 表现有宽度与访问频率的判断。未来预约字段也可考虑单独建表或存 Redis，实现时决定。

### 2.2 扩展向量库（DashVector）

当前向量库只有用户记忆一类，改造后需支持按类型区分。

**新增元数据字段（type 标识）：**

| 类型标识 | 语义 | 写入时机 | 是否含 user_id |
|---------|------|---------|---------------|
| `character_setting` | 角色公开背景设定 | 初始化数据导入 + Step6 增量写入 | 否（角色级） |
| `character_knowledge` | 角色知识与技能 | 初始化数据导入 + Step6 增量写入 | 否（角色级） |
| `user` | 用户相关记忆 | 现有 memory_service（用户手动/自动）+ Step6 增量写入 | 是 |

> **注意：** 写入与检索侧的 type 字符串必须完全一致，避免 A 项目中 `chatacter_knowledge`（错别字）导致知识类记忆检索不命中的问题。B 项目从一开始统一用 `character_knowledge`。

**[待确认]** 角色设定与知识的初始数据从哪里导入？是通过管理后台配置还是初始化脚本？`character_setting` 和 `character_knowledge` 类型的条目是否需要对应 MySQL 侧的存储？

### 2.3 活动状态数据源

`activity_str`（活动描述串）的语义是：基于当前时间推算虚拟人正在做什么。

**[待确认]** 活动计划的数据来源：
- 方案 A：在 `admin_config` 中配置时间段与对应活动描述，运行时按时间匹配生成描述串
- 方案 B：新建活动计划表，按时间段 + 概率生成活动描述
- 实现时读代码后选择最优方案

## 三、各步骤功能需求

### Step1：并行获取（扩展）

**现有功能：** 并行拉取近期对话、关系信息、情绪上下文、用户 Embedding。

**新增装载内容：**

1. **时间描述串（`[时间描述串]`）**
   - 内容：将当前时间格式化为自然语言描述（如"现在是周三下午三点"）
   - 生成方式：代码直接根据系统时间生成，无需 LLM，无需 DB 查询
   - 用途：进入 Step1.5 / Step3 Prompt

2. **活动描述串（`[活动描述串]`）**
   - 内容：基于当前时间，描述虚拟人当前状态（如"她现在应该在午休"）
   - 生成方式：从活动计划数据源按时间查找对应描述 **[待确认数据源，见 2.3]**
   - 用途：进入 Step1.5 / Step3 Prompt

3. **关系文字描述（`[关系描述]`）**
   - 内容：从 `relationship` 表读取关系文字描述字段（初始为空或默认文案）
   - 用途：进入 Step3 Prompt 的关系模块 / Step1.5 context

4. **用户印象（`[用户印象]`）**
   - 内容：从 `relationship` 表读取用户真实称呼 / 昵称 / 印象描述
   - 用途：进入 Step3 Prompt 的关系模块 / Step1.5 context

**字段流转：** 以上内容在本轮请求的内存上下文中携带，供后续步骤使用，不单独持久化到 DB。

### Step1.5：查询重写 LLM（新增）

**位置：** Step1 完成后、Step2 向量检索前，单独一次同步 LLM 调用。

**目的：** 让模型理解用户本轮输入的真实意图，生成后续多路向量检索所需的查询语句与关键词。

#### Prompt 结构

```
【系统指令】
你是 {虚拟人名称}，请根据用户最新消息，分析你的内心理解，并为记忆检索生成查询语句。
输出格式要求：仅输出 JSON，不含任何额外内容。

【上下文模块 - 当前时间与状态】
{[时间描述串]}
{[活动描述串]}

【上下文模块 - 人格设定】
{人格设定内容（来自 admin_config persona）}

【上下文模块 - 关系状态】
当前关系等级：{level_name}
关系描述：{[关系描述]}
用户称呼：{[用户印象].称呼}

【上下文模块 - 近期对话】
{最近 N 轮对话记录}
（N 值与 Step5 保持一致，避免上下文不对齐）

【上下文模块 - 用户当前消息】
{本轮用户输入}

【任务】
根据以上内容，输出：
1. 你对用户消息的内心理解（InnerMonologue）
2. 用于检索角色背景设定的问题与关键词
3. 用于检索角色知识技能的问题与关键词
4. 用于检索用户相关记忆的问题与关键词
```

#### 输出 Schema（JSON）

```json
{
  "InnerMonologue": "对用户意图的内心理解，不超过 100 字",
  "CharacterSettingQueryQuestion": "用于检索角色设定的自然语言问题",
  "CharacterSettingQueryKeywords": "关键词1 关键词2 ...",
  "CharacterKnowledgeQueryQuestion": "用于检索角色知识的自然语言问题",
  "CharacterKnowledgeQueryKeywords": "关键词1 关键词2 ...",
  "UserProfileQueryQuestion": "用于检索用户相关记忆的自然语言问题",
  "UserProfileQueryKeywords": "关键词1 关键词2 ..."
}
```

**输出流转：** 本步骤产出写入本轮内存上下文（`[查询重写结果]`），供 Step2 消费。InnerMonologue 不持久化、不返回前端。

**[待确认]** 近期对话取多少轮？建议与 Step5 Prompt 保持一致（当前 Step5 用最近 10 轮）。

### Step2：多路向量检索（扩展）

**现有功能：** 基于用户当前输入的 Embedding，检索 TopK=5、阈值≥0.7 的用户记忆。

**改造内容：**

基于 Step1.5 输出的三组查询语句 + 关键词，分别执行三路向量检索：

| 检索路 | 查询来源 | 向量库 type 过滤 | 结果存入 |
|-------|---------|----------------|---------|
| 角色设定路 | `CharacterSettingQueryQuestion/Keywords` | `character_setting` | `[角色设定检索结果]` |
| 角色知识路 | `CharacterKnowledgeQueryQuestion/Keywords` | `character_knowledge` | `[角色知识检索结果]` |
| 用户记忆路 | `UserProfileQueryQuestion/Keywords` | `user`（加 user_id 过滤） | `[用户记忆检索结果]` |

**检索参数：**
- TopK：**[待确认]** 各路 TopK 是否统一为 5，或按路设置不同值
- 相似度阈值：≥0.7（与现有保持一致）
- 检索方式：向量检索 + 关键词补充合并（参考 A 项目做法）

**注意：** 现有 Step1 中已经生成 user Embedding 用于向量检索，改造后用户记忆路的查询向量改为 Step1.5 生成的 `UserProfileQueryQuestion` 的 Embedding，不再直接用原始用户消息的 Embedding。**[待确认]** 是否需要保留原始消息 Embedding 用于其他用途。

**结果流转：** 三路检索结果合并写入内存上下文，格式为可直接插入 Prompt 的文本块。

### Step5.5：响应优化 LLM（新增）

**位置：** Step5 完成后、Step7 SSE 推送前。

**触发条件（两个门闩，任一满足则触发）：**
- 门闩 A：随机数 < 0.12（无差别 12% 概率）
- 门闩 B：`ChatCatelogue == "是"` 且 随机数 < 0.50（约 50% 概率）
- 两条叠加：若 ChatCatelogue 为"是"，总触发概率约为 1 − (1−0.12)×(1−0.50) ≈ 56%

**未触发时：** 直接使用 Step5 的 `MultiModalResponses`，跳过本步骤。

**触发时：**

#### Prompt 结构

```
【系统指令】
你是 {虚拟人名称}，请对以下初步回复进行优化改写，使其更自然、更符合角色风格。
仅输出 JSON 数组，不含任何额外内容。

【人格设定】
{人格设定内容}

【关系状态】
{当前关系等级与描述}

【初步回复（待优化）】
{[Step5 的 MultiModalResponses] JSON 序列化}

【任务】
对上述回复进行润色优化，保持相同的结构格式（type / content / emotion 字段），
返回优化后的 JSON 数组。
```

#### 输出

```json
[
  {
    "type": "text",
    "content": "优化后的文本内容"
  },
  {
    "type": "voice",
    "content": "优化后的语音文案",
    "emotion": "情绪标签"
  }
]
```

**失败兜底：** 若 LLM 返回内容无法解析为合法 JSON 数组，丢弃本次优化结果，**回退使用 Step5 原始 `MultiModalResponses`**，不中断整条链路。（这是对 A 项目已知边界风险的改进）

**字段流转：** Step5.5 成功后，用优化结果覆盖内存上下文中的 `[多模态回复列表]`。Step7 SSE 始终读取该内存上下文中最新的 `[多模态回复列表]`。

### Step6：后台任务（扩展）

**现有异步任务保留不变。**

**新增内容（同步执行，在 SSE 推送完成后、本轮请求结束前）：**

#### 6.1 RelationChange 处理

- 读取内存上下文中的 `[关系变化量]`（来自 Step5 输出）
- 将 LLM 建议的增减量整合进现有 `RelationshipService.add_growth` 机制
- 执行边界限制（上下限处理，与现有 growth_value 量纲对齐）
- **[待确认]** 具体整合方式：读 `relationship_service.py` 确认是否直接调 `add_growth`，或单独写 `growth_value` 字段，避免与现有日限制逻辑冲突

#### 6.2 FutureResponse 写入

- 读取内存上下文中的 `[未来预约]`（来自 Step5 输出）
- 若 `FutureResponseAction != "无"`：将时间戳（解析自 `FutureResponseTime`）和意图摘要写入 `relationship` 表的未来预约字段
- 若 `FutureResponseAction == "无"`：将未来预约字段置空（清除旧预约）
- **注意：** 每轮都写（无随机门控），由 Step8 轮询时判断是否执行

#### 6.3 记忆 LLM 调用（新增，同步）

**触发时机：** 在上述 6.1、6.2 完成后执行，在本轮请求关闭前完成。

**Prompt 结构：**

```
【系统指令】
你是 {虚拟人名称}，请对本轮对话进行总结，提取有价值的记忆信息。
输出格式要求：仅输出 JSON，所有文本类字段的内容格式为多行 "key：value"（全角冒号分隔），
无内容时输出"无"。

【上下文模块 - 当前时间与状态】
{[时间描述串]}

【上下文模块 - 人格设定】
{人格设定内容}

【上下文模块 - 关系状态】
{当前关系信息（等级、描述、用户印象）}

【上下文模块 - 近期历史摘要】
{最近 N 轮历史对话（不含本轮）}

【上下文模块 - 本轮完整对话】
用户：{本轮用户输入}
{虚拟人名称}：{本轮 AI 回复（来自最终 MultiModalResponses 的 content 合并）}

【任务】
基于以上内容，提取并输出：
1. 本次对话中新增的角色公开背景信息
2. 本次对话中新增的角色私有信息
3. 本次对话中体现的角色知识或技能
4. 本次对话中获取的用户相关信息
5. 更新后的关系描述文案
6. 用户的真实称呼（若无变化或无法判断则输出"无"）
7. 用户的昵称/绰号（若无变化则输出"无"）
8. 用户的综合印象描述（若无变化则输出"无"）
```

**输出 Schema（JSON）：**

```json
{
  "CharacterPublicSettings": "公开设定1：内容\n公开设定2：内容",
  "CharacterPrivateSettings": "私有信息1：内容",
  "CharacterKnowledges": "知识点1：内容",
  "UserSettings": "用户信息1：内容",
  "RelationDescription": "对两人关系的文字描述",
  "UserRealName": "用户真实姓名或无",
  "UserHobbyName": "用户昵称/绰号或无",
  "UserDescription": "对用户的综合印象或无"
}
```

**`_posthandle` 逻辑（记忆 LLM 成功后）：**

1. **写向量库（4路）：**
   - 将各字段按 `\n`（或 `<换行>` 标记）拆分为多行
   - 每行格式须为 `key：value`（全角冒号），不满足则丢弃该行
   - 按 type 分别 upsert 到 DashVector（同 key 已存在则更新，否则新增）：
     - `CharacterPublicSettings` → type=`character_setting`，无 user_id
     - `CharacterPrivateSettings` → type=`character_setting`，携带 user_id（私有）**[待确认]** 是否需要区分公有/私有 type
     - `CharacterKnowledges` → type=`character_knowledge`，无 user_id
     - `UserSettings` → type=`user`，携带 user_id
   - 若某字段值为"无"，跳过该路写入

2. **写 DB（关系表扩展字段）：**
   - `RelationDescription` 非"无"：更新 `relationship` 表的关系描述字段
   - `UserRealName` 非"无"：更新用户真实称呼字段
   - `UserHobbyName` 非"无"：更新用户昵称字段
   - `UserDescription` 非"无"：更新用户印象描述字段

### Step7：SSE 推送（对接）

**改造点：** SSE 推送的内容来源从 Step5 直接输出，改为读取 Step5.5 之后内存上下文中的最终 `[多模态回复列表]`。

**推送方式：** 维持现有 SSE 协议（`type: delta` 切片 + `type: done` 结束），**[待确认]** 多模态消息列表如何切片推送：
- 方案 A：将 `MultiModalResponses` 各条 content 拼接后按字符切片 delta 推送，done 时带完整 emotion 与多模态结构
- 方案 B：每条 MultiModal 消息作为独立的 SSE 事件段推送

**情绪字段：** `type: done` 事件中的 emotion 改为从 Step5 输出的 `EmotionLabel` / `EmotionConfidence` 读取（保持与 emotion_log 对齐）。

### Step8：主动消息轮询 Handler（新增）

**定位：** 独立后台轮询进程，与现有定时巡视机制并存。

#### 8.1 Future 槽数据结构

存储位置：`relationship` 表扩展字段（或 **[待确认]** 独立表）

| 字段语义 | 类型 | 说明 |
|---------|------|------|
| 未来预约时间戳 | Integer，可空 | Unix 时间戳，到时触发 |
| 未来预约意图摘要 | String，可空 | 约 20 字的行动概要 |
| 主动消息计数 | Integer，默认 0 | 衰减惩罚用 |

**清零时机：** 用户发送新消息时（`POST /api/chat/send` 入口），将该用户的主动消息计数置 0。

#### 8.2 两路写入与优先级保护

| 写入路 | 触发条件 | 写入内容 | 优先级 |
|-------|---------|---------|-------|
| 路 A（对话驱动） | Step6 每轮结束 | LLM 规划的时间 + 意图 | 高 |
| 路 B（定时巡视） | 现有定时任务触发 | 周期性检查后生成的话题 | 低（兜底） |

**路 B 写入前的优先级检查：**
```
if future_timestamp 存在 AND future_timestamp > 当前时间:
    跳过写入（路 A 预约未过期，不覆盖）
else:
    正常写入（槽为空或已过期）
```

#### 8.3 轮询消费逻辑

**轮询频率：** **[待确认]**，参考 A 项目为每秒轮询。

**消费条件（同时满足）：**
1. `future_timestamp` 不为空
2. `future_timestamp` ≤ 当前时间（已到期）
3. `future_timestamp` > 当前时间 - 1800（30 分钟过期窗口，超期任务忽略）
4. 用户未被封禁

**消费后执行子链路：** 与主对话同源结构，全程使用主动消息专用 Prompt 版本：

```
触发条件满足
  → 加载用户上下文（关系、历史、时间、活动）
  → Step1.5 变体：查询重写（将 future.意图摘要 嵌入查询意图）
  → Step2：三路向量检索（同主对话）
  → Step3：拼装主动消息 Prompt（见下方 Prompt 结构）
  → Step5 变体：主动消息 LLM 调用
  → Step5.5：概率触发优化（default_refine_chance 可配置为较低值）
  → 写入发送队列 / agent_message 表（对接现有主动消息展示接口）
  → Step6 变体：记忆 LLM（与主对话相同逻辑）
  → 更新 relationship 表（主动消息计数 +1）
  → 判断是否写入下一轮预约（主动消息计数衰减门控）
```

**主动消息 Prompt 核心差异（相对主对话）：**

```
【用户消息】模块替换为：
【主动发起】
{虚拟人名称} 打算主动联系用户。
当前规划的行动：{future.意图摘要}
请生成一条自然的主动发起消息。
```

**写入下一轮预约的门控（主动消息专用）：**
- 主动消息计数（proactive_times）每次执行后 +1
- 若 LLM 输出了新的 FutureResponse：以 `0.15^(proactive_times+1)` 概率决定是否写入下一轮预约（越打扰越低概率继续）
- 未命中：清空 future 槽
