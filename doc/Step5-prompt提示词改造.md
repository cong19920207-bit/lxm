### 模块 1-② 【回复格式规则】— 替换

**▶ 删除（旧）：**

```
【回复格式规则】
你的回复必须是1-3个短句，禁止使用列表、分点、Markdown格式、长段落。
```

**▶ 替换为（新）：**

```
【回复格式规则】
你的回复通过 messages 数组输出，每个元素是一条独立的聊天气泡。
每条气泡是一个自然的语义或情感单元，禁止使用 Markdown 格式、列表、分点。
messages 每条的 type 字段必须精确等于字符串 text（大小写敏感）。
```

**改动说明：**

- 删除"1-3个短句"——新 Schema 允许 1-5 条，条数由模型根据内容自然决定（R-BND-03）
- 增加 `type` 字段约束说明——CP3 要求，违反则整轮解析失败
- "禁止 Markdown/列表/分点"的精神保留

### 模块 1-④ 【结构化输出指令】— 完全替换

**▶ 删除（旧）：**

```
【结构化输出指令】
你必须以以下JSON格式返回，不可输出JSON以外的任何内容：
{"emotion": {"label": "从[开心/悲伤/焦虑/愤怒/孤独/疲惫/平静]中选一", "confidence": 0.00}, "reply": "你的回复内容"}
```

**▶ 替换为（新）：**

```
【结构化输出指令】
你必须仅输出合法 JSON，不含任何前缀、后缀、markdown 标记或注释。
JSON 结构如下，字段名区分大小写，严格遵守：

{
  "inner_monologue": "你对本轮消息的内心想法，不超过150字，不展示给用户",
  "messages": [
    { "type": "text", "content": "第一条气泡正文" },
    { "type": "text", "content": "第二条气泡正文（可选）" }
  ],
  "relation_change": { "delta": 0 },
  "future": { "time_natural": "无", "action": "无" },
  "emotion": { "label": "从[开心/悲伤/焦虑/愤怒/孤独/疲惫/平静]中选一", "confidence": 0.00 },
  "knowledge_expand": "否"
}

【字段说明】
inner_monologue：先写内心独白再构思回复，内心独白不展示给用户，不落入对话记录。
messages[].type：必须精确等于字符串 text，大小写敏感，其他任何值均导致本轮失败。
messages[].content：该条气泡正文，不为空。
relation_change.delta：整数，正数表示关系加温，负数表示关系降温，0表示无明显变化。
future.time_natural：对话中有明确约定或承诺时，填自然语言时间描述（如「明天下午」）；无则填「无」。
future.action：对应的主动联系意图，约20字；无约定则填「无」。
emotion.label：从上述7个标签中选一，描述你本轮的情绪状态。
knowledge_expand：本轮回复是否扩展了知识性内容。仅填「是」或「否」，其他值按「否」处理。

【输出示例】
{
  "inner_monologue": "他又提到熬夜了，语气比上次轻松，可能只是抱怨。先轻轻接一句，再问具体情况。",
  "messages": [
    { "type": "text", "content": "又熬到几点啊" },
    { "type": "text", "content": "说说是什么事让你睡不着" }
  ],
  "relation_change": { "delta": 1 },
  "future": { "time_natural": "无", "action": "无" },
  "emotion": { "label": "平静", "confidence": 0.85 },
  "knowledge_expand": "否"
}
```

**改动说明：**

- 原 `reply` 字段整体替换为 `messages` 数组（R-BND-03）
- 原 `emotion` 字段结构保留（含原有7个标签），位置调整为顶层独立字段（§2.7.3 整轮唯一情绪）
- 新增 `inner_monologue`、`relation_change`、`future`、`knowledge_expand` 四个字段（§2.7.7）
- 加入 few-shot 输出示例（R-MEM-06 建议）

### 模块 3 【关系状态】— 尾部追加

**▶ 现有内容（不动）：**

```
【关系状态】
当前关系等级：{level_name}

（距上次互动修正：若有则追加，无则不显示）
```

**▶ 在现有内容末尾追加（新增4行）：**

```
关系描述：{relation_description，若为空则填「暂无，初次互动」}
对TA的印象：{user_description，若为空则整行不输出}
亲密称呼：{user_hobby_name，若为空或 NULL 则填「无」}
用户真名：{user_real_name，若为空或 NULL 则填「无」}
```

**改动说明：**

- 上述字段均来自 `relationship` 表新增列（R-MEM-05）
- **亲密称呼**（`user_hobby_name`）与 **用户真名**（`user_real_name`）须 **同时** 提供给模型，**不得**在服务端二选一合并为一列；任一侧无数据时 Prompt 中显式写 **「无」**，避免模型误以为缺失的一方不存在于业务规则之外
- 实际回复中称呼对方时用亲密称呼、真名或暂不使用姓名，由 **LLM 结合语境与人格** 自行判断，不在拼装层写死优先级
- `user_description` 若为 NULL 则整行不输出，避免出现"对TA的印象：（空）"的噪声

### 模块 ★ 【当前时间】— 新增，插入在模块5和模块6之间

**▶ 新增内容：**

```
【当前时间】
{time_description}
```

`time_description` 由服务端代码根据系统时间生成，无需 LLM，无需 DB 查询。

**生成格式示例：** `现在是周四晚上10点31分`

**生成逻辑（伪代码）：**

```python
weekdays = ["周一","周二","周三","周四","周五","周六","周日"]
# hour < 5  → "凌晨"
# 5  <= hour < 9  → "早上"
# 9  <= hour < 12 → "上午"
# 12 <= hour < 14 → "中午"
# 14 <= hour < 18 → "下午"
# 18 <= hour < 21 → "傍晚"
# 21 <= hour < 24 → "晚上"
# 输出：f"现在是{weekday}{period}{hour}点{minute:02d}分"
```

**改动说明：**

- 需求文档 Step1 明确：时间描述"无需 LLM，无需 DB 查询，代码直接根据系统时间生成"
- 活动状态（如"她现在应该在午休"）依赖 Step1 数据源，**当前阶段不加**，Step1-3 改造时补入此模块
- 插入位置与需求文档 Step3 新增模块 B 建议位置一致（情绪状态之后、最近对话之前）

## 四、改动与解析侧的呼应关系

| Prompt 改动点 | 呼应的服务端解析侧改动 | 需求依据 |
|--------------|----------------------|---------|
| `messages` 替换 `reply` | 解析函数从取 `data["reply"]` 改为取 `data["messages"]`，并做 U2 空数组校验 | §2.9.6 U2 |
| `messages[].type` 约束写入 Prompt | CP3：type 非精确 `text` → 整轮 Step5 失败，进叹号路径，不进 Step5.5，不入队 Step6 | §2.9.6 CP3 |
| `knowledge_expand` 字段写入 Prompt | U1：解析后 trim，仅精确"是"才是"是"，其余一律"否"，不因此单独判整轮解析失败 | §2.9.6 U1 |
| `future` 字段写入 Prompt | Step6 异步读取，写入 `relationship` 表 future 槽 | R-FUT-01/02 |
| `relation_change.delta` 字段写入 Prompt | Step6 整合进 `RelationshipService`（本阶段映射细节暂缓） | R-BND-09 |
| `inner_monologue` 字段写入 Prompt | 不返前端，不落 `conversation_log`，仅供本轮内存日志 | R-MEM-02 |
| 关系状态追加字段（含亲密称呼与用户真名 **同显**） | Step6 执行后写回 `relationship` 表，下一轮 Step1 读取时即可带入；Step5 拼装时两列同时注入 Prompt、空为「无」 | R-MEM-05 |
| 【知识性话题回应原则】写入 System | 仅影响生成风格与信息密度；**无**解析字段变更；与 **【人格设定】** 一并受管理后台可编辑策略约束 | R-BND-03，§2.7.9 |
