**变更 8.1 — Future 槽消费绕过频控（v0.31 拍板）**

- **现状**：现网 `AgentService.check_and_trigger`（`backend/services/agent_service.py:142-160`）对所有主动消息执行三项频控：①每日上限、②两次间隔、③黑名单；定时扫描 P0~P4 全程必过此三项。
- **决策**：当 Step8 子链路消费 Future 槽（来自对话 Step5 + Step6 写入的 `future.action`）触发主动消息时，**绕过①②频控**，**保留③黑名单 + 行动评分≥6**两项约束（与时间无关）。
- **理由**：Future 槽体现的是「对话中明确约定/承诺」的特定时间，是产品最高优先级（用户已感知到约定），不应被运营级频控阻挡；定时扫描 P0~P4 是「兜底关怀」，频控阻挡合理。
- **副作用控制**：消费成功后**额外计入** `agent:count:{user_id}:{date}` 计数器，让定时扫描 P0~P4 看到正确剩余次数（避免一天内 Future + 定时扫描合计远超 8 次）。

**变更 8.2 — 全局频控调整（v0.31 拍板）**

| 参数 | 现网 | 调整后 |
|------|------|--------|
| 每日上限 | 2 次 | **8 次**（含 Future 槽消费计入） |
| 两次间隔 | ≥6 小时 | **≥30 分钟** |

- **影响清单**：
  1. `backend/services/agent_service.py:144`（`>= 2` → `>= 8`）
  2. `backend/services/agent_service.py:158`（`timedelta(hours=6)` → `timedelta(minutes=30)`）
  3. **`.cursorrules`「Agent 主动消息5种触发」段** 已**同步更新**（v0.31 同步），含 Future 槽消费规则
  4. **TD-004**（Agent 规则后台可配但运行时未读）：本次直接改硬编码值；TD-004 清偿后由 `admin_config:agent_rules` 接管
- **业务面提示**：从 2 次/天提升到 8 次/天 = 4 倍主动消息频率，叠加 30min 间隔（原 6h）= 12 倍触发密度上限。**已经产品确认**，灰度发布需观察用户打扰投诉率。

**Future 槽消费失败决策（v0.31 拍板）**

- 黑名单/行动评分不通过 → **清空槽位**（不推迟），写 §2.9.5 Q5 结构化日志
- 不引入「推迟到下一轮重试」机制，避免 Future 槽长期占用与反复触发

> **实现提示（非需求）**：路 B「不覆盖槽」与「仍发 agent_message」需在代码层区分「写槽」与「发消息」两个动作；变更 8.1 中「Future 槽消费」是 Step8 子链路新通道，与定时扫描 P0~P4 是两套独立写 `agent_message` 的路径，共享同一计数器但**不共享**频控判定。

#### R-MEM-04 实施清单（v0.31 新增）

> **背景**：v0.31 修订前，计划稿 Step6._posthandle 错误地把 `CharacterPrivateSettings` 写为 `type=character_setting + user_id`，与 R-L1L3-08「四类 type 全局唯一」直接冲突，会导致 Step2 按 `type='character_private'` filter 时**0 命中**，等同私有记忆永久失忆。本表为最终落地依据。

| Step6 输出字段 | DashVector `type` | 是否携带 `user_id` | Step2 检索路 | 备注 |
|---------------|------------------|-------------------|------------|------|
| `CharacterPublicSettings` | **`character_global`** | **否** | 角色设定路 - 公有 | 与管理后台「角色知识库 CRUD」录入的 `character_global` 共用 type |
| `CharacterPrivateSettings` | **`character_private`**（**非** `character_setting + user_id`） | **必须** | 角色设定路 - 私有 | **不在**管理后台手动录入，仅由 Step6 自动写入 |
| `CharacterKnowledges` | **`character_knowledge`** | **否** | 角色知识路 | 与管理后台「角色知识库 CRUD」录入的 `character_knowledge` 共用 type |
| `UserSettings` | **`user`** | **必须** | 用户记忆路 | 与现网 `memory_service` 写入路径合并（一致）|

**写入约束**：
1. 字段值为「无」时**整路跳过**，不写空文档
2. 字段非「无」时按 `\n` 拆行，每行须为 `key：value`（中文全角冒号，R-MEM-01）；不满足该格式的行**丢弃**
3. 文档键策略：`memory_type` + 稳定 key（如 `外貌-体态`） + user_id 维度（仅 `character_private` 与 `user`）→ 生成稳定 doc_id
4. 同 key 再次输出 → **整条覆盖**并重算 embedding；不同 key 并存；本轮未再出现的 key **不自动删除**


**与计划稿的对应关系**：**仅**将 **A 类（R-FUT、R-MEM）** 与 Step6/Future 强相关段落同步到 [对话链路改造需求文档.md](./对话链路改造需求文档.md)。**B 类（R-AGT、R-VEC）** 在计划稿中保留为**目标态**，文内不再使用「已与 lxm 对齐」字样误导。

### 2.7 Step5～Step7 契约细节与 JSON 指导（**已确认，开发指导用**）

> 本节为**实现指导**：字段名、数值可在编码期微调，但**语义与层级**以本节为准；**代码稳定后再反向更新** `docs/contract.md`（与 R-BND-11 一致）。**现网契约表述**可参考 `docs/contract.md` **POST `/api/chat/send`**（如 `delta` / `done` 与 H5 `chatSendSession`、`meta.generation_id` 等）；**实现与契约差异以本记录 + 发版说明为准，直至回写 contract**。

#### 2.7.1 Step5 主模型输出（**A1 扁平 + `messages` 数组**）

**整表进入 Step5.5**：`messages` 数组内**每一条**均参与 5.5 的**单次**整表优化（见 R-BND-04）；5.5 可把 **1 条拆成多条、多条并成 1 条或任意条数（仅 text）**；**优化后** `messages` **最多 5 条**（**J**，硬上限，可配置）。

**Step5.5 触发（B1 + B3 + `knowledge_expand` 加权；v0.31 公式定稿）**

> **v0.31 修订要点**：原稿 `rand < min(1, 0.5+0.5) = 1.0` 数学上恒为真（即知识扩展 100% 触发）与产品意图「`knowledge_expand=='是'` 是**增加概率**而非必触发」不符，已按候选 a 修订为门闄 B 独立 50%。

- **优先级总览（自上而下，前者关闭则跳过后者全部判定）**：
  1. **管理后台总开关（B3）**：管理后台**独立页面**提供「是否启用 Step5.5」（G1）；**关闭时**永远不调用 Step5.5（与计划稿 §2.7.6 真话表第一行一致）。仅**具备相应权限的管理员**（如超级管理员，实现时与 RBAC 对齐）可改
  2. **门闄 A（无差别 12%）**：`rand < 0.12`，**固定值**，不因 `knowledge_expand` 改变（满足「保留原来的 12% 优化概率」）
  3. **门闄 B（知识扩展独立 50%）**：仅当 Step5 JSON 中 **`knowledge_expand == "是"`** 时生效，**`rand < 0.5`**（独立随机数，不与门闄 A 共享）；当 `knowledge_expand == "否"` 时**不进入门闄 B**
- **触发条件**：总开关开启 **AND**（命中门闄 A **OR** 命中门闄 B）→ 执行 Step5.5
- **总触发率**：
  - `knowledge_expand == "是"`：`1 − (1−0.12)×(1−0.5) = 56%`
  - `knowledge_expand == "否"`：`12%`
- **运营预估**：假设线上 `knowledge_expand=='是'` 占比 **30%**（产品后续可调），整体 Step5.5 触发率 ≈ `30% × 56% + 70% × 12% = 25.2%`；用于成本与时延评估输入（详见 §2.11）
- **`knowledge_expand`**：字符串 **「是」|「否」**；与计划稿 **ChatCatelogue / 知识扩展型** **同源语义**；**不再使用**独立字段 `chat_better`。  

#### 2.7.2 Step5.5 输出形态（**F**）

- **输出**：与 Step5 **同形**——**仅** `messages` **JSON 数组**（元素为 `{ "type": "text", "content": "..." }`），**条数可与 Step5 输入不同**（可拆、可并）；**仍全部为文字**；解析失败回退 Step5（R-BND-06）。  
- **条数上限**：优化后 **≤ 5 条**（**J**）；若模型仍输出多于 5 条，服务端 **合并**规则见 **§2.9.1**（**不再**采用未定义的「截断或合并二选一」模糊表述）。

#### 2.7.3 情绪（**E1**）

- **整轮一个 `emotion`** 对象，置于 Step5 JSON 顶层及 SSE `done` 中（与现网「`done` 含 emotion」一致）；**不为每条 `messages[i]` 单独配情绪**（后续若上 voice 再议）。

#### 2.7.4 超时（**D2**）

- **Step5.5 独立子超时**：**30 秒**；超时或解析失败 → **回退 Step5 原始 `messages`**（R-BND-06）。  
- **整轮 bundle 总等待（v0.31 修订）**：原稿建议「55s → 90s」**已被 §2.11.2 关键超时数字对齐表覆盖**，最终值为 **`_BUNDLE_WAIT_TIMEOUT_SEC = 120s`**（覆盖整链最坏路径含 Step1.5 `chat_sync` 内层多 POST、Step5.5 + Step5）；Nginx `proxy_read_timeout` 建议 **≥ 130s**。详见 §2.11.2。

#### 2.7.5 Step7 / H5 客户端（**C + H**）

- **客户端**：**仅 H5**（如 `frontend/pages/chat.html`），**与后端同步改版**；**不考虑**旧版 App 兼容。  
- **展示**：**收到 `done` 后**再按 **`done` 内有序 `messages`** 渲染 **N 个独立气泡**（N = 数组长度，≤5）；**`delta` 仅作过程态**（打字机/预览），**最终以 `done` 为准**（与先前 C 建议一致）。  
- **多气泡流式（与 §2.9.4 一致）**：SSE 需扩展为**按条**推送增量（如带 **`message_index`**）；端上**不得**按预期条数 **N 预先铺 N 个空气泡**，应**按服务端下发顺序**逐条挂载/更新气泡；**数组下标 = 展示顺序**，见 §2.9.4。  

#### 2.7.6 叹号、重发与 5.5 判定（**I — 真话核对**）

| 陈述 | 结论 |
|------|------|
| 未跑 5.5 或 5.5 失败回退时，用户看到的是 **Step5** 的 `messages` | **对**（R-BND-06）。 |
| **叹号**表示 **Step5 未完成/失败**，不是「仅 5.5 失败」 | **对**；仅 5.5 失败应回退 Step5 且无叹号（若 Step5 已成功）。 |
| 用户点叹号 **重发** ≈ **新一轮** `send`/`resend`，会重新走 Step5，再按**当轮**开关与门闄决定是否 5.5 | **对**。 |
| 「5.5 是否跑**只**由对话自己判断」 | **不严谨**：除对话内容外，还依赖 **(1) 管理后台总开关** **(2) 门闄 A 固定 12%** **(3) 仅当 `knowledge_expand` 为「是」时才存在的门闄 B（带 +0.5）**。其中 **`knowledge_expand` 由 Step5 模型输出**，故「是否跑 5.5」是 **配置 + 随机 + 模型字段** 的组合，不是单一「对话引擎自己」判断。 |
| Step5 成功后，若总开关 **关**，则 **不跑 5.5** | **对**。 |

#### 2.7.7 Step5 JSON 示例（**无行内注释**；含义见下表）

```json
{
  "inner_monologue": "本轮内部思考，不展示给用户。",
  "messages": [
    { "type": "text", "content": "第一条给用户的气泡正文" },
    { "type": "text", "content": "第二条" },
    { "type": "text", "content": "第三条" }
  ],
  "relation_change": { "delta": 0 },
  "future": { "time_natural": "无", "action": "无" },
  "emotion": { "label": "平静", "confidence": 0.92 },
  "knowledge_expand": "否"
}
```

| 字段路径 | 含义 |
|----------|------|
| `inner_monologue` | 字符串。角色内心独白；**不返前端**；不落主会话正文（与 R-MEM 总结用 Inner 职责区分）。 |
| `messages` | **数组**。每条为一条可展示消息（**一条 = 一个独立气泡**）；**当前仅** `type: "text"`；**展示顺序 = 数组下标顺序**（**不新增** `order` / `seq` 等字段）；**整表**交给 Step5.5；**5.5 输出后 ≤5 条**（超限合并见 §2.9.1）。 |
| `messages[].type` | 字符串。消息类型；当前**仅允许** JSON 中**精确等于**字面量 **`text`**（**大小写敏感**，值内**无**首尾空格；如 `"text "`、`"Text"`、缺失该字段、或其它取值均**不合法**）。**不合法** → **整轮 Step5 JSON 解析失败**，按 **Step5 失败**处理（叹号，**不**进入 Step5.5，**不入队** Step6；**确认点 3**）。 |
| `messages[].content` | 字符串。该条气泡正文。 |
| `relation_change.delta` | 整数。关系增量建议；**与 growth 映射仍按 R-BND-09 暂缓**。 |
| `future.time_natural` | 字符串。**严格 3 种格式之一**（**v0.31 收口**，避免服务端解析歧义）：① `「N 分钟后」/「N 小时后」/「N 天后」`（N 为正整数）；② `「今天 HH:MM」/「明天 HH:MM」`（24 小时制，HH 与 MM 均两位数字）；③ `「无」`（表示本轮无预约）。**任何其它格式**（含「两小时后」「明天上午」「下周一」等含糊表述）服务端**一律视为「无」**处理（清空槽，**保留** `proactive_times`，写 §2.9.5 Q5 结构化日志，**不**计入 §2.8.4 Future 写入失败重试）。Prompt 中须 few-shot 强约束此 3 种格式（详见 [Step5-prompt提示词改造.md](./Step5-prompt提示词改造.md) 模块 1-④ 字段说明），降低模型输出歧义概率。 |
| `future.action` | 字符串。主动预约意图摘要或 **`无`**。约 20 字。 |
| `emotion.label` / `emotion.confidence` | **整轮唯一**情绪；供 `done` 与 emotion 落库（**E1**）。 |
| `knowledge_expand` | **「是」|「否」**。为「是」时启用 **门闄 B**（基线 0.5 **+0.5** → 1.0）；为「否」时 **仅可能** 被 **门闄 A（12%）** 命中。**规范化（U1）**：先做 **trim**；仅当值**精确等于**「是」时按「是」参与门闄 B；**其余任意无法识别值（含空串、错别字、英文等）一律按「否」**，**不**因此单独判整轮 Step5 JSON 解析失败。 |

#### 2.8.4 附 · `future.time_natural` 时间解析规则（v0.31 新增）

> **背景**：§2.7.7 已将 `future.time_natural` 严格约束为 3 种格式，本段定义服务端解析与失败行为。

- **解析方式**：**纯 Python 正则解析**（不引入第三方库 `dateparser` / `parsedatetime` 等，避免中文 NLP 不可控与依赖膨胀，也不再二次调用 LLM 解析以免增加链路时延）
- **时区基准**：**服务器 UTC**（与 `agent_service.py` 等现网时间逻辑对齐，使用 `datetime.utcnow()`）
- **3 种格式正则**：
  - `「N 分钟后」/「N 小时后」/「N 天后」` → 匹配 `^(\d+)(分钟|小时|天)后$`，按当前 UTC 时间加偏移
  - `「今天 HH:MM」/「明天 HH:MM」` → 匹配 `^(今天|明天) (\d{2}):(\d{2})$`，按 UTC 当前日期 / 次日计算
  - `「无」` → 视为「本轮无预约」，清空 `relationship.future_*` 字段
- **解析失败行为**（统一处理，与「无」等价）：
  1. **清空槽位**（`future_timestamp` / `future_action` 置 NULL）
  2. **保留** `proactive_times` 字段值不变（不重置）
  3. 写 §2.9.5 Q5 结构化日志（含 `user_id`、原始 `time_natural` 字符串、解析失败原因）
  4. **不**计入 §2.8.4「Future 写入失败重试」语义（解析失败 ≠ DB 写入失败）
- **30 分钟过期窗口**（与计划稿 Step8 8.3 一致）：解析得到的时间戳若已过期超过 30 分钟（`now - future_timestamp > 1800s`），Step8 轮询消费时**忽略并清空**该槽

> **与计划稿 D 的关系**：**不再要求**「Step6 记忆 LLM 必须同步阻塞至本轮 HTTP 结束」；以本节 **M2 + 重试** 为准。

#### 2.9.1 **Q1**：Step5.5 解析后仍 **>5 条** 时的合并规则

- **超长**：合并后的第 5 条总长可设 **可配置上限**；超长时对该条 **`content` 尾部截断**（并打日志/监控），保证 ≤5 **条数** 与单条长度护栏同时成立。

#### 2.9.4 **Q4**：多气泡 JSON 顺序与 **SSE 按条流式**

**JSON（Step5 / 最终 `done.messages` 同源）**

- **一条 `messages[i]` = 一个独立气泡**；**排序 = `messages` 数组下标顺序**（从 **0** 起）；**不新增** `order`、`seq` 等冗余字段（与 §2.7.7 表一致）。

**SSE 扩展（与服务端、H5 对齐）**

- 为实现「**过程态也是按条气泡**」，须在现有 `delta`（或等价事件）上 **扩展字段**，至少包含：  
  - **`message_index`**（整数，**0 ≤ index < N**）：本条增量属于 **`messages[message_index]`** / 最终 **`done.messages[message_index]`**，与 JSON 数组下标一致；**允许**与 **`done` 到达顺序不一致** 的**乱序**到达，客户端按 index **填槽**（**U4-A**）。  
  - **`message_count`**（整数，**N**）：本轮助手气泡条数上界，与最终 **`done.messages.length`** 一致；**不**用于预铺空气泡，仅辅助客户端与调试（**U4-C**）。  
  - **发包顺序（确认点 2）**：在发出**任意**带有 **`message_index`** 的 `delta`（或等价流式事件）**之前**，服务端**必须先**发出携带 **`message_count`** 的包；**推荐**与 **`generation_id`**（或现网 **`meta.generation_id`**）**同一首包或紧邻前置包**发出，便于客户端丢弃旧轮。若违反「先 `message_count`、后带 index 的 delta」顺序，视为**服务端实现缺陷**（联调与 CI 可检）。  
  - **文本增量**：与现网键名对齐（如沿用 **`delta`** 字符串表示**追加片段**），在契约中写死语义（**追加** vs 全文覆盖二选一，**推荐追加**）。  
  - **`generation_id`**（或沿用现网 **`meta.generation_id`**）：标识**整轮**生成；用于丢弃上一轮未闭合流、与收尾 **`done`** 关联。  
- **端上行为**：**禁止**根据「预期 N 条」预先创建 **N 个空气泡**；按服务端**下发顺序**为对应 `message_index` **挂载或更新**气泡（先来先绑下标）。

#### 2.11.3 Token 成本估算（参考值，待产品确认）

> 仅做量级对比，具体单价与流量预估需运营提供。

- **基线**：原 `MAX_TOTAL_TOKENS = 4096`，单轮主 LLM 调用 ≤ 4096 input + ~500 output

  - Step5 主 LLM：`MAX_TOTAL_TOKENS = 7373`（×1.8），单轮 input ≤ 7373 + ~500 output

## 9. Step4 内容安全与新结构化输出兼容性（**v0.31 新增**）

> 计划稿 Step4 标「无变动」，但 Step5 输出从单条 `reply` 字符串变为 `messages` 数组（最多 5 条）+ `inner_monologue`，须显式定义新结构下的检测范围与失败行为。**本节方案 A 已拍板**。

### 9.1 检测范围

| 检测对象 | 是否过内容安全 | 失败行为 |
|---------|-------------|---------|
| **对外 `messages[].content` 逐条** | **是**（每条独立检测） | 任一条违规 → **整轮 Step5 失败**（按 CP3 等同处理） |
| **`inner_monologue`** | **是**（但失败仅写日志） | 失败**不**触发整轮拦截；仅写 §2.9.5 Q5 结构化日志（避免内心独白触发整轮拦截，且内心独白仅供 Step6 记忆参考，违规内容不入记忆向量） |
| Step5.5 输出的 `messages[].content` | **是**（每条独立检测） | 任一条违规 → **回退 Step5 原始 `messages`**（与 R-BND-06 一致），若 Step5 也违规则整轮失败 |
| Step6 输出（向量记忆 + 关系标量） | **否**（不再过 Step4） | Step6 输出已是后台记忆，不返前端，且经 Step5 已过过 Step4，无需重复 |

### 9.2 整轮 Step5 失败处理

- **触发条件**：任一对外 `messages[].content` 内容安全检测失败
- **处理流程**：
  1. **不**进入 Step5.5
  2. **不**入队 Step6
  3. user 行按 `_mark_pack_failed` 标 `failed_blocked`（或现网等价字段）
  4. SSE 走叹号路径，与现网 `chat.py` 行为一致
  5. `inner_monologue` 已生成的部分写日志即可，不入记忆
- **重发**：用户点叹号重发 → 新一轮 `send`/`resend` → 重新走 Step5 → 重新检测

### 9.3 `inner_monologue` 检测失败的特殊处理

- **不拦截整轮**：`inner_monologue` 是模型内部思考，违规多源于模型对用户原话的复述或推断，本身不影响用户体验
- **仅写日志**：写 §2.9.5 Q5 「内心独白内容安全失败」分类日志（建议**不告警**，仅作排查素材）
- **不入 Step6**：Step6 入队前过滤掉违规的 `inner_monologue`（替换为空字符串「」），避免污染记忆向量
- **业务面意义**：让模型保留思考自由，但避免违规内容长期存于 DashVector 与 `relationship` 列
