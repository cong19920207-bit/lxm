### 2.7 Step5～Step7 契约细节与 JSON 指导（**已确认，开发指导用**）

> 本节为**实现指导**：字段名、数值可在编码期微调，但**语义与层级**以本节为准；**代码稳定后再反向更新** `docs/contract.md`（与 R-BND-11 一致）。**现网契约表述**可参考 `docs/contract.md` **POST `/api/chat/send`**（如 `delta` / `done` 与 H5 `chatSendSession`、`meta.generation_id` 等）；**实现与契约差异以本记录 + 发版说明为准，直至回写 contract**。

#### 2.7.1 Step5 主模型输出（**A1 扁平 + `messages` 数组**）

**整表进入 Step5.5**：`messages` 数组内**每一条**均参与 5.5 的**单次**整表优化（见 R-BND-04）；5.5 可把 **1 条拆成多条、多条并成 1 条或任意条数（仅 text）**；**优化后** `messages` **最多 5 条**（**J**，硬上限，可配置）。

**Step5.5 触发（B1 + B3 + `knowledge_expand` 加权；已按产品修订）**

- **总开关（B3）**：管理后台 **独立页面** 提供 **「是否启用 Step5.5」**（**G1**）；**关闭时**永远不调用 Step5.5。仅 **具备相应权限的管理员**（如超级管理员，实现时与 RBAC 对齐）可改。  
- **开启时（双门闄 OR）**：  
  - **门闄 A（保留原 12%）**：**无条件**参与；`rand < p_A`，**固定** **`p_A = 0.12`**，**不因 `knowledge_expand` 改变**（满足「保留原来的 12% 优化概率」）。  
  - **门闄 B（仅当知识扩展）**：**仅当** Step5 JSON 中 **`knowledge_expand == "是"`** 时，第二门闄才生效；此时 `rand < min(1, 0.5 + 0.5)`，即基线 **0.5** 再 **+0.5** 后 **= 1.0**（随机段恒过）。当 **`knowledge_expand == "否"`**：**不进入门闄 B**（**否不增加**、也不存在第二路随机）。  
  - **命中门闄 A 或门闄 B** → 执行 Step5.5。  
- **`knowledge_expand`**：字符串 **「是」|「否」**；与计划稿 **ChatCatelogue / 知识扩展型** **同源语义**；**不再使用**独立字段 `chat_better`。  

#### 2.7.2 Step5.5 输出形态（**F**）

- **输出**：与 Step5 **同形**——**仅** `messages` **JSON 数组**（元素为 `{ "type": "text", "content": "..." }`），**条数可与 Step5 输入不同**（可拆、可并）；**仍全部为文字**；解析失败回退 Step5（R-BND-06）。  
- **条数上限**：优化后 **≤ 5 条**（**J**）；若模型超出，服务端 **截断或合并**策略实现时定义并写入注释/配置。

#### 2.7.3 情绪（**E1**）

- **整轮一个 `emotion`** 对象，置于 Step5 JSON 顶层及 SSE `done` 中（与现网「`done` 含 emotion」一致）；**不为每条 `messages[i]` 单独配情绪**（后续若上 voice 再议）。

#### 2.7.4 超时（**D2**）

- **Step5.5 独立子超时**：**30 秒**；超时或解析失败 → **回退 Step5 原始 `messages`**（R-BND-06）。  
- **整轮 bundle 总等待**：在现网基线上 **额外预留 5.5 的 30s**（实现上建议将 `_BUNDLE_WAIT_TIMEOUT_SEC` 等调至 **≥ 原值 + 30s**，例如由 **55s → 90s**；**与 `docs/contract.md` 中 Nginx 超时建议** 一并上调，避免网关先断）。

#### 2.7.5 Step7 / H5 客户端（**C + H**）

- **客户端**：**仅 H5**（如 `frontend/pages/chat.html`），**与后端同步改版**；**不考虑**旧版 App 兼容。  
- **展示**：**收到 `done` 后**再按 **`done` 内有序 `messages`** 渲染 **N 个独立气泡**（N = 数组长度，≤5）；**`delta` 仅作过程态**（打字机/预览），**最终以 `done` 为准**（与先前 C 建议一致）。  

#### 2.7.6 叹号、重发与 5.5 判定（**I — 真话核对**）

| 陈述 | 结论 |
|------|------|
| 未跑 5.5 或 5.5 失败回退时，用户看到的是 **Step5** 的 `messages` | **对**（R-BND-06）。 |
| **叹号**表示 **Step5 未完成/失败**，不是「仅 5.5 失败」 | **对**；仅 5.5 失败应回退 Step5 且无叹号（若 Step5 已成功）。 |
| 用户点叹号 **重发** ≈ **新一轮** `send`/`resend`，会重新走 Step5，再按**当轮**开关与门闄决定是否 5.5 | **对**。 |
| 「5.5 是否跑**只**由对话自己判断」 | **不严谨**：除对话内容外，还依赖 **(1) 管理后台总开关** **(2) 门闄 A 固定 12%** **(3) 仅当 `knowledge_expand` 为「是」时才存在的门闄 B（带 +0.5）**。其中 **`knowledge_expand` 由 Step5 模型输出**，故「是否跑 5.5」是 **配置 + 随机 + 模型字段** 的组合，不是单一「对话引擎自己」判断。 |
| Step5 成功后，若总开关 **关**，则 **不跑 5.5** | **对**。 |

#### 2.7.7 Step5 JSON 示例（**无行内注释**；含义见下表）

```json
{
  "inner_monologue": "本轮内部思考，不展示给用户。",
  "messages": [
    { "type": "text", "content": "第一条给用户的气泡正文" },
    { "type": "text", "content": "第二条" },
    { "type": "text", "content": "第三条" }
  ],
  "relation_change": { "delta": 0 },
  "future": { "time_natural": "无", "action": "无" },
  "emotion": { "label": "平静", "confidence": 0.92 },
  "knowledge_expand": "否"
}
```

| 字段路径 | 含义 |
|----------|------|
| `inner_monologue` | 字符串。角色内心独白；**不返前端**；不落主会话正文（与 R-MEM 总结用 Inner 职责区分）。 |
| `messages` | **数组**。每条为一条可展示消息；**当前仅** `type: "text"`；**顺序 = 客户端展示顺序**；**整表**交给 Step5.5；**5.5 输出后 ≤5 条**。 |
| `messages[].type` | 字符串。消息类型；当前固定 **`text`**。 |
| `messages[].content` | 字符串。该条气泡正文。 |
| `relation_change.delta` | 整数。关系增量建议；**与 growth 映射仍按 R-BND-09 暂缓**。 |
| `future.time_natural` | 字符串。自然语言时间或 **`无`**。 |
| `future.action` | 字符串。主动预约意图摘要或 **`无`**。 |
| `emotion.label` / `emotion.confidence` | **整轮唯一**情绪；供 `done` 与 emotion 落库（**E1**）。 |
| `knowledge_expand` | **「是」|「否」**。为「是」时启用 **门闄 B**（基线 0.5 **+0.5** → 1.0）；为「否」时 **仅可能** 被 **门闄 A（12%）** 命中。 |

#### 2.7.8 Step5.5 输出 JSON 示例（**仅数组**）

```json
[
  { "type": "text", "content": "润色后可能合并或拆分后的第一条" },
  { "type": "text", "content": "第二条" }
]
```

| 含义 | 说明 |
|------|------|
| 顶层为数组 | 与 Step5 的 `messages` **元素结构相同**；**顺序 = 展示顺序**；**长度 ≤ 5**。 |

### 2.8 `conversation_log`（L2）、叹号/重发、**`role`/`sort_seq`**、Future 写入（**已收口**）

#### 2.8.1 展示与落库（L2）

- **H5 样式**：与产品截图一致——**用户**侧每条消息为**独立绿气泡**（`role=user`），**林小梦**侧每条为**独立白气泡**（`role=assistant`）；**一条气泡对应一行** `conversation_log`。  
- **一轮多句助手**：最终对外展示的 **`messages` 有几条，就写入几条 `role=assistant`**；每条 `content` 为该条气泡正文（当前仅 `text`）；**行数 ≤ 5**（与 §2.7 一致）。  
- **`round_id`**：成功闭环时，**本包内参与打包的所有 `user` 行**与**本轮写入的全部 `assistant` 行**使用**同一** `round_id`（UUID）；与现网 `_persist_bundle_success` 语义一致，**由「一条 assistant」扩展为「N 条 assistant」**。

#### 2.8.2 叹号与重发（**与现网 `chat.py` 行为对齐**）

- **Step5 / LLM 失败**（未生成有效助手落库）：仅将**本包 `user` 行**标为 `failed_timeout` / `failed_error`；**不写入** `assistant` 行；**此时失败 user 行通常尚无 `round_id`**（与现网 `_mark_pack_failed` 一致）。  
- **叹号展示**：仍以**未闭环窗口内 user 行出现失败态**等现网规则为准（见 `_open_window_has_bang` 等）；**不因「多条 assistant」单独改叹号语义**。  
- **`POST /api/chat/resend`**：**不插入新 user**；对当前有叹号的未闭环窗口**整包重新调度** LLM（与现网一致）；成功后按本轮 **`messages` 条数**写入 **N 条 assistant**，共享新分配的 **`round_id`**，并为本包 **user** 标 `delivered`。  
- **仅 Step5.5 失败**：回退 Step5 的 `messages` 落库，**不**因 5.5 单独产生「整轮叹号」（与 R-BND-06 一致）。

#### 2.8.3 **`role` 与 `sort_seq` 规则**（实现指导）

| 字段 | 规则 |
|------|------|
| **`role`** | 仅 **`user`** / **`assistant`** 两值，与现网一致；**用户每条气泡**一行 `user`（现网已支持多条 user 打包）；**林小梦每条气泡**一行 `assistant`。 |
| **`sort_seq`** | 全表 per-user **单调递增**（`allocate_sort_seq(user_id, count=N)` 原子分配，与 `timeline_seq_service` 一致）。**同一成功闭环内**：为本轮 **N 条 assistant** 一次性申请 **N 个连续** `sort_seq`，赋值顺序与 **`messages` 数组下标顺序一致**（第 0 条气泡 → 分配到的 N 个序号中最小者，第 1 条 → 次小，依此类推），保证 **`GET /api/chat/timeline` 升序** 时气泡顺序与产品一致。新分配的序号**天然大于**此前已提交的用户消息序号（与现网「先 user 落库、成功时再分配 assistant 序号」同一机制）。 |
| **`delivery_status` / `skipped_in_prompt`** | **assistant** 行与现网一致可为 **`NULL`**；**user** 行送达态逻辑保持现网。 |

#### 2.8.4 Step6 与 **Future 写 `relationship`（异步 + 重试）**

- **Step6 记忆 LLM、向量写回、关系字段更新等**：采用 **M2 半异步**——**不阻塞**用户主对话 SSE 成功返回；失败时 **自动重试 1 次**（共 **2 次尝试**）；重试后仍失败 → **允许不落库/不写向量**，**不影响**客户端正常对话流程。  
- **`relationship` 上 Future 槽位写入**（由 Step5 解析出的 `future` 等驱动）：**异步**执行即可；**可被 Step8 轮询**读取；**不要求**与 SSE 同步 await。  
- **Future 写入失败**：**最多重试 1 次**（与记忆策略一致）；再失败 → **可接受未写入**，不阻断对话；监控与后续补偿策略实现时可选。  

> **与计划稿 D 的关系**：**不再要求**「Step6 记忆 LLM 必须同步阻塞至本轮 HTTP 结束」；以本节 **M2 + 重试** 为准。
