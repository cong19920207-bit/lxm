## 二、数据层变更

> 具体字段命名与迁移方式，以实现时读代码为准；下述为语义描述。

### 2.1 扩展现有表

**`relationship` 表新增字段（语义）：**
- 关系文字描述（长文本）：由 Step6 记忆 LLM 更新，供 Step3 Prompt 使用
- 用户真实称呼（短文本）：Step6 更新，供下一轮 Prompt 使用
- 用户昵称/绰号（短文本）：Step6 更新
- 用户印象描述（长文本）：Step6 更新
- 未来预约时间戳（整数，可空）：Step6 写入，Step8 轮询消费
- 未来预约意图摘要（短文本，可空）：Step6 写入，Step8 消费
- 主动消息计数（整数，默认 0）：每次主动消息触发后累加，用户发新消息时清零

> **[待确认]** 上述字段是扩展 `relationship` 表还是新建关联表，取决于读代码后对 relationship 表现有宽度与访问频率的判断。未来预约字段也可考虑单独建表或存 Redis，实现时决定。

### 2.2 扩展向量库（DashVector）

当前向量库只有用户记忆一类，改造后需支持按类型区分。

**新增元数据字段（type 标识）：**

> **v0.31 同步**：以确认记录 R-L1L3-08 / R-VEC-01 为准，type 已扩为 4 类（旧 `character_setting` 拆为 `character_global` 公有 + `character_private` 私有，二者**互不混用**，命名全局唯一禁止拼写变体）。

| 类型标识 | 语义 | 写入时机 | 是否含 user_id |
|---------|------|---------|---------------|
| `character_global` | 角色公开标准设定（虚拟人统一形象） | 管理后台 CRUD 录入（R-L1L3-20）+ Step6 增量写入 | 否（角色级） |
| `character_private` | 对当前用户可见的角色私有设定 | Step6 增量写入（不在管理后台手动录入） | **是**（必须按 user_id 过滤） |
| `character_knowledge` | 角色知识与技能 | 管理后台 CRUD 录入（R-L1L3-20）+ Step6 增量写入 | 否（角色级） |
| `user` | 用户相关记忆 | 现有 memory_service（用户手动/自动）+ Step6 增量写入 | **是** |

> **注意：** 写入与检索侧的 type 字符串必须完全一致，避免 A 项目中 `chatacter_knowledge`（错别字）导致知识类记忆检索不命中的问题。B 项目从一开始统一用上述 4 类常量。

**初始数据**（已由确认记录 R-L1L3-20 收口）：初始为空，通过 Step6 对话积累逐步填充，**不做初始化脚本**；管理后台「角色知识库 CRUD 页」仅维护 `character_global` 与 `character_knowledge` 两类，`character_private` 不在此页维护（由 Step6 自动写入）。

## 三、各步骤功能需求

### Step1：并行获取（扩展）

**现有功能：** 并行拉取近期对话、关系信息、情绪上下文、用户 Embedding。

**新增装载内容：**

1. **时间描述串（`[时间描述串]`）**
   - 内容：将当前时间格式化为自然语言描述（如"现在是周三下午三点"）
   - 生成方式：代码直接根据系统时间生成，无需 LLM，无需 DB 查询
   - 用途：进入 Step1.5 / Step3 Prompt

2. **活动描述串（`[活动描述串]`）**
   - 内容：基于当前时间，描述虚拟人当前状态（如"她现在应该在午休"）
   - 生成方式：从活动计划数据源按时间查找对应描述 **[待确认数据源，见 2.3]**
   - 用途：进入 Step1.5 / Step3 Prompt

3. **关系文字描述（`[关系描述]`）**
   - 内容：从 `relationship` 表读取关系文字描述字段（初始为空或默认文案）
   - 用途：进入 Step3 Prompt 的关系模块 / Step1.5 context

4. **用户印象（`[用户印象]`）**
   - 内容：从 `relationship` 表读取用户真实称呼 / 昵称 / 印象描述
   - 用途：进入 Step3 Prompt 的关系模块 / Step1.5 context

**字段流转：** 以上内容在本轮请求的内存上下文中携带，供后续步骤使用，不单独持久化到 DB。

### Step1.5：查询重写 LLM（新增）

**位置：** Step1 完成后、Step2 向量检索前，单独一次同步 LLM 调用。

**目的：** 让模型理解用户本轮输入的真实意图，生成后续多路向量检索所需的查询语句与关键词。

#### Prompt 结构

```
【系统指令】
你是 {虚拟人名称}，请根据用户最新消息，分析你的内心理解，并为记忆检索生成查询语句。
输出格式要求：仅输出 JSON，不含任何额外内容。

【上下文模块 - 当前时间与状态】
{[时间描述串]}
{[活动描述串]}

【上下文模块 - 人格设定】
{人格设定内容（来自 admin_config persona）}

【上下文模块 - 关系状态】
当前关系等级：{level_name}
关系描述：{[关系描述]}
用户称呼：{[用户印象].称呼}

【上下文模块 - 近期对话】
{最近 N 轮对话记录}
（N 值与 Step5 保持一致，避免上下文不对齐）

【上下文模块 - 用户当前消息】
{本轮用户输入}

【任务】
根据以上内容，输出：
1. 你对用户消息的内心理解（InnerMonologue）
2. 用于检索角色背景设定的问题与关键词
3. 用于检索角色知识技能的问题与关键词
4. 用于检索用户相关记忆的问题与关键词
```

#### 输出 Schema（JSON）

> **v0.31 同步**：字段名以确认记录 R-L1L3-09 为准，旧名 `CharacterSetting*` 已作废，统一为 `CharacterGlobal*`（覆盖 `character_global` 与 `character_private` 两路检索用）。

```json
{
  "InnerMonologue": "对用户意图的内心理解，不超过 100 字",
  "CharacterGlobalQueryQuestion": "用于检索角色公共/私有设定的自然语言问题",
  "CharacterGlobalQueryKeywords": "关键词1 关键词2 ...",
  "CharacterKnowledgeQueryQuestion": "用于检索角色知识的自然语言问题",
  "CharacterKnowledgeQueryKeywords": "关键词1 关键词2 ...",
  "UserProfileQueryQuestion": "用于检索用户相关记忆的自然语言问题",
  "UserProfileQueryKeywords": "关键词1 关键词2 ..."
}
```

**输出流转：** 本步骤产出写入本轮内存上下文（`[查询重写结果]`），供 Step2 消费。InnerMonologue 不持久化、不返回前端。

**近期对话轮数**：与 Step5 一致，注入 **最近 10 轮**（确认记录 R-L1L3-14；直接复用 `_execute_llm_bundle` 内已截取的 `recent_10`，不新增 DB 查询）。

### Step2：多路向量检索（扩展）

**现有功能：** 基于用户当前输入的 Embedding，检索 TopK=5、阈值≥0.7 的用户记忆。

**改造内容：**

基于 Step1.5 输出的 3 组查询语句 + 关键词，分别执行 **3 逻辑路 / 4 次 DashVector 子查询**（确认记录 R-L1L3-10 方案 A1）：

| 检索路 | 查询来源 | 向量库 type 过滤 | 结果存入 |
|-------|---------|----------------|---------|
| 角色设定路 - 公有 | `CharacterGlobalQueryQuestion/Keywords` | `character_global`（无 user_id 过滤） | 合并到 `[角色设定检索结果]` |
| 角色设定路 - 私有 | `CharacterGlobalQueryQuestion/Keywords` | `character_private`（含 user_id 过滤） | 合并到 `[角色设定检索结果]` |
| 角色知识路 | `CharacterKnowledgeQueryQuestion/Keywords` | `character_knowledge`（无 user_id 过滤） | `[角色知识检索结果]` |
| 用户记忆路 | `UserProfileQueryQuestion/Keywords` | `user`（含 user_id 过滤） | `[用户记忆检索结果]` |

**检索参数**（已由确认记录 R-L1L3-17 收口）：
- TopK：**全路统一 TopK=3**，4 次子查询合计最多注入 ≤12 条
- 相似度阈值：≥0.7（与现有保持一致）
- 管理后台「向量召回配置」页热更（`config_key = "vector_retrieval_config"`）

**Embedding 与并行**（已由确认记录 R-L1L3-18 收口）：
- 正常路径：3 次 Embedding（`CharacterGlobal`、`CharacterKnowledge`、`UserProfile` 各一）→ 4 次 DashVector 检索（`CharacterGlobal` Embedding 复用于公有/私有两路）
- 降级路径（Step1.5 失败）：`last_user_text` 单 Embedding → 全部 4 路检索（确认记录 R-L1L3-12）
- 并行结构：`asyncio.gather` 并行 3 个 Embedding → `asyncio.gather` 并行 4 次 DashVector
- 现网 `_execute_llm_bundle` 中原有 `user_embedding = await _get_embedding(last_user_text)` 改造时**直接删除**，Embedding 生成职责完全移入 Step2 内部（确认记录 R-L1L3-21）

**结果流转：** 三路检索结果合并写入内存上下文，格式为可直接插入 Prompt 的文本块。

```json
  "messages": [
    { "type": "text", "content": "第一条给用户的气泡正文" },

    { "type": "text", "content": "第三条" }
```

**关键差异（计划稿 → 确认记录）**：
- `InnerMonologue` → `inner_monologue`

**`RelationChange.delta` 与 growth 映射**（已由 R-BND-09 收口）：本阶段**不优先定稿**，可按整体消息与 Step6 6.1 后续再定，**不阻塞** Step5～7 包定义。

### Step5.5：响应优化 LLM（新增）

**位置：** Step5 完成后、Step7 SSE 推送前。

**触发条件（最终以确认记录 §2.7.1 为准）：**
- **总开关（B3，最高优先级）**：管理后台「Step5.5 总开关」**关闭时永远不触发**，下方门闄判定全部跳过
- 总开关开启时（双门闄 OR）：
  - **门闄 A（保留原 12%）**：无条件参与；`rand < 0.12`，固定值，**不因 `knowledge_expand` 改变**
  - **门闄 B（仅当知识扩展）**：仅当 Step5 JSON 中 **`knowledge_expand == "是"`** 时生效，`rand < 0.5`（独立 50%）；当 `knowledge_expand == "否"` 时**不进入门闄 B**
  - 命中门闄 A **或** 门闄 B → 执行 Step5.5
- **总触发率**：知识扩展时 ≈ **56%**（`1 − (1−0.12)×(1−0.5)`）；非知识扩展时 12%

**`knowledge_expand` 字段名**（替换原 `ChatCatelogue`）：与 Step5 JSON 字段名对齐；trim 后仅精确「是」按是处理，其余一律按「否」（U1）。

**未触发时：** 直接使用 Step5 的 `messages`，跳过本步骤。

**触发时：**

#### Prompt 结构

```
【系统指令】
你是 {虚拟人名称}，请对以下初步回复进行优化改写，使其更自然、更符合角色风格。
```

### Step8：主动消息轮询 Handler（新增）

**定位：** 独立后台轮询进程，与现有定时巡视机制并存。

#### 8.1 Future 槽数据结构

存储位置：`relationship` 表扩展字段（或 **[待确认]** 独立表）

| 字段语义 | 类型 | 说明 |
|---------|------|------|
| 未来预约时间戳 | Integer，可空 | Unix 时间戳，到时触发 |
| 未来预约意图摘要 | String，可空 | 约 20 字的行动概要 |
| 主动消息计数 | Integer，默认 0 | 衰减惩罚用 |

**清零时机：** 用户发送新消息时（`POST /api/chat/send` 入口），将该用户的主动消息计数置 0。

#### 8.2 两路写入与优先级保护

| 写入路 | 触发条件 | 写入内容 | 优先级 |
|-------|---------|---------|-------|
| 路 A（对话驱动） | Step6 每轮结束 | LLM 规划的时间 + 意图 | 高 |
| 路 B（定时巡视） | 现有定时任务触发 | 周期性检查后生成的话题 | 低（兜底） |

**路 B 写入前的优先级检查：**
```
if future_timestamp 存在 AND future_timestamp > 当前时间:
    跳过写入（路 A 预约未过期，不覆盖）
else:
    正常写入（槽为空或已过期）
```

#### 8.3 轮询消费逻辑

**轮询频率：** **[待确认]**，参考 A 项目为每秒轮询。

**消费条件（同时满足）：**
1. `future_timestamp` 不为空
2. `future_timestamp` ≤ 当前时间（已到期）
3. `future_timestamp` > 当前时间 - 1800（30 分钟过期窗口，超期任务忽略）
4. 用户未被封禁

**消费后执行子链路：** 与主对话同源结构，全程使用主动消息专用 Prompt 版本：

```
触发条件满足
  → 加载用户上下文（关系、历史、时间、活动）
  → Step1.5 变体：查询重写（将 future.意图摘要 嵌入查询意图）
  → Step2：三路向量检索（同主对话）
  → Step3：拼装主动消息 Prompt（见下方 Prompt 结构）
  → Step5 变体：主动消息 LLM 调用
  → Step5.5：概率触发优化（default_refine_chance 可配置为较低值）
  → 写入发送队列 / agent_message 表（对接现有主动消息展示接口）
  → Step6 变体：记忆 LLM（与主对话相同逻辑）
  → 更新 relationship 表（主动消息计数 +1）
  → 判断是否写入下一轮预约（主动消息计数衰减门控）
```

**主动消息 Prompt 核心差异（相对主对话）：**

```
【用户消息】模块替换为：
【主动发起】
{虚拟人名称} 打算主动联系用户。
当前规划的行动：{future.意图摘要}
请生成一条自然的主动发起消息。
```

**写入下一轮预约的门控（主动消息专用）：**
- 主动消息计数（proactive_times）每次执行后 +1
- 若 LLM 输出了新的 FutureResponse：以 `0.15^(proactive_times+1)` 概率决定是否写入下一轮预约（越打扰越低概率继续）
- 未命中：清空 future 槽
