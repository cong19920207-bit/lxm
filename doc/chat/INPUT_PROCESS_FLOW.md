## 用户输入处理全流程详解

当用户向虚拟人发送消息后，该消息会经历一系列处理流程，最终生成回复发送给用户。整个过程可以分为以下几个阶段：

### 第一阶段：输入接收与预处理

1. **消息接收** - 微信Bot通过API接收到用户发送的消息（典型实现：E 云回调本服务 `POST /message`，请求体为 JSON。）
2. **消息存储** - 将原始消息存入inputmessages数据库集合（实现上多为先经 `ecloud_message_to_std` 标准化再 `insert_one`；语音/图片会先拉取媒体并识别后再写入同一集合结构。）
3. **状态标记** - 将消息状态设置为"pending"（待处理）（主循环拉取前保持 `pending`，处理中会为 `handling` 等，见详解。）

### 第二阶段：上下文构建与信息检索

> **说明（澄清）**：本阶段在实现上主要是**从数据库装配会话、关系与用户文档**，并生成供大模型使用的**时间串、新闻串、历史对话串**等；**多路向量记忆检索**在**第四阶段**由专用 Agent 完成。原标题中「信息检索」易与第四阶段混淆，此处保留原标题作为目录，含义以上述为准。

1. **会话信息获取** - 系统首先获取或创建与该用户的相关会话信息（`ConversationDAO`，集合 `conversations`。）
2. **关系信息获取** - 从relations集合中获取用户与虚拟人之间的关系数据（亲密度、信任度、反感度等）（`context_prepare` 读 `relations`，缺省则插入默认文档。）
3. **用户信息获取** - 获取用户和虚拟人的个人信息（`UserDAO.get_user_by_id` 分别加载用户文档与角色文档。）

### 第三阶段：输入理解与查询优化

1. **查询重写** - AI模型分析用户输入，理解其真实意图（`QiaoyunQueryRewriteAgent`，输出写入 `context["query_rewrite"]`。）
2. **关键词提取** - AI生成用于信息检索的关键词和查询语句（与第 1 点同一轮结构化输出中的 `*QueryQuestion` / `*Keywords` 字段。）
3. **意图分类** - 确定用户输入的意图类型（如询问、闲聊、情感表达等）（无独立意图分类服务；意图融于上述结构化输出，如 `InnerMonologue` 与各类 Query。）

### 第四阶段：多库多路信息检索

> **说明（澄清）**：**当日新闻**与**历史对话正文**主要在**第二阶段**的 `context_prepare` 中已装入 `context`（`news_str`、`chat_history_str`）；本阶段 Agent 主要负责基于第三阶段的 Query，对 `**embeddings` 向量库**做多路检索与合并。下列第 5、6 点与原文档条目一致，在详解中区分「本 Agent 检索」与「上下文已带数据」。

1. **角色设定检索** - 基于关键词从向量数据库中检索角色的背景设定（`QiaoyunContextRetrieveAgent`，`embeddings`，类型含 `character_global` 等。）
2. **角色知识检索** - 检索角色的专业知识和技能（同上 Agent，`character_knowledge` 一路。）
3. **用户资料检索** - 检索关于该用户的过往信息（同上 Agent，`user` 一路。）
4. **相册检索** - 检索角色的相册信息，决定是否分享照片（同上 Agent，`character_photo` 一路。）
5. **新闻资讯检索** - 获取最新的新闻和资讯信息（数据在 `context_prepare` 中自 `dailynews` 装入 `news_str`，非本 Agent 查库；语义与本条一致。）
6. **历史对话检索** - 获取与该用户的过往对话记录（数据来自会话 `chat_history` 及 `context_prepare` 生成的 `chat_history_str`；语义与本条一致。）

### 第五阶段：响应生成

1. **上下文整合** - 将检索到的所有信息整合成AI处理的上下文（`QiaoyunChatResponseAgent` 将 `context` 格式化为 prompt。）
2. **内容生成** - AI根据上下文生成回复的内心独白（含 `InnerMonologue`、`ChatResponse` 等结构化字段。）
3. **多模态响应规划** - AI规划可能的多模态响应（文本、语音、照片）（`MultiModalResponses` 数组；出站时按类型合成或解析相册。）
4. **关系影响评估** - AI评估此次对话对双方关系的影响（`RelationChange`；**亲密度、信任度数值在本阶段 `_posthandle` 写回 `relations`**。）

### 第六阶段：响应优化

1. **质量评估** - AI评估生成内容的质量和适宜性（无独立评分微服务；是否细化由概率与内容特征等启发。）
2. **个性化调整** - 根据用户特点调整回复内容（依赖上下文与 prompt，无独立规则引擎。）
3. **概率性细化** - 根据一定概率使用更高级的AI模型进行内容优化（`QiaoyunChatResponseRefineAgent`，可改写 `MultiModalResponses`。）

### 第七阶段：输出格式化与发送

1. **多模态包装** - 将生成的内容按照多模态格式包装（文本、语音或照片）（写入 `outputmessages` 及 `metadata`，语音走 TTS 与 OSS 等。）
2. **发送时序规划** - 计划消息的发送时序，模拟真实人类的打字时间（`expect_output_timestamp` 与分段间隔计算。）
3. **消息发送** - 通过微信Bot API发送消息给用户（如 E 云 `sendText` / `sendVoice` / `sendImage2`。）
4. **状态更新** - 将发送的消息记录到outputmessages数据库集合（并更新发送状态、回写会话历史等，见详解。）

### 第八阶段：对话后处理与记忆更新

> **说明（澄清）**：下列第 1～5 条**粗体与破折号前文字**与历史原文档一致。实现上**亲密度、信任度**的数值多在**第五阶段**已根据 `RelationChange` 写回；本阶段 `QiaoyunPostAnalyzeAgent` 侧重总结、向量记忆、`dislike`、关系描述与用户印象等。详见下文「详解：第八阶段」第 3 条。

1. **对话总结** - AI分析本次对话的重要信息
2. **记忆更新** - 将新的用户信息、角色信息更新到向量数据库
3. **关系调整** - 根据对话内容调整用户与虚拟人的关系数值（亲密度、信任度）
4. **反感度评估** - 更新反感度数值，如果到达阈值会拉黑用户
5. **长期目标更新** - 更新虚拟人的短期和长期目标

### 第九阶段：未来主动消息规划

1. **主动消息规划** - AI规划未来的主动消息时机和内容（模型输出经第五阶段逻辑写入 `conversation_info.future`。）
2. **时间安排** - 根据对话情况安排合理的时间发送主动消息（时间串解析为时间戳并带概率门控。）
3. **任务调度** - 将主动消息任务安排到后台任务队列（实现为 `qiaoyun_background_handler` 等轮询消费 `future`，非独立 MQ 表。）

## 系统特性

### 智能化特征

- **多模态交互**：支持文本、语音、图片等多种交互方式（入站以 `ecloud_input` 中 `supported_message_types` 为准。）
- **长期记忆**：通过向量数据库实现长期信息存储和检索（集合 `embeddings`，第四阶段检索、第八阶段写入。）
- **人格一致性**：通过丰富的角色设定确保回复符合虚拟人的人格特征（设定与记忆进 prompt。）
- **关系感知**：根据关系亲密度和信任度调整回复方式（`RelationChange` 在第五阶段更新数值。）

### 行为模拟特征

- **延迟回复**：模拟真实人类的思考和打字时间（`expect_output_timestamp`、分段延迟。）
- **主动沟通**：虚拟人会主动发送消息维持关系（`future` + 后台 handler。）
- **忙闲状态**：虚拟人有忙闲状态，模拟真实生活节奏（`relations.relationship.status`；非「空闲」时输入可 `hold`。）
- **好感度系统**：根据好感度调整回复频率和内容（亲密度、信任度等；与忙闲、主动消息等策略相关，见代码。）

### 安全机制

- **反感度监控**：监控用户的不当行为，达到阈值自动拉黑用户（`dislike` 与 `qiaoyun_handler` 拦截分支。）
- **内容过滤**：对生成内容进行安全检查（主要为模型与 prompt 约束，无独立审核服务调用。）
- **行为限制**：根据关系亲密度控制内容开放度（关系信息与 prompt 共同作用。）

## 详解：系统特性（与上文「系统特性」三节对应）

### 与「智能化特征」四条对应


| 原文档要点 | 实现要点                                               |
| ----- | -------------------------------------------------- |
| 多模态交互 | 入站适配器支持类型见 `ecloud_input`；出站 `outputmessages` 多类型。 |
| 长期记忆  | `embeddings` + 第四阶段检索 + 第八阶段 `upsert_one`。         |
| 人格一致性 | 角色与用户上下文 + prompt。                                 |
| 关系感知  | `relations` + 第五阶段数值更新 + 第八阶段文本与反感度。               |


### 与「行为模拟特征」四条对应


| 原文档要点 | 实现要点                               |
| ----- | ---------------------------------- |
| 延迟回复  | `expect_output_timestamp` 与分段延迟。   |
| 主动沟通  | `future` + 后台 handler。             |
| 忙闲状态  | `relationship.status` 与 `hold` 分支。 |
| 好感度系统 | `closeness`/`trustness` 等与后台策略。    |


### 与「安全机制」三条对应


| 原文档要点 | 实现要点                    |
| ----- | ----------------------- |
| 反感度监控 | `dislike` 与主循环拉黑分支。     |
| 内容过滤  | 以模型与 prompt 为主，无独立审核服务。 |
| 行为限制  | 关系、反感度与 prompt 共同约束。    |
